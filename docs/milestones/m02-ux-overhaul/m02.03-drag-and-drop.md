# m02.03 — Drag-and-Drop Reorganization
_Parent: [`m02-ux-overhaul.md`](m02-ux-overhaul.md)_

Drag-and-drop reorder + reparent within an Initiative, per [`ProductSpec.md`](../../ProductSpec.md) § Reorganization. Cross-Initiative moves are out of scope.

All items are **pending owner approval** until the owner moves checkboxes from `[ ]` to confirmed.

## Items

1. [ ] **Drag handle on each task row.** A 3-vertical-dots glyph at the far left of the row (left of the chevron / checkbox) is the only place a drag starts. Prevents accidental drags from row-body clicks. Handle has its own touch target meeting the 44×44 px rule from UX_GUARDRAILS.
2. [ ] **Vertical drag → reparent.** Dropping task A onto task B's row makes A a child of B (appended at the end of B's children). Drop target highlights during hover. After commit, ancestor progress recomputes for both old and new parent chains.
3. [ ] **Horizontal drag → indent / dedent.** While dragging, horizontal cursor position controls depth: dragging right past a threshold indents A under its previous sibling; dragging left dedents A toward an ancestor's level. A subtle indent guide tracks the cursor so the target depth is visible before drop.
4. [ ] **Snap-back zone.** A horizontal cursor band inside the row that means "no change" — letting go inside that band aborts the move with no commit. Lets the user back out mid-drag without committing a wrong drop.
5. [ ] **Cross-list moves within the same Initiative.** Dragging across List trees changes A's root List. Cross-Initiative moves remain out of scope (ProductSpec § Reorganization).
6. [ ] **Persistence + roll-up recompute.** Every drop computes new `parent_id` + `sort_order`, persists in a single transaction, and recomputes ancestor progress for both the old and new parent chains. Failures roll back the entire drop and surface a flash; the UI reverts to pre-drag state.
7. [ ] **Keyboard alternative — reuse task-selected state.** A task is "selected" when its Details panel is open (the existing selection model from Arc 2 item 6 / 21). While a task is selected, keyboard shortcuts reorder and reparent that task: `Alt+↑/↓` reorders within siblings; `Alt+←/→` dedents / indents. No mouse needed; no separate "drag focus" concept — selection IS the keyboard focus for reorganization.
8. [ ] **Mobile touch — long-press to start drag.** Long-press (~400 ms) on the drag handle starts the drag; the rest of the gesture mirrors mouse behavior. Short taps on the handle do nothing — the handle is drag-only, not a button.

## Cross-arc note

The drag infrastructure picked here (library, hook abstraction, motion conventions, drop-feedback visuals) needs to be **reusable for Arc 4 item 11** — drag-a-person from the cross-Initiative people pane onto an Initiative entry in the left rail. Same primitives, different source / target. Pick something that can host both gestures without forking; favor a small custom hook over a heavy library if the heavy library only fits one of the two cases.

## Implementation Notes

- **Library vs. roll-your-own.** Phoenix has community Sortable.js wrappers (e.g. `live_view_sortable`); evaluate one against rolling our own minimal hook. The keyboard-alternative + snap-back-zone + indent-guide combination may push toward custom — and the Arc 4 reuse requirement above tilts further in that direction.
- **Sort order.** Use float-style ordering (`sort_order` as a numeric column) so single-row updates are common and full re-numbering is rare. Re-number lazily when ordering precision degrades.
- **Drop target visuals.** Highlight on hover; show indent guide for horizontal position; respect motion-reduce.
- **Optimistic vs. server-confirmed.** Render the move client-side immediately on drop; if the server transaction fails, revert and flash. Don't wait on the round-trip to update UI.
- **Cross-list within-Initiative bookkeeping.** When A moves between Lists, both the source List and the destination List need ancestor-progress recompute (their root-task progress changes), and the Initiative-level aggregate underbar (Arc 2 item 7) updates accordingly.
