# m02.04 — Wide-Width Layout
_Parent: [`m02-ux-overhaul.md`](m02-ux-overhaul.md)_

Make the app earn its width on larger displays. Two phases inside one arc:

1. **Responsive groundwork (items 1–4).** The existing two-pane layout (main + right-rail) opens up at `xl:` and `2xl:` so we stop wasting horizontal real estate on wide monitors.
2. **Unified ultrawide layout (items 5–12).** Above the widest breakpoint, a left rail joins the layout — it IS the Initiatives index, plus a cross-Initiative people pane. The whole account is in chrome; the main column is the current Initiative's task tree; the right rail is the existing Details slot. People in the people pane can be dragged onto an Initiative entry to add them as a member.

Owner-approved 2026-05-09. Items below are scoped; checkboxes flip to `[x]` as each ships. Open Questions below remain to be resolved in-flight or before specific items.

## Items

1. [ ] **Container max-width refresh.** Today the Initiative-show and Initiatives-index pages live inside `max-w-6xl` (1152 px), leaving large empty gutters at `xl:`/`2xl:`. Widen (or remove) the cap at those breakpoints so the layout breathes. Header `max-w-6xl` follows the body so the chrome stays visually aligned with content. Pick a widening that keeps line-length readable for body text — not unbounded.
2. [ ] **Right-rail sizing at `xl:`/`2xl:`.** Right rail is a fixed 360 px from `lg:` up. Grow it stepwise at `xl:` and `2xl:` so Task details / Initiative editor / Members aren't cramped on wide displays — and so the rail stays visually proportional once the main column widens.
3. [ ] **Task-row breathing room on wide screens.** Larger horizontal padding and gutter spacing at `xl:`/`2xl:`; allow task description to wrap to a second line (instead of `truncate`) once room exists; consider a slightly larger title size at depth 0 for List rows on wide screens. Keep the row composable single-line on narrow.
4. [ ] **Initiatives index grid expands.** Today's card grid jumps from 1 column to ~3. Add `xl:` (4 cols) and `2xl:` (5 cols) breakpoints so the index uses available width on wide monitors.
5. [ ] **Triple-pane layout at an ultrawide breakpoint.** Pick a breakpoint above `2xl:` (Tailwind has no built-in `3xl:`; either add a custom one or use a min-width media query) at which the layout becomes left-rail · main · right-rail. Below the breakpoint, the layout is the wide-tuned two-pane from items 1–4. Crossing the threshold causes no visible jank.
6. [ ] **Left rail — Initiatives list (the index, in chrome).** The rail isn't a bespoke widget — it's the Initiatives index, rendered in compressed form for the rail width. Same data, same components where practical. Each entry: grove icon, Initiative name, role badge (consistent with Arc 2 item 26), small progress indicator. The current Initiative is visually highlighted.
7. [ ] **Main empty state when no Initiative is selected.** At `/initiatives` on ultrawide, the rail shows the list as usual; the main column shows a "pick an Initiative" empty state (not the card grid — the rail covers that). The right rail is present but empty. Below the ultrawide breakpoint, `/initiatives` keeps today's behavior (card grid in main, no rail).
8. [ ] **Click the current Initiative on the rail to close it.** Clicking the highlighted (current) Initiative entry navigates back to `/initiatives` — i.e. closes it, returning to the empty-state main column from item 7. Clicking any other Initiative entry navigates to that Initiative's show page as usual.
9. [ ] **Left rail — cross-Initiative people pane.** Below the Initiatives list. Aggregates everyone the current user shares any Initiative with, deduplicated. Default sort: most-shared-Initiatives-first. Each row: avatar / color blob, name, count of shared Initiatives.
10. [ ] **Click a person to add or remove from the current Initiative.** Click surfaces a small menu / action with the contextual choice: "Add to <current Initiative>" if they aren't a member, "Remove from <current Initiative>" if they are. This is the keyboard / a11y / non-drag path; honored at every breakpoint where the people pane appears. If no Initiative is selected (main column is the empty state), the click is a no-op (or surfaces a hint that an Initiative needs to be open first).
11. [ ] **Drag a person onto an Initiative entry to add them.** Direct-manipulation shortcut on top of item 10. Drag a person row from the people pane onto any Initiative entry in the rail to add them as a member of that Initiative. If they're already a member, the drop is a no-op with a brief visual indicator. Drag-to-remove is intentionally NOT a gesture — removal is click-only via item 10. Reuses the drag infrastructure from Arc 3 (see the cross-arc note in `m02.03`).
12. [ ] **Mobile / narrow flyout for the left rail.** Below the ultrawide breakpoint, the left rail collapses out of the layout. On `/initiatives/:id`, a flyout affordance appears (mirroring the right-rail flyout from Arc 2 item 16) that brings up the left rail as an overlay. The affordance does NOT appear on `/initiatives` itself — there, the index IS the main column, so the rail would be redundant. Drag-to-add (item 11) is desktop-only; touch users use click (item 10).

## Cross-cutting requirements

- **No new schema; pure aggregation.** The people-pane data comes from a single Ecto query joining `initiatives` → `initiative_members` → `users`. Cache or precompute later only if measured to be slow.
- **Doesn't conflict with per-Initiative Members panel.** The right-rail Members panel (today's Initiative-scoped membership UI) stays as the canonical "who's on THIS Initiative" surface and the place to change roles after the fact. The left-rail people pane is a different question — "who do I work with across everything?" — so it's a different answer.
- **No layout shift across any breakpoint.** Resizing across `xl:`, `2xl:`, and the ultrawide threshold is smooth — no layout jank, no scroll-position loss, no flashes. Honors `UX_GUARDRAILS` B.1.
- **Polished light + dark at every breakpoint.** Both themes pass visual review at every breakpoint that introduces or alters chrome.
- **Drag a11y.** Item 11 is the rich-input shortcut; item 10 is the universal path. Every drag-to-add must be achievable via click-to-add. Same rule as Arc 3.

## Open Questions

- **Ultrawide breakpoint value.** Tailwind's default `2xl:` is 1536 px. Most ultrawides start around 2560 px. Pick the right boundary, or define a custom `3xl:`.
- **Default role on add (items 10 & 11).** Click-add and drag-add both need a default role since neither path includes a role-selection step. Editor is the safest default; viewer is the most conservative. Whatever we pick, the existing Members panel is the place to change it after the fact.
- **Empty state for the people pane.** Solo user with no shared Initiatives — does the section hide entirely, or render an empty-state hint?
- **Description-wrap policy (item 3).** Two-line max with ellipsis, or unlimited? Two-line keeps row heights predictable.
- **Naming for the people-pane feature.** "People you work with"? Something else? "Collaborators" is the obvious word but the owner has flagged it as not their favorite.
- **Drop-target visual feedback (item 11).** Highlight the target Initiative entry on hover-while-dragging? Show a ghost? Decide at implementation; should feel of a piece with Arc 3's drop feedback.

## Implementation Notes

- **Re-render frequency.** The people-pane data changes only on membership writes — subscribe via PubSub on the existing membership topic rather than polling.
- **Selection state in the rail.** Current Initiative comes from the URL / LiveView assigns; no separate state.
- **Header alignment.** The header's container width must mirror whatever max-width policy the body adopts (item 1) so the chrome doesn't visually float relative to content.
- **Index-as-rail reuse.** Item 6 should reuse the existing index card components in a compressed variant (probably a slot prop or a `compact` flag) rather than forking a parallel widget. This keeps the "rail IS the index" promise true at the code level too, and means future index changes propagate to the rail for free.
