# m03.02 — Roll-up progress & completion

_Parent: [`m03-api-mcp.md`](m03-api-mcp.md)_

Not an MCP feature — a data-layer and UX fix that Arc 3's real MCP test-drive happened to surface. Today, roll-up progress and the completion cascade (a parent flips done when every child is) are computed **eagerly on every write**: a leaf edit walks the whole ancestor chain, refetches and reconciles the **entire Initiative's tree**, and persists each changed ancestor with its own write — inside the same transaction, holding a lock on the shared Initiative root the whole time. A ~50-80 op MCP batch hit Postgres's connection-checkout limit and crashed because of this. It's also the same mechanism behind a UX pain point raised independently during M02 (and dismissed at the time, on record in `app.js`: "don't fake a number") — ordinary edits on a bigger tree visibly lag while the server does this work. Same root cause at two different scales.

`computed_progress` itself is **not going away** — it's load-bearing at every task level, not just Initiative roots: the API tree response (`serializer.ex:256`), every atomic-operation result (`operations.ex:1435`), the LiveView template's row rendering (`initiative_workspace_live.ex`, several call sites), and sort-by-progress (`tasks/sort.ex:98`) all read it directly, for any task, anywhere in a tree. Dropping or narrowing it would be a real regression to the API and the UI both. What actually costs today isn't "writing the ancestors" — it's doing it against the **whole Initiative** instead of just the affected chain, and with N individual writes instead of one. Fix those two things and keep the write synchronous; **only add an async layer if a real concurrency benchmark says it's still needed** — the underlying fix is identical either way, so starting simple isn't a bet that costs anything if it turns out wrong. The roll-up **formula itself doesn't change** — same leaf-average / single-level-average math from `ProductSpec.md`.

1. [x] **Backend: scope and batch the ancestor-chain update, synchronously**

	1. [x] **Scope the ancestor recompute to just the affected chain — not the whole Initiative.**
		1. [x] Replace `reconcile_progress`'s (`tasks.ex:2551`) call into `list_initiative_tasks` (`tasks.ex:55`, the whole-Initiative refetch) with a targeted fetch of just the changed task's ancestor chain up to the root — the `list_descendants` recursive-CTE pattern (`tasks.ex:1528`), inverted for an up-walk. Built as `fetch_ancestor_chain/1` (reuses `ancestor_chain_query/1`) + `leaf_rows_for_ancestors/1`, one recursive CTE tagging every chain ancestor's leaf descendants at once (the same bucketing technique already proven in `DoIt.Tasks.Assigned.leaf_counts/1`) rather than one query per ancestor — fewer round trips, same "SQL only fetches rows" rule.
		2. [x] Recompute progress bottom-up along just that chain.
		3. [x] Drop the redundant per-call `progress_calc_mode` query (`tasks.ex:2588`) — thread the Initiative's mode through from the caller instead of re-fetching it on every reconcile.
	2. [x] **Batch the ancestor writes into one query.**
		1. [x] Replace `persist_progress`'s (`tasks.ex:2574`) per-task `Repo.update!` loop with one batched bulk update (`Repo.update_all` with a `CASE WHEN id = ... THEN ...` fragment, or the Postgres `UPDATE ... FROM (VALUES ...)` join form) for the whole chain in a single query. `persist_progress/2` and the now-dead `Task.computed_progress_changeset/2` removed; `batch_persist_progress/1` (dynamic `CASE WHEN` per id) is the one write path.
	3. [x] **Same scoping + batching for the completion cascade.**
		1. [x] `check_completed_ancestors`/`uncheck_done_ancestors` (`tasks.ex:1445-1468`) stop triggering a full nested `update_task` (its own whole-tree reconcile) at each level — walk the chain once, computing and batching the status flips directly, same as 1.1-1.2. Split into a read-only collect phase + one batched status write; turned out the cascade never needs its own progress recompute at all (progress is a pure function of leaf state, and 1.1's chain recompute already runs immediately before the status walk on every caller's path) — zero redundant work, confirmed by the full undo/redo/cascade suite.
	4. [x] **Confirm the existing readers need zero code changes.** The API serializer, the atomic-operations per-op result, the LiveView template's row rendering, and sort-by-progress all keep reading the column exactly as today. As a side effect of the update landing in the same transaction as the write (not an independent goal), there's no staleness window in this baseline — but that's a property of choosing sync-first, not a guarantee to protect if 1.5's async path ever gets triggered.
	5. [x] **Document the async escape hatch — don't build it yet.**
		1. [x] If item 5.2's concurrency benchmark shows real contention at a realistic future scale (not just theoretically nonzero), the upgrade is: move the same scoped-and-batched function (1.1-1.3) out of the triggering transaction into a detached async job, riding the leaf-change broadcast item 2 already fires. A call-site change on top of already-built logic, not a rebuild.
		2. [x] That upgrade would also need, only if it's ever triggered: coalescing rapid edits to the same chain into one job, and a retry/reconciliation story for a failed job. Real, additional scope — deferred unless the benchmark actually calls for it, not built preemptively.

2. [x] **Live collaborator updates — cheap, in-memory, still one formula**

	When someone else changes a leaf, your browser's connected LiveView process already holds the whole tree in memory — no need to wait on a database read to tell you about it, even though the database itself stays fast now too.

	1. [x] On a leaf-change broadcast, the LiveView process recomputes the affected ancestor chain from its own in-memory tree assign (not a DB read) and sends the normal render patch — reusing the one Elixir formula function, not a new implementation. Built as `patched_lineage/2`: one cheap single-row read for the changed task's own new fields (a broadcast only names *which* task changed, not its new values) via new `Tasks.task_row/1`, merged into the socket's already-loaded `@tree`, then `Progress.compute_all/2` runs once over the in-memory tree — zero additional DB access per collaborator per edit, down from one `Tasks.lineage/1` DB read each.
	2. [x] Same for the completion cascade. No special-casing needed — the backend already emits one `{:task_updated, id}` broadcast per flipped task (leaf and every cascaded ancestor); each is handled identically by the same path.

3. [x] **Client-side instant prediction (your own edit only)**

	The one place this formula gets duplicated — deliberately scoped small, and never trusted as the record.

	1. [x] **Extract the formula into a small, pure, dependency-free JS file** (e.g. `assets/js/rollup.js`) — `computeRollup(leafProgressList, mode)`, `computeDoneCascade(childStatusList)`. No DOM access, no LiveView hook dependencies, no imports beyond the file itself — trivial to read and verify in isolation, and importable by both `app.js` and the standalone test page in 3.4.
	2. [x] **Wire it into the existing optimistic-write hooks in `app.js`:**
		1. [x] The progress-slider change handler (currently in the `pinkFields` block) — after the leaf's own optimistic value is set, call `computeRollup` over `savingAncestors(li)` and paint each ancestor's bar with the predicted number, replacing today's `markRecomputing` indeterminate-gradient treatment.
		2. [x] The `data-complete-toggle` click handler (`applyToggleOptimism`) — same idea, calling `computeDoneCascade` for the ancestor chain's done-state. Scope extended beyond the literal text here: ancestor *progress* numbers are predicted too (not just done-state), since a toggle snaps its subtree to a fully-known 100/0 just like a slider edit — riding the existing `DoitPendingToggle`/revert hold so a rejected or cancelled toggle can't leave a lying predicted value standing.
	3. [x] **The client never writes this predicted value anywhere** — no new field in the write payload carries a computed ancestor value; the request only ever carries the leaf's own `manual_progress` or `done` flip, same as today. Display only; the next real update (the write's own confirmation, or any later broadcast) always wins if it's ever different.
	4. [x] **A lean JS test file** (e.g. `assets/js/rollup_test.html`): plain `<script>` loading `rollup.js`, a handful of hand-written cases, a homemade `assert()`, opened directly in a browser. Cases to cover: a single leaf, several leaves, a zero-leaf branch (edge case), both `progress_calc` modes, a multi-level nested tree, and the done-cascade's own edge cases (empty children, mixed done/open siblings). No Node, no npm, no build step — this project deliberately has neither (the e2e rig is gone for good; `mix test` is the only *automated* gate). The homemade `assert()` is chosen because this case count is small and fixed, not because a framework is categorically off the table — a zero-install, vendored single-file framework would cost the same as our own and stays available if the JS test surface outgrows this (tracked in `PLAN.md`'s Deferred Decisions). 26/26 cases pass.

4. [x] **Async escape hatch — coalesce ancestor recomputes off the triggering transaction**

	Item 5.2's benchmark: 93% write failure at 150 concurrent writers to one Initiative vs. 6% split across separate ones — the shared root, not pool math. Fix: coalesce concurrent recomputes into one write per window instead of N.

	1. [x] **Mechanism: per-Initiative debounce `GenServer`** (`Registry` + `DynamicSupervisor`, no new dependency, no Oban). Built as `DoIt.Tasks.RollupDebounce`: one-shot (`restart: :temporary`, flush then stop — no idle processes). Self-healing confirmed — the pass recomputes from current rows, and the kill-mid-window test proves the next edit heals.
	2. [x] Leaf's own write stays immediate — **including its own `computed_progress`**, a pure function of its own row, set synchronously in the write transaction (`reconcile_self_row/2`) so a write's API response never echoes a stale `progress` (`operations.ex:1435`). Only true ancestors defer.
	3. [x] Every `recompute_ancestors` call site enqueues to the debounce `GenServer` instead of running inline — routed inside the two entry points (`recompute_ancestors/2`, `recompute_self_and_ancestors/2`) so create, delete, undo/redo, and both sides of a move are covered without touching call sites. Enqueue fires post-commit via new `DoIt.Broadcast.after_commit/1`, so the pass never reads pre-commit state and a rolled-back preview never enqueues; previews force the inline route (they diff mid-transaction progress).
	4. [x] Debounce ~100-200ms, bounded max wait so a busy tree isn't starved, then one pass over all dirty seeds (`run_rollup_pass/2` — `leaf_average` unions the chains into one batched write; `single_level` runs per-seed sequentially). Terminates when idle. Defaults 150ms/500ms, retunable via `config :doit, DoIt.Tasks.RollupDebounce`.
	5. [x] The debounce pass broadcasts post-commit — one `{:task_updated}` per changed ancestor (`reconcile_chain_nodes/2` returns the changed ids), absorbed by item 2's existing path. Without this, a full tree reload landing mid-window (mount, or a structural broadcast's `load_tree`) keeps stale ancestors indefinitely; with it, staleness = the debounce delay, matching the Cross-cutting bullet below.
	6. [x] **Reconciliation:** accept the crash-mid-window gap — a lost pass self-heals on the tree's next edit; no periodic sweep.
	7. [x] Test env runs the recompute inline (config flag) so the existing suite stays deterministic; only 5.4's tests exercise the real `GenServer`.

## Cross-cutting requirements

- **One formula, one place.** The roll-up and completion-cascade logic lives in exactly one Elixir function. SQL is only ever used to fetch rows efficiently, never to compute the math itself.
- **`computed_progress` stays a full-coverage cache.** Every task keeps a readable value, matching today's API and UI surface exactly. Sync (items 1-3) keeps it exact, no staleness window. Once item 4 lands, it relaxes to "correct within a short, bounded window" (the debounce delay) — deliberate, not a regression.
- **Async is an available upgrade, not a starting assumption.** Item 4 only exists because item 5.2's benchmark, not intuition, showed it's needed.
- **Client prediction is display-only.** The browser-side copy (item 3) is the sole deliberate exception to "one place," scoped to the acting user's own edit; it is never persisted and never treated as authoritative.
- **No Node/npm/build-step dependency.** JS verification stays a lean, manually-run file with no install step — consistent with this project's standing choice to keep `mix test` as the only automated gate. A framework's fine in principle if it's a zero-install vendored file; item 3.4 sticks with a homemade `assert()` because the current case count doesn't need one.
