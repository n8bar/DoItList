# m03.02 — Roll-up progress & completion

_Parent: [`m03-api-mcp.md`](m03-api-mcp.md)_

Not an MCP feature — a data-layer and UX fix that Arc 3's real MCP test-drive happened to surface. Today, roll-up progress and the completion cascade (a parent flips done when every child is) are computed **eagerly on every write**: a leaf edit walks the whole ancestor chain, refetches and reconciles the **entire Initiative's tree**, and persists each changed ancestor with its own write — inside the same transaction, holding a lock on the shared Initiative root the whole time. A ~50-80 op MCP batch hit Postgres's connection-checkout limit and crashed because of this. It's also the same mechanism behind a UX pain point raised independently during M02 (and dismissed at the time, on record in `app.js`: "don't fake a number") — ordinary edits on a bigger tree visibly lag while the server does this work. Same root cause at two different scales.

`computed_progress` itself is **not going away** — it's load-bearing at every task level, not just Initiative roots: the API tree response (`serializer.ex:256`), every atomic-operation result (`operations.ex:1435`), the LiveView template's row rendering (`initiative_workspace_live.ex`, several call sites), and sort-by-progress (`tasks/sort.ex:98`) all read it directly, for any task, anywhere in a tree. Dropping or narrowing it would be a real regression to the API and the UI both. What actually costs today isn't "writing the ancestors" — it's doing it against the **whole Initiative** instead of just the affected chain, and with N individual writes instead of one. Fix those two things and keep the write synchronous; **only add an async layer if a real concurrency benchmark says it's still needed** — the underlying fix is identical either way, so starting simple isn't a bet that costs anything if it turns out wrong. The roll-up **formula itself doesn't change** — same leaf-average / single-level-average math from `ProductSpec.md`.

1. [ ] **Backend: scope and batch the ancestor-chain update, synchronously**

	1. [ ] **Scope the ancestor recompute to just the affected chain — not the whole Initiative.**
		1. [ ] Replace `reconcile_progress`'s (`tasks.ex:2551`) call into `list_initiative_tasks` (`tasks.ex:55`, the whole-Initiative refetch) with a targeted fetch of just the changed task's ancestor chain up to the root — the `list_descendants` recursive-CTE pattern (`tasks.ex:1528`), inverted for an up-walk.
		2. [ ] Recompute progress bottom-up along just that chain.
		3. [ ] Drop the redundant per-call `progress_calc_mode` query (`tasks.ex:2588`) — thread the Initiative's mode through from the caller instead of re-fetching it on every reconcile.
	2. [ ] **Batch the ancestor writes into one query.**
		1. [ ] Replace `persist_progress`'s (`tasks.ex:2574`) per-task `Repo.update!` loop with one batched bulk update (`Repo.update_all` with a `CASE WHEN id = ... THEN ...` fragment, or the Postgres `UPDATE ... FROM (VALUES ...)` join form) for the whole chain in a single query.
	3. [ ] **Same scoping + batching for the completion cascade.**
		1. [ ] `check_completed_ancestors`/`uncheck_done_ancestors` (`tasks.ex:1445-1468`) stop triggering a full nested `update_task` (its own whole-tree reconcile) at each level — walk the chain once, computing and batching the status flips directly, same as 1.1-1.2.
	4. [ ] **Confirm the existing readers need zero code changes.** The API serializer, the atomic-operations per-op result, the LiveView template's row rendering, and sort-by-progress all keep reading the column exactly as today — and it's always immediately correct, no staleness window, since the ancestor update lands in the same transaction as the triggering write.
	5. [ ] **Document the async escape hatch — don't build it yet.**
		1. [ ] If item 4's concurrency benchmark shows real contention at a realistic future scale (not just theoretically nonzero), the upgrade is: move the same scoped-and-batched function (1.1-1.3) out of the triggering transaction into a detached async job, riding the leaf-change broadcast item 2 already fires. A call-site change on top of already-built logic, not a rebuild.
		2. [ ] That upgrade would also need, only if it's ever triggered: coalescing rapid edits to the same chain into one job, and a retry/reconciliation story for a failed job. Real, additional scope — deferred unless the benchmark actually calls for it, not built preemptively.

2. [ ] **Live collaborator updates — cheap, in-memory, still one formula**

	When someone else changes a leaf, your browser's connected LiveView process already holds the whole tree in memory — no need to wait on a database read to tell you about it, even though the database itself stays fast now too.

	1. [ ] On a leaf-change broadcast, the LiveView process recomputes the affected ancestor chain from its own in-memory tree assign (not a DB read) and sends the normal render patch — reusing the one Elixir formula function, not a new implementation.
	2. [ ] Same for the completion cascade.

3. [ ] **Client-side instant prediction (your own edit only)**

	The one place this formula gets duplicated — deliberately scoped small, and never trusted as the record.

	1. [ ] **Extract the formula into a small, pure, dependency-free JS file** (e.g. `assets/js/rollup.js`) — `computeRollup(leafProgressList, mode)`, `computeDoneCascade(childStatusList)`. No DOM access, no LiveView hook dependencies, no imports beyond the file itself — trivial to read and verify in isolation, and importable by both `app.js` and the standalone test page in 3.4.
	2. [ ] **Wire it into the existing optimistic-write hooks in `app.js`:**
		1. [ ] The progress-slider change handler (currently in the `pinkFields` block) — after the leaf's own optimistic value is set, call `computeRollup` over `savingAncestors(li)` and paint each ancestor's bar with the predicted number, replacing today's `markRecomputing` indeterminate-gradient treatment.
		2. [ ] The `data-complete-toggle` click handler (`applyToggleOptimism`) — same idea, calling `computeDoneCascade` for the ancestor chain's done-state.
	3. [ ] **The client never writes this predicted value anywhere** — no new field in the write payload carries a computed ancestor value; the request only ever carries the leaf's own `manual_progress` or `done` flip, same as today. Display only; the next real update (the write's own confirmation, or any later broadcast) always wins if it's ever different.
	4. [ ] **A lean, no-framework JS test file** (e.g. `assets/js/rollup_test.html`): plain `<script>` loading `rollup.js`, a handful of hand-written cases, a homemade `assert()`, opened directly in a browser. Cases to cover: a single leaf, several leaves, a zero-leaf branch (edge case), both `progress_calc` modes, a multi-level nested tree, and the done-cascade's own edge cases (empty children, mixed done/open siblings). No Node, no npm, no test runner — this project deliberately has neither (the e2e rig is gone for good; `mix test` is the only *automated* gate), so this stays a manual, on-demand check, not a new automated pipeline.

4. [ ] **Testing**

	1. [ ] **[Agent] Backend unit tests.** ExUnit coverage for: the extracted formula function directly (leaf-average, single-level-average, zero-leaf edge case); the chain-scoped fetch returns only the right rows (not the whole Initiative); the batched write lands in one query; the completion cascade's chain-scoped, batched behavior.
	2. [ ] **[Agent] Performance benchmark — the decision point for item 1.5.** Two separate numbers, not one:
		1. [ ] Raw batch performance: a synthetic large batch (100, 150, 200, 250 ops, several `done: true` cascades included) against the scoped-and-batched path; record real wall-clock time.
		2. [ ] Concurrency: simulate concurrent writers (not real testers — a script spawning simultaneous writes) touching disjoint subtrees of the same large Initiative, across a *range* (10, 50, 100, 200 simulated concurrent writers), and measure wait time at each level. A roughly flat curve means the synchronous version holds; a sharp upward bend means the async escape hatch (1.5) should actually get built, and shows where the real ceiling is.
	3. [ ] **[Guided] Manual pass.**
		1. [ ] Edit a leaf's progress and watch the ancestor bars update instantly, no round trip.
		2. [ ] Toggle a leaf done/undone and watch the ancestor done-state update instantly.
		3. [ ] Open the same Initiative in a second session (or have a collaborator do it); edit from the first, confirm the second's view updates live too.
		4. [ ] Confirm no visible "snap" or flicker between the predicted and confirmed values in normal use.

## Cross-cutting requirements

- **One formula, one place.** The roll-up and completion-cascade logic lives in exactly one Elixir function. SQL is only ever used to fetch rows efficiently, never to compute the math itself.
- **`computed_progress` stays a full-coverage cache, always immediately correct.** Every task keeps a readable value, matching today's API and UI surface exactly, updated in the same transaction as the triggering write — no staleness window in the baseline design.
- **Async is an available upgrade, not a starting assumption.** Item 1.5 documents the escape hatch; item 4.2's concurrency benchmark, not intuition, decides whether it's actually needed.
- **Client prediction is display-only.** The browser-side copy (item 3) is the sole deliberate exception to "one place," scoped to the acting user's own edit; it is never persisted and never treated as authoritative.
- **No new test framework.** JS verification stays a lean, dependency-free, manually-run file — consistent with this project's standing choice to keep `mix test` as the only automated gate.
