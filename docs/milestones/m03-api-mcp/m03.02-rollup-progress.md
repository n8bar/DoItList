# m03.02 — Roll-up progress & completion

_Parent: [`m03-api-mcp.md`](m03-api-mcp.md)_

Not an MCP feature — a data-layer and UX fix that Arc 3's real MCP test-drive happened to surface. Today, roll-up progress and the completion cascade (a parent flips done when every child is) are computed **eagerly on every write**: a leaf edit walks the whole ancestor chain, refetches and reconciles the **entire Initiative's tree**, and persists each changed ancestor with its own write — inside the same transaction, holding a lock on the shared Initiative root the whole time. A ~50-80 op MCP batch hit Postgres's connection-checkout limit and crashed because of this. It's also the same mechanism behind a UX pain point raised independently during M02 (and dismissed at the time, on record in `app.js`: "don't fake a number") — ordinary edits on a bigger tree visibly lag while the server does this work. Same root cause at two different scales.

`computed_progress` itself is **not going away** — it's load-bearing at every task level, not just Initiative roots: the API tree response (`serializer.ex:256`), every atomic-operation result (`operations.ex:1435`), the LiveView template's row rendering (`initiative_workspace_live.ex`, several call sites), and sort-by-progress (`tasks/sort.ex:98`) all read it directly, for any task, anywhere in a tree. Dropping or narrowing it would be a real regression to the API and the UI both. What changes is **when it gets written** — moved off the synchronous write path into an async refresh, so the column stays exactly as useful as it is today without the lock-holding cost of updating it eagerly. The roll-up **formula itself doesn't change** — same leaf-average / single-level-average math from `ProductSpec.md`.

1. [ ] **Backend: keep the cache, refresh it asynchronously instead of eagerly**

	1. [ ] **The triggering write becomes single-row.**
		1. [ ] `reconcile_progress` (`tasks.ex:2551`) stops calling `list_initiative_tasks` (`tasks.ex:55`, the whole-Initiative refetch) and stops running `persist_progress`'s (`tasks.ex:2574`) ancestor-write loop *inside* the triggering transaction. A leaf's `manual_progress`/`done` edit writes exactly one row, synchronously, as today's response depends on.
		2. [ ] Same for the completion cascade: `check_completed_ancestors`/`uncheck_done_ancestors` (`tasks.ex:1445-1468`) stop doing their nested `update_task` calls inside the triggering write.
	2. [ ] **The ancestor recompute moves to an async job, riding the same broadcast item 2 already fires.**
		1. [ ] After the triggering write commits, the leaf-change broadcast (already being sent for live-collaborator updates) also drives an async recompute of the affected ancestor chain — a detached job (e.g. `Task.start` under a supervisor), never inside the original transaction, never blocking the response already sent to the user.
		2. [ ] That job computes the chain using the recursive-CTE-fetch-then-Elixir-compute pattern (`list_descendants`, `tasks.ex:1528`, inverted for an up-walk) — scoped to just the affected chain, not the whole Initiative.
		3. [ ] It persists the result with one batched query (`Repo.update_all` with a `CASE WHEN id = ... THEN ...` fragment, or the Postgres `UPDATE ... FROM (VALUES ...)` join form) for the whole chain — not N individual `Repo.update!` calls.
		4. [ ] Same treatment for the completion cascade's ancestor status flips — same job, same scoping, same batching.
	3. [ ] **Coalesce rapid edits.** If several leaf edits land under the same ancestor chain in quick succession (e.g. a large batch), the async job should recompute that chain once, not once per leaf edit — debounce or coalesce queued jobs for the same chain before they run.
	4. [ ] **Confirm the existing readers need zero code changes** — they already read the column directly and keep doing so: the API serializer, the atomic-operations per-op result, the LiveView template's row rendering, and sort-by-progress. This item is a verification pass, not new implementation.

2. [ ] **Live collaborator updates — cheap, in-memory, still one formula**

	When someone else changes a leaf, your browser's connected LiveView process already holds the whole tree in memory — no need to wait on the database (or the async cache refresh) to tell you about it.

	1. [ ] On a leaf-change broadcast, the LiveView process recomputes the affected ancestor chain from its own in-memory tree assign (not a DB read) and sends the normal render patch — reusing the one Elixir formula function, not a new implementation.
	2. [ ] Same for the completion cascade.
	3. [ ] This broadcast is the same one item 1.2 rides for the async DB refresh — one event, two independent listeners (your screen updates live; the database catches up moments later). Confirm the payload carries enough raw info (which leaf, its new value) for both.

3. [ ] **Client-side instant prediction (your own edit only)**

	The one place this formula gets duplicated — deliberately scoped small, and never trusted as the record.

	1. [ ] **Extract the formula into a small, pure, dependency-free JS file** (e.g. `assets/js/rollup.js`) — `computeRollup(leafProgressList, mode)`, `computeDoneCascade(childStatusList)`. No DOM access, no LiveView hook dependencies, no imports beyond the file itself — trivial to read and verify in isolation, and importable by both `app.js` and the standalone test page in 3.4.
	2. [ ] **Wire it into the existing optimistic-write hooks in `app.js`:**
		1. [ ] The progress-slider change handler (currently in the `pinkFields` block) — after the leaf's own optimistic value is set, call `computeRollup` over `savingAncestors(li)` and paint each ancestor's bar with the predicted number, replacing today's `markRecomputing` indeterminate-gradient treatment.
		2. [ ] The `data-complete-toggle` click handler (`applyToggleOptimism`) — same idea, calling `computeDoneCascade` for the ancestor chain's done-state.
	3. [ ] **The client never writes this predicted value anywhere** — no new field in the write payload carries a computed ancestor value; the request only ever carries the leaf's own `manual_progress` or `done` flip, same as today. Display only; the next real update (the write's own confirmation, or any later broadcast) always wins if it's ever different.
	4. [ ] **A lean, no-framework JS test file** (e.g. `assets/js/rollup_test.html`): plain `<script>` loading `rollup.js`, a handful of hand-written cases, a homemade `assert()`, opened directly in a browser. Cases to cover: a single leaf, several leaves, a zero-leaf branch (edge case), both `progress_calc` modes, a multi-level nested tree, and the done-cascade's own edge cases (empty children, mixed done/open siblings). No Node, no npm, no test runner — this project deliberately has neither (the e2e rig is gone for good; `mix test` is the only *automated* gate), so this stays a manual, on-demand check, not a new automated pipeline.

4. [ ] **Testing**

	1. [ ] **[Agent] Backend unit tests.** ExUnit coverage for: the extracted formula function directly (leaf-average, single-level-average, zero-leaf edge case); the triggering write touches exactly one row (no synchronous ancestor writes); the async job computes and persists the correct chain, batched into one query; the coalescing behavior (item 1.3) under rapid repeated edits.
	2. [ ] **[Agent] Performance benchmark.** Two numbers, not one: the triggering write's own latency (should be flat regardless of tree size), and the async job's own duration for a large synthetic tree (should be fast — chain-scoped, batched — even though it's off the critical path, a slow async job still means a longer window of stale reads).
	3. [ ] **[Agent] Async-refresh reliability — what happens when it fails.** This is new work, not a reframed benchmark: bringing back a cache means bringing back a staleness-and-failure story.
		1. [ ] Retry logic for the async job — how many attempts, what backoff, before giving up.
		2. [ ] A periodic reconciliation sweep as a backstop (e.g. on Initiative load, or a scheduled pass) that corrects any chain whose cached value drifted and was never fixed by the normal async path — so a silently-failed job can't leave a column stale indefinitely.
		3. [ ] Test: deliberately fail the async job, confirm the backstop corrects the value within a bounded time.
	4. [ ] **[Guided] Manual pass.**
		1. [ ] Edit a leaf's progress and watch the ancestor bars update instantly, no round trip.
		2. [ ] Toggle a leaf done/undone and watch the ancestor done-state update instantly.
		3. [ ] Open the same Initiative in a second session (or have a collaborator do it); edit from the first, confirm the second's view updates live too.
		4. [ ] Confirm no visible "snap" or flicker between the predicted and confirmed values in normal use.
		5. [ ] Reload the page a few seconds after an edit and confirm the persisted value has caught up — the async refresh window is real but short.

## Cross-cutting requirements

- **One formula, one place.** The roll-up and completion-cascade logic lives in exactly one Elixir function. SQL is only ever used to fetch rows efficiently, never to compute the math itself.
- **`computed_progress` stays a full-coverage cache, not a root-only one.** Every task keeps a readable value, matching today's API and UI surface exactly — only the write timing changes (async, not eager-synchronous).
- **A live compute always outranks the cached column.** Item 2/3's in-memory and client-side values are the freshest available at any moment; the persisted column is a best-effort cache for readers that don't have live state (a fresh page load, an API/MCP call, the Initiatives index) — never assume it's instantaneously correct, only correct within a short, bounded window.
- **Client prediction is display-only.** The browser-side copy (item 3) is the sole deliberate exception to "one place," scoped to the acting user's own edit; it is never persisted and never treated as authoritative.
- **No new test framework.** JS verification stays a lean, dependency-free, manually-run file — consistent with this project's standing choice to keep `mix test` as the only automated gate.
