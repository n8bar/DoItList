# m03.02 ‚Äî Roll-up progress & completion

_Parent: [`m03-api-mcp.md`](m03-api-mcp.md)_

Not an MCP feature ‚Äî a data-layer and UX fix that Arc 3's real MCP test-drive happened to surface. Today, roll-up progress and the completion cascade (a parent flips done when every child is) are computed **eagerly on every write**: a leaf edit walks the whole ancestor chain, refetches and reconciles the **entire Initiative's tree**, and persists each changed ancestor with its own write ‚Äî inside the same transaction, holding a lock on the shared Initiative root the whole time. A ~50-80 op MCP batch hit Postgres's connection-checkout limit and crashed because of this. It's also the same mechanism behind a UX pain point raised independently during M02 (and dismissed at the time, on record in `app.js`: "don't fake a number") ‚Äî ordinary edits on a bigger tree visibly lag while the server does this work. Same root cause at two different scales.

This arc moves the computation from **eager, persisted, server-side-on-write** to **lazy, on-read** ‚Äî and adds **instant client-side prediction** for the one case that genuinely needs zero-latency feedback (your own edit). The roll-up **formula itself doesn't change** ‚Äî same leaf-average / single-level-average math from `ProductSpec.md`; only when and where it runs.

1. [ ] **Backend: lazy computation, not eager persistence**

	Stop writing progress/completion to ancestors on every mutation; compute it when it's actually needed.

	1. [ ] **Stop persisting `computed_progress` on write.**
		1. [ ] Remove `reconcile_progress`'s (`tasks.ex:2551`) call into `list_initiative_tasks` (`tasks.ex:55`, the whole-Initiative refetch) ‚Äî a leaf write no longer looks past its own row.
		2. [ ] Remove `persist_progress`'s (`tasks.ex:2574`) per-ancestor `Repo.update!` loop entirely ‚Äî no ancestor row gets written at all on a leaf edit.
		3. [ ] Confirm every write path that currently routes through `reconcile_progress` (`create_task`, `update_task`, `move_task`, the cascade functions) still writes the ACTED task's own field correctly ‚Äî only the ancestor-walk-and-persist part goes away.
		4. [ ] Verify at the DB level: after a leaf's `manual_progress` edit, exactly one `tasks` row changed ‚Äî a direct row-count check, not just "the UI looks right."
	2. [ ] **Same treatment for the completion cascade.**
		1. [ ] Remove `check_completed_ancestors`/`uncheck_done_ancestors`'s (`tasks.ex:1445-1468`) per-level nested `update_task` calls and their own persistence.
		2. [ ] Confirm a branch's `status` column (not just `computed_progress`) also stops being eagerly written on ancestors ‚Äî "is this branch done" becomes a derived, read-time fact, same as progress. This is the one place scope quietly grows past "just progress" if it's missed.
		3. [ ] The acted leaf's own done/undone flip still writes its own row as today ‚Äî unchanged.
	3. [ ] **Fresh-load computation: SQL fetches, Elixir computes.**
		1. [ ] Identify or extract the single existing Elixir function that implements the leaf-average / single-level-average formula (per `ProductSpec.md`) ‚Äî if it's currently inlined inside `reconcile_progress` rather than its own named function, pull it out now; every consumer below calls this one function, nothing else.
		2. [ ] Add a recursive-CTE fetch (the proven `list_descendants` pattern, `tasks.ex:1528`) that returns the relevant rows for a tree/subtree in one query.
		3. [ ] Wire it into `Tasks.initiative_task_tree` (or wherever a full tree read currently reads the stored `computed_progress` column) so a LiveView mount and `GET /api/v1/initiatives/:id` both compute fresh instead of reading a persisted value.
		4. [ ] Confirm the MCP resource reads (`initiative_tree.ex` and friends) need no changes of their own ‚Äî they already call through the same context function, so they inherit the fix automatically.
	4. [ ] **Sort-by-computed-progress keeps working without a persisted column.**
		1. [ ] Locate the current sort implementation (`resolve_sort`/`maybe_resort_children`, `tasks.ex:1626-1641`) and its "by computed progress" branch.
		2. [ ] For that branch: fetch the sibling group + each sibling's descendants (bounded, small), compute via the one Elixir formula function, sort the resulting small list in memory ‚Äî not in SQL.
	5. [ ] **Decide what happens to the `computed_progress` column itself** ‚Äî a real open call, not a placeholder:
		1. [ ] Option A ‚Äî drop it (a migration), since nothing reads it anymore.
		2. [ ] Option B ‚Äî keep it as a non-blocking cache, refreshed outside the write's critical path (e.g. a background job), purely as a query-convenience escape hatch.
		3. [ ] Whichever is picked, confirm nothing in the write path still blocks on updating it.

2. [ ] **Live collaborator updates ‚Äî cheap, in-memory, still one formula**

	When someone else changes a leaf, your browser's connected LiveView process already holds the whole tree in memory ‚Äî no need to touch the database to tell you about it.

	1. [ ] **Progress.** Find the LiveView's existing broadcast handlers (`handle_info({:task_updated, id}, socket)` and siblings in the workspace LiveView) and change them to recompute the affected ancestor chain from the already-loaded `@task`/tree assign ‚Äî calling the one Elixir formula function from item 1.3, not a DB query ‚Äî then send the normal render patch as today.
	2. [ ] **Completion.** Same change for whatever handler currently reacts to a status-cascade broadcast.
	3. [ ] Confirm the broadcast payload already carries (or gets) enough raw info ‚Äî which leaf, its new value ‚Äî for this in-memory recompute to work without an extra query triggered by the broadcast itself.

3. [ ] **Client-side instant prediction (your own edit only)**

	The one place this formula gets duplicated ‚Äî deliberately scoped small, and never trusted as the record.

	1. [ ] **Extract the formula into a small, pure, dependency-free JS file** (e.g. `assets/js/rollup.js`) ‚Äî `computeRollup(leafProgressList, mode)`, `computeDoneCascade(childStatusList)`. No DOM access, no LiveView hook dependencies, no imports beyond the file itself ‚Äî trivial to read and verify in isolation, and importable by both `app.js` and the standalone test page in 3.4.
	2. [ ] **Wire it into the existing optimistic-write hooks in `app.js`:**
		1. [ ] The progress-slider change handler (currently in the `pinkFields` block) ‚Äî after the leaf's own optimistic value is set, call `computeRollup` over `savingAncestors(li)` and paint each ancestor's bar with the predicted number, replacing today's `markRecomputing` indeterminate-gradient treatment.
		2. [ ] The `data-complete-toggle` click handler (`applyToggleOptimism`) ‚Äî same idea, calling `computeDoneCascade` for the ancestor chain's done-state.
	3. [ ] **The client never writes this predicted value anywhere** ‚Äî no new field in the write payload carries a computed ancestor value; the request only ever carries the leaf's own `manual_progress` or `done` flip, same as today. Display only; the next real update (the write's own confirmation, or any later broadcast) always wins if it's ever different.
	4. [ ] **A lean, no-framework JS test file** (e.g. `assets/js/rollup_test.html`): plain `<script>` loading `rollup.js`, a handful of hand-written cases, a homemade `assert()`, opened directly in a browser. Cases to cover: a single leaf, several leaves, a zero-leaf branch (edge case), both `progress_calc` modes, a multi-level nested tree, and the done-cascade's own edge cases (empty children, mixed done/open siblings). No Node, no npm, no test runner ‚Äî this project deliberately has neither (the e2e rig is gone for good; `mix test` is the only *automated* gate), so this stays a manual, on-demand check, not a new automated pipeline.

4. [ ] **Testing**

	1. [ ] **[Agent] Backend unit tests.** ExUnit coverage for: the extracted formula function directly (leaf-average, single-level-average, zero-leaf edge case); the fresh-load recursive-CTE query returns the right rows; a leaf write touches exactly one row (no ancestor writes); the completion cascade no longer persists ancestor status but still computes it correctly on read.
	2. [ ] **[Agent] Performance benchmark.** Build a synthetic large tree (seed script), time the fresh-load query directly, record a real wall-clock number ‚Äî don't just assert it's fast.
	3. [ ] **[Agent] Dropped-update safety net.**
		1. [ ] Confirm a LiveView reconnect always does a fresh authoritative reload (self-heals a missed broadcast) ‚Äî no special-case code needed if this already holds, just confirm it.
		2. [ ] Confirm the acting user's own edit reuses this app's existing timeout-based self-heal pattern (already used elsewhere for optimistic writes) if the confirming update never arrives.
	4. [ ] **[Guided] Manual pass.**
		1. [ ] Edit a leaf's progress and watch the ancestor bars update instantly, no round trip.
		2. [ ] Toggle a leaf done/undone and watch the ancestor done-state update instantly.
		3. [ ] Open the same Initiative in a second session (or have a collaborator do it); edit from the first, confirm the second's view updates live too.
		4. [ ] Confirm no visible "snap" or flicker between the predicted and confirmed values in normal use.

## Cross-cutting requirements

- **One formula, one place.** The roll-up and completion-cascade logic lives in exactly one Elixir function. SQL is only ever used to fetch rows efficiently, never to compute the math itself.
- **Client prediction is display-only.** The browser-side copy (item 3) is the sole deliberate exception to "one place," scoped to the acting user's own edit; it is never persisted and never treated as authoritative.
- **No new test framework.** JS verification stays a lean, dependency-free, manually-run file ‚Äî consistent with this project's standing choice to keep `mix test` as the only automated gate.
