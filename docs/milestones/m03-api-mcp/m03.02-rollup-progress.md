# m03.02 — Roll-up progress & completion

_Parent: [`m03-api-mcp.md`](m03-api-mcp.md)_

Not an MCP feature — a data-layer and UX fix that Arc 3's real MCP test-drive happened to surface. Today, roll-up progress and the completion cascade (a parent flips done when every child is) are computed **eagerly on every write**: a leaf edit walks the whole ancestor chain, refetches and reconciles the **entire Initiative's tree**, and persists each changed ancestor with its own write — inside the same transaction, holding a lock on the shared Initiative root the whole time. A ~50-80 op MCP batch hit Postgres's connection-checkout limit and crashed because of this. It's also the same mechanism behind a UX pain point raised independently during M02 (and dismissed at the time, on record in `app.js`: "don't fake a number") — ordinary edits on a bigger tree visibly lag while the server does this work. Same root cause at two different scales.

This arc moves the computation from **eager, persisted, server-side-on-write** to **lazy, on-read** — and adds **instant client-side prediction** for the one case that genuinely needs zero-latency feedback (your own edit). The roll-up **formula itself doesn't change** — same leaf-average / single-level-average math from `ProductSpec.md`; only when and where it runs.

1. [ ] **Backend: lazy computation, not eager persistence**

	Stop writing progress/completion to ancestors on every mutation; compute it when it's actually needed.

	1. [ ] Stop persisting `computed_progress` on write. `reconcile_progress` (`tasks.ex:2551`) no longer refetches/reconciles the whole Initiative (`list_initiative_tasks`, `tasks.ex:55`) or does the per-ancestor `Repo.update!` loop (`persist_progress`, `tasks.ex:2574`) inside the write transaction. A leaf write touches exactly one row.
	2. [ ] Same treatment for the completion cascade — `check_completed_ancestors`/`uncheck_done_ancestors` (`tasks.ex:1445-1468`) stop triggering a full nested `update_task` (its own whole-tree reconcile) per level.
	3. [ ] Fresh-load computation: a recursive CTE (the proven `list_descendants` pattern, `tasks.ex:1528`) fetches the relevant rows in one query; the roll-up **math stays in Elixir**, operating on the fetched rows. SQL fetches rows, it never computes the formula — keeps the formula in exactly one place.
	4. [ ] Sort-by-computed-progress (`ProductSpec.md`'s sort criteria) still works without a persisted column: fetch the bounded sibling group + their descendants, compute via the same Elixir function, sort the small resulting list in memory.
	5. [ ] Decide what happens to the `computed_progress` column itself — remove it, or keep it as a non-blocking cache refreshed outside the write's critical path. Not required for correctness either way; a scoping call to make when this is built.

2. [ ] **Live collaborator updates — cheap, in-memory, still one formula**

	When someone else changes a leaf, your browser's connected LiveView process already holds the whole tree in memory — no need to touch the database to tell you about it.

	1. [ ] On a leaf-change broadcast, the LiveView process recomputes the affected ancestor chain from its own in-memory tree assign (not a DB query) and sends the normal render patch — reusing the exact same Elixir formula function used for a fresh load, not a new implementation.
	2. [ ] Same for the completion cascade.

3. [ ] **Client-side instant prediction (your own edit only)**

	The one place this formula gets duplicated — deliberately scoped small, and never trusted as the record.

	1. [ ] Extract the formula into a small, pure, dependency-free JS file — `computeRollup(leaves, mode)`, `computeDoneCascade(children)`. No DOM, no LiveView hook dependencies, so it's trivial to read and verify in isolation.
	2. [ ] Wire it into the existing optimistic-write hooks (`markSaving`/`markRecomputing`/`applyToggleOptimism` in `app.js`) so ancestor bars and done-state update the instant you edit a leaf, before any round trip.
	3. [ ] The client **never writes** this predicted value anywhere — display only. The next update it receives (its own write's confirmation, or any later broadcast) always wins if it's ever different.
	4. [ ] A lean, no-framework JS test file for the formula: plain `<script>` + a handful of hand-written cases + a homemade `assert()`, opened directly in a browser. No Node, no npm, no test runner — this project deliberately has neither (the e2e rig is gone for good; `mix test` is the only *automated* gate), so this stays a manual, on-demand check, not a new automated pipeline.

4. [ ] **Testing**

	1. [ ] **[Agent] Backend unit tests.** Normal ExUnit, same as any other domain logic: the formula, the lazy-read query, the completion cascade.
	2. [ ] **[Agent] Performance benchmark.** A synthetic large tree, a real wall-clock number for the fresh-load query — don't just assert it's fast, measure it.
	3. [ ] **[Agent] Dropped-update safety net.** Confirm a LiveView reconnect always does a fresh authoritative reload (self-heals a missed broadcast); confirm the acting user's own edit reuses this app's existing timeout-based self-heal pattern if the confirming update never arrives.
	4. [ ] **[Guided] Manual pass.** Edit a leaf and watch the ancestor bars update instantly with no round trip; have a second session (or collaborator) watch the same Initiative and confirm their view updates live too; confirm no visible "snap" or flicker between the predicted and confirmed values in normal use.

## Cross-cutting requirements

- **One formula, one place.** The roll-up and completion-cascade logic lives in exactly one Elixir module. SQL is only ever used to fetch rows efficiently, never to compute the math itself.
- **Client prediction is display-only.** The browser-side copy (item 3) is the sole deliberate exception to "one place," scoped to the acting user's own edit; it is never persisted and never treated as authoritative.
- **No new test framework.** JS verification stays a lean, dependency-free, manually-run file — consistent with this project's standing choice to keep `mix test` as the only automated gate.
