# m03.05 — Documentation & maintenance

_Parent: [`m03-api-mcp.md`](m03-api-mcp.md)_

_Status: **Draft — pending operator approval.** Stub: the worklists below are a skeleton to refine before any work starts._

A published, consumer-facing **reference** for the HTTP API and the MCP server, plus a **living-docs discipline** that keeps the reference in lockstep with the contract as ops, fields, endpoints, and MCP tools are added. Motivated directly by a stale example that cost a debugging session (`progress` vs `manual_progress`): the contract is only as trustworthy as the docs that describe it, and docs that aren't maintained rot silently.

This arc runs **last and then continuously**: it lands the reference once the Arc 1 / Arc 2 / Arc 3 surfaces are stable, and the maintenance plan keeps every later change in sync. The reference content is a **spec** (lives under `docs/specs/`); this arc is the action list that produces and maintains it.

1. [ ] **API reference**

	A thorough reference for `/api/v1`, sourced from the existing module docs (`operations.ex`, `serializer.ex`) rather than reinvented.

	1. [ ] **Auth, tokens, rate limits.** The `doit_pat_` Bearer scheme, mint/revoke, and the `401` / `403` / `429` semantics (with retry hints).
	2. [ ] **The operations envelope + full op table.** Every wired `{verb, type}`, its accepted `data` keys, the `lid` / forward-reference rules, the per-op error vocabulary, and the read/write field split (e.g. read-only `progress` vs writable `manual_progress`).
	3. [ ] **Read endpoints + response shapes.** The tree, list, activity, members, and comment reads and their JSON shapes.
	4. [ ] **Home & format.** Decide where it lives (e.g. `docs/specs/api.md`) and whether it's hand-written or generated from source.

2. [ ] **MCP reference**

	Reference for the Arc 3 MCP server — lands with / after [`m03.03-mcp-server.md`](m03.03-mcp-server.md).

	1. [ ] **Tools, resources, prompts.** Each MCP capability, its schema, and how it maps onto the public HTTP API.
	2. [ ] **Setup & usage (Claude Code first).** Install / config / transport, and an end-to-end "drive an Initiative" walkthrough.
	3. [ ] **Claude Code skill for `apply_operations`.** A companion skill documenting the batch-op vocabulary (verb/type/data/`lid`) for the `apply_operations` tool specifically — sharpens first-try reliability for atomic/multi-step requests beyond what the granular tools' own self-description in `tools/list` already covers (flagged during Arc 3 scoping, m03.03).

3. [ ] **Maintenance plan (living docs)**

	The discipline that keeps the reference matched to the contract as features are added.

	1. [ ] **Doc-sync definition-of-done.** Adding or changing an op, field, endpoint, or MCP tool isn't done until its reference entry is updated — folded into each arc's Testing gate, not left to memory.
	2. [ ] **Drift guard.** A lightweight automated check that the reference matches the code (e.g. a test asserting the operations `@accepted_data_keys` against the documented op table, or a generated section), so a stale doc fails the suite rather than a user.
	3. [ ] **Ownership & cadence.** Who updates what, and a periodic reconciliation pass.

4. [ ] **Testing**

	1. [ ] **[Guided] Reference accuracy pass.** Drive a representative slice of the API and MCP straight from the published reference — every documented shape, field, and error matches the running surface.
	2. [ ] **[Agent] Drift-guard test.** The maintenance check (3.2) fails on an intentional doc/code mismatch and passes once synced.
