# m03.01 — HTTP API

_Parent: [`m03-api-mcp.md`](m03-api-mcp.md)_

_Draft — pending operator approval._

The clean, general programmatic surface for Do It List's core operations — a REST/JSON API under `/api/v1` (Q1, Q3), authenticated by per-user Bearer tokens that act as the user under their existing role (Q2). Worklists are sequenced for fan-out: the contract is pinned first, then the foundation plumbing the read and write surfaces sit on; the read surface lands first (with a smoke-client to pressure-test ergonomics) before the atomic write surface; cross-references ride both; per-token rate limiting — orthogonal to the contract — is deferred to late in the arc. The surface is the **reversible** operation set only — irreversible ops stay LiveView-only (Q6).

**Execution sequence (workflow fan-out).** Two serial spines bracket two parallel bursts — the lead agent owns the spines; each burst fans out one subagent per item, mix tests riding along:

1. **Spine** — contract (1.1) → `api_tokens` context (1.2) → pipeline + Bearer (1.3) → role checks (1.4). Both migrations (`api_tokens` and the cross-ref link table, 4.1) batch here.
2. **Fan-out** — the three reads (2.1–2.3) ∥ the account token UI (1.5, timed after Arc 9 lands to avoid the lone `account_live.ex` overlap).
3. **Spine** — the smoke-client (2.4) → the atomic dispatcher (3.1–3.3) + irreversible rejection (3.5).
4. **Fan-out** — the op-families (3.4.1–3.4.6) ∥ cross-ref create/remove (4.2).
5. **Wrap** — cross-ref reads (4.3) → rate limiting (5.1) → Testing gate (worklist 6).

1. [ ] **API foundation**

	Identity, routing, and contract; no domain logic.

	1. [ ] **JSON request/response + error conventions.** A documented envelope: success shapes, a single-error shape (`status`, `code`, `message`), and the **per-op error shape** the atomic endpoint (worklist 3) reports against — pinned first so every read and the write surface have a contract to fill. Consistent field casing and timestamp format.
	2. [ ] **`api_tokens` schema + context.** Per-user tokens: store a hash (never the plaintext — SHA-256 of a high-entropy token), a label, `last_used_at`, `inserted_at`; plaintext shown once at mint. Mint and revoke functions; tokens belong to a user and carry no extra scope (Q2 — broad-and-revocable; scoped tokens are a deferred refinement, not built here).
	3. [ ] **`/api/v1` pipeline + Bearer auth.** A router scope with a JSON pipeline; a plug that reads `Authorization: Bearer <token>`, resolves token → user, and assigns the acting user. Missing/invalid/revoked token → `401`. Updates `last_used_at`.
	4. [ ] **Role checks reused unchanged.** Authorization runs through the existing `owner` / `editor` / `viewer` checks on the acting user (`Initiatives.can_view?/can_edit?/can_admin?`) — the token adds identification only, no new authorization layer. A permission failure → `403`.
	5. [ ] **Mint / revoke UI in account settings.** A "API tokens" section on the account page: list active tokens (label, last-used), mint a new one (plaintext revealed once with a copy affordance), revoke. Matches the account page's hand-rolled style ([`m02.04`](../m02-ux-buildout/m02.04-account-details.md)). Off the API critical path — built in the fan-out, timed after Arc 9 lands (the one `account_live.ex` overlap). §6 applies: optimistic revoke, plaintext-once reveal.

2. [ ] **Read surface**

	The GET side; reads of everything the write surface and the MCP need as context. The three reads are independent — a parallel fan-out.

	1. [ ] **Whole-Initiative tree read.** `GET /api/v1/initiatives/:id` returns the nested Initiative in one response — the same shape already assembled server-side for the LiveView via `Tasks.initiative_task_tree/1` (tree of tasks with order, nesting, roll-up progress, the m02.07 index label). `GET /api/v1/initiatives` lists the acting user's Initiatives.
	2. [ ] **Activity rollup read.** `GET` for an Initiative- and subtree-level activity rollup over `activity_events` (which already carries both `initiative_id` and `task_id`) — read queries + a surface, **no schema change** (carrier table). Subtree scoping by task id; chronological, paginated.
	3. [ ] **Member + comment reads.** `GET` for an Initiative's members (with roles) and for a task's comments (including tombstones for soft-deleted ones, per Q6).
	4. [ ] **MCP smoke-client pulled forward.** A tiny throwaway client that drives the read endpoints (and, once worklist 3 lands, a write) to pressure-test API ergonomics while the contract is still cheap to change — not the Arc 2 server, just an ergonomics probe.

3. [ ] **Atomic mutation surface**

	The write side: one general endpoint over the reversible op set. The dispatcher (3.1–3.3) is the serial core; the op set (3.4) fans out per family once it lands.

	1. [ ] **`POST /api/v1/operations` — the atomic-operations endpoint (Q7).** Takes an **ordered list of operations**, applied **all-or-nothing** in one `Ecto.Multi` transaction. Adopts the JSON:API `atomic:operations` pattern.
	2. [ ] **`lid` local-id resolution.** A create may assign a local id (`lid`); later ops in the same batch reference it (create parent → create child under it). Resolve `lid` → real id within the transaction.
	3. [ ] **Per-op error reporting.** Each op reports against the per-op error shape from worklist 1 (1.1); a failure rolls the whole batch back (all-or-nothing) and identifies the offending op.
	4. [ ] **The reversible operation set (Q6).** Wire the ops over the dispatcher — one subagent per family, each its own tests; a single-item write is a batch of one.
		1. [ ] **Task ops** — task CRUD / progress / reorder / reparent.
		2. [ ] **Initiative ops** — create Initiative, Initiative update.
		3. [ ] **Comment ops** — add / edit / soft-delete-with-tombstone.
		4. [ ] **Membership ops** — add / remove / role-change.
		5. [ ] **Notification ops** — read / mark-read.
		6. [ ] **Lifecycle ops** — archive / hide, move-to-Trash (soft delete).
	5. [ ] **Irreversible ops excluded.** Permanent delete / empty-Trash, transfer ownership, and account self-management are **not** reachable here — they stay LiveView-only (Q6). The endpoint rejects them.

4. [ ] **Cross-references**

	The worklist-parity capability: ID-anchored task→task links.

	1. [ ] **Link data model.** A new task→task link table anchored on **stable task IDs** (a link survives reorder); a reference resolves to the target task's **live index label** (m02.07 §1.7) at render time, so the displayed reference never rots. Standalone schema with no dependency on worklists 2–3 — its migration batches into the foundation spine alongside `api_tokens` (1.2).
	2. [ ] **Create / remove via the atomic surface.** Adding and removing a cross-reference are reversible ops, so they're operations on worklist 3's endpoint — not their own verbs.
	3. [ ] **Read via the tree / task read.** A task's outgoing (and, where useful, incoming) cross-references surface in worklist 2's reads, each carrying the target's id and current index label.

5. [ ] **Rate limiting**

	Orthogonal to the contract — deferred here so the `hammer` dependency stays off the read/write critical path (it blocks nothing upstream).

	1. [ ] **Per-token rate limiting (Q4).** A fixed-window / token-bucket cap (N requests per window, per token) enforced in the pipeline via `hammer`; over-limit → `429` with a retry hint. Limits live in config (tunable). Tiered / per-plan quotas are out.

6. [ ] **Testing**

	Mix tests ride each implementation item; this worklist is the end-of-arc gate.

	1. [ ] **Tests.** Token mint/revoke and hash-not-plaintext storage; Bearer auth resolves token → user and rejects missing/invalid/revoked (`401`); role checks deny across `owner`/`editor`/`viewer` (`403`); rate limit trips at the cap (`429`); tree read returns the nested shape; activity rollup scopes to Initiative + subtree with no schema change; member/comment reads include tombstones; atomic endpoint applies an ordered batch all-or-nothing, resolves `lid` (create parent → child), reports per-op errors and rolls back on one failure; irreversible ops are rejected; cross-reference create/remove via the atomic endpoint, link survives a reorder and renders the live index label.
	2. [ ] **[Agent] Smoke / ergonomics pass.** Drive the read + a representative write through the smoke-client (worklist 2 item 4); confirm the contract is composable for an agent before Arc 2.
	3. [ ] **Manual API pass.**
		1. [ ] [Human] Mint a token in account settings; copy it once; a curl/HTTP call with the Bearer token reads an Initiative tree; revoking the token makes the same call fail.
		2. [ ] [Human] A `viewer`-role token can read but a write is refused; an `editor`/`owner` token writes.
		3. [ ] [Human] A multi-op batch (create parent + child via `lid`, reorder, update) applies as one; an intentionally-bad op in the batch leaves nothing applied.

## Cross-cutting requirements

- **Schema.** New `api_tokens` table (worklist 1) and the task→task cross-reference link table (worklist 4) — both migrations land together in the foundation spine. The read surface adds **no** schema (activity rollup is read queries over existing `activity_events`).
- **Reuse over reinvention.** The tree read reuses the LiveView's assembled shape (`Tasks.initiative_task_tree/1`); authorization reuses the existing role checks (`Initiatives.can_view?/can_edit?/can_admin?`); the write surface mirrors LiveView paths (Preconditions) so behavior stays consistent across UI and API.
- **Contract stability (Q3).** All additions within `/api/v1` stay additive; a breaking change earns `/api/v2`.
