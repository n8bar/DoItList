# m03.01 — HTTP API

_Parent: [`m03-api-mcp.md`](m03-api-mcp.md)_

The clean, general programmatic surface for Do It List's core operations — a REST/JSON API under `/api/v1` (Q1, Q3), authenticated by per-user Bearer tokens that act as the user under their existing role (Q2). Worklists are sequenced: the foundation is the plumbing the read and write surfaces sit on; the read surface lands first (with a smoke-client to pressure-test ergonomics) before the atomic write surface; cross-references ride both. The surface is the **reversible** operation set only — irreversible ops stay LiveView-only (Q6).

1. [x] **API foundation**

	Identity, routing, limits, and contract; no domain logic.

	1. [x] **`api_tokens` schema + context.** Per-user tokens: store a hash (never the plaintext), a label, `last_used_at`, `inserted_at`; plaintext shown once at mint. Mint and revoke functions; tokens belong to a user and carry no extra scope (Q2 — broad-and-revocable; scoped tokens are a deferred refinement, not built here).
	2. [x] **Mint / revoke UI in account settings.** A "API tokens" section on the account page: list active tokens (label, last-used), mint a new one (plaintext revealed once with a copy affordance), revoke. Matches the account page's hand-rolled style ([`m02.04`](../m02-ux-buildout/m02.04-account-details.md)).
	3. [x] **`/api/v1` pipeline + Bearer auth.** A router scope with a JSON pipeline; a plug that reads `Authorization: Bearer <token>`, resolves token → user, and assigns the acting user. Missing/invalid/revoked token → `401`. Updates `last_used_at`.
	4. [x] **Role checks reused unchanged.** Authorization runs through the existing `owner` / `editor` / `viewer` checks on the acting user — the token adds identification only, no new authorization layer. A permission failure → `403`.
	5. [x] **Per-token rate limiting (Q4).** A fixed-window / token-bucket cap (N requests per window, per token) enforced in the pipeline; over-limit → `429` with a retry hint. Limits live in config (tunable). Tiered / per-plan quotas are out.
	6. [x] **JSON request/response + error conventions.** A documented envelope: success shapes, a single-error shape (`status`, `code`, `message`), and the **per-op error shape** the atomic endpoint (worklist 3) reports against — pinned here so the write surface has a contract to fill. Consistent field casing and timestamp format.

2. [x] **Read surface**

	The GET side; reads of everything the write surface and the MCP need as context.

	1. [x] **Whole-Initiative tree read.** `GET /api/v1/initiatives/:id` returns the nested Initiative in one response — the same shape already assembled server-side for the LiveView (tree of tasks with order, nesting, roll-up progress, the m02.07 index label). `GET /api/v1/initiatives` lists the acting user's Initiatives.
	2. [x] **Activity rollup read.** `GET` for an Initiative- and subtree-level activity rollup over `activity_events` (which already carries both `initiative_id` and `task_id`) — read queries + a surface, **no schema change** (carrier table). Subtree scoping by task id; chronological, paginated.
	3. [x] **Member + comment reads.** `GET` for an Initiative's members (with roles) and for a task's comments (including tombstones for soft-deleted ones, per Q6).
	4. [x] **MCP smoke-client pulled forward.** A tiny throwaway client that drives the read endpoints (and, once worklist 3 lands, a write) to pressure-test API ergonomics while the contract is still cheap to change — not the Arc 2 server, just an ergonomics probe.

3. [x] **Atomic mutation surface**

	The write side: one general endpoint over the reversible op set.

	1. [x] **`POST /api/v1/operations` — the atomic-operations endpoint (Q7).** Takes an **ordered list of operations**, applied **all-or-nothing** in one `Ecto.Multi` transaction. Adopts the JSON:API `atomic:operations` pattern.
	2. [x] **`lid` local-id resolution.** A create may assign a local id (`lid`); later ops in the same batch reference it (create parent → create child under it). Resolve `lid` → real id within the transaction.
	3. [x] **Per-op error reporting.** Each op reports against the per-op error shape from worklist 1; a failure rolls the whole batch back (all-or-nothing) and identifies the offending op.
	4. [x] **The reversible operation set (Q6).** Wire the ops over the exposed surface: task CRUD / progress / reorder / reparent, create Initiative, Initiative update, comment add / edit / soft-delete-with-tombstone, membership add / remove / role-change, notifications read / mark-read, archive / hide, and move-to-Trash (soft delete). A single-item write is a batch of one.
	5. [x] **Irreversible ops excluded.** Permanent delete / empty-Trash, transfer ownership, and account self-management are **not** reachable here — they stay LiveView-only (Q6). The endpoint rejects them.

4. [x] **Cross-references**

	The worklist-parity capability: ID-anchored task→task links.

	1. [x] **Link data model.** A new task→task link table anchored on **stable task IDs** (a link survives reorder); a reference resolves to the target task's **live index label** (m02.07 §1.7) at render time, so the displayed reference never rots.
	2. [x] **Create / remove via the atomic surface.** Adding and removing a cross-reference are reversible ops, so they're operations on worklist 3's endpoint — not their own verbs.
	3. [x] **Read via the tree / task read.** A task's outgoing (and, where useful, incoming) cross-references surface in worklist 2's reads, each carrying the target's id and current index label.

5. [ ] **Testing**

	Mix tests ride each implementation item; this worklist is the end-of-arc gate.

	1. [x] **Tests.** Token mint/revoke and hash-not-plaintext storage; Bearer auth resolves token → user and rejects missing/invalid/revoked (`401`); role checks deny across `owner`/`editor`/`viewer` (`403`); rate limit trips at the cap (`429`); tree read returns the nested shape; activity rollup scopes to Initiative + subtree with no schema change; member/comment reads include tombstones; atomic endpoint applies an ordered batch all-or-nothing, resolves `lid` (create parent → child), reports per-op errors and rolls back on one failure; irreversible ops are rejected; cross-reference create/remove via the atomic endpoint, link survives a reorder and renders the live index label.
	2. [x] **[Agent] Smoke / ergonomics pass.** Drive the read + a representative write through the smoke-client (worklist 2 item 4); confirm the contract is composable for an agent before Arc 2. → Live end-to-end drive against the running stack: composable for an agent (lids, the echoed id map, errors with code/message/pointer/index, and cross-references resolving to live labels across a reparent). Two findings fixed: per-token rate limiting wasn't enforcing live (a wall-clock fixed window let a boundary-straddling burst evade the cap → converted to a sliding window, live-verified); and a one-batch Initiative+first-task bootstrap now works (add-task defaults the parent to the Initiative's root when no parent is given).
	3. [ ] **Manual API pass.** _Method: drive `/api/v1` with Bruno (or curl) against the running app — base URL is the host you load the app from; mint the token in **Account → API tokens** (the `doit_pat_…` plaintext shows once). In Bruno set a `{{token}}` env var and send `Authorization: Bearer {{token}}` on every request._
		1. [ ] [Human] **Setup — mint a token and wire Bruno.**
			1. [ ] In the app, open **Account → API tokens**, mint a token, copy the plaintext (shown once).
			2. [ ] In Bruno, set the base URL + a `{{token}}` env var, and add `Authorization: Bearer {{token}}` to the collection headers.
		2. [ ] [Human] **Token auth — a Bearer token reads; a missing or revoked one is rejected.**
			1. [ ] `GET /api/v1/me` → `200` with your user (the token identifies you).
			2. [ ] `GET /api/v1/initiatives` → `200` with your Initiatives; note an `id`.
			3. [ ] `GET /api/v1/initiatives/{id}` → `200` with the whole nested tree (tasks, nesting, roll-up progress, index labels).
			4. [ ] A call with no `Authorization` header, or a garbage token → `401`.
			5. [ ] Revoke the token in **Account → API tokens**, then repeat `GET /api/v1/me` → `401`.
		3. [ ] [Human] **Role gating — a viewer reads but cannot write; an editor/owner writes.** _Your token acts as you, under your role on each Initiative — so test against one Initiative where you're a **viewer** and one where you're **owner/editor** (the viewer case needs an Initiative shared to you as viewer)._
			1. [ ] On an Initiative where you're a **viewer**: `GET /api/v1/initiatives/{id}` → `200` (reads are allowed).
			2. [ ] Same token: `POST /api/v1/operations` with one `update` to a task in that Initiative → `403` (writes refused for a viewer).
			3. [ ] On an Initiative where you're **owner/editor**: the same shape of `POST` → `200`; re-`GET` the tree and confirm the change landed.
		4. [ ] [Human] **Atomic batch — all-or-nothing.** _Body is `{"operations":[<op>,…]}`; each op is `{"op":"add"|"update"|"remove", "type":"task"|"initiative"|…, "id"|"lid":…, "data":{…}}`. `add` creates (optional `lid`); a later op references it via its own `lid` or a `…_lid` key in `data` (e.g. `"parent_lid":"p"`)._
			1. [ ] `POST /api/v1/operations` with one batch: add a task (`lid:"p"`, `data:{"initiative_id":{id},"title":"Parent via API"}`); add its child (`lid:"c"`, `data:{"parent_lid":"p","title":"Child via API"}`); update the child (`lid:"c"`, `data:{"progress":50}`).
			2. [ ] → `200`; the per-op results echo each `lid` to its real `id`; re-`GET /api/v1/initiatives/{id}` shows the child nested under the parent at progress 50.
			3. [ ] Repeat with one **intentionally-bad** op inserted (e.g. an `update` on a non-existent `id`, or an irreversible op like a permanent delete) → the whole batch fails, the offending op is flagged, and re-`GET` shows **none** of the batch applied.

## Cross-cutting requirements

- **Schema.** New `api_tokens` table (worklist 1) and the task→task cross-reference link table (worklist 4). The read surface adds **no** schema (activity rollup is read queries over existing `activity_events`).
- **Reuse over reinvention.** The tree read reuses the LiveView's assembled shape; authorization reuses the existing role checks; the write surface mirrors LiveView paths (Preconditions) so behavior stays consistent across UI and API.
- **Contract stability (Q3).** All additions within `/api/v1` stay additive; a breaking change earns `/api/v2`.
