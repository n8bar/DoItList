# m03.01 ‚Äî HTTP API

_Parent: [`m03-api-mcp.md`](m03-api-mcp.md)_

The clean, general programmatic surface for Do It List's core operations ‚Äî a REST/JSON API under `/api/v1` (Q1, Q3), authenticated by per-user Bearer tokens that act as the user under their existing role (Q2). Worklists are sequenced: the foundation is the plumbing the read and write surfaces sit on; the read surface lands first (with a smoke-client to pressure-test ergonomics) before the atomic write surface; cross-references ride both. The surface is the **reversible** operation set only ‚Äî irreversible ops stay LiveView-only (Q6).

1. [x] **API foundation**

	Identity, routing, limits, and contract; no domain logic.

	1. [x] **`api_tokens` schema + context.** Per-user tokens: store a hash (never the plaintext), a label, `last_used_at`, `inserted_at`; plaintext shown once at mint. Mint and revoke functions; tokens belong to a user and carry no extra scope (Q2 ‚Äî broad-and-revocable; scoped tokens are a deferred refinement, not built here).
	2. [x] **Mint / revoke UI in account settings.** A "API tokens" section on the account page: list active tokens (label, last-used), mint a new one (plaintext revealed once with a copy affordance), revoke. Matches the account page's hand-rolled style ([`m02.04`](../m02-ux-buildout/m02.04-account-details.md)).
	3. [x] **`/api/v1` pipeline + Bearer auth.** A router scope with a JSON pipeline; a plug that reads `Authorization: Bearer <token>`, resolves token ‚Üí user, and assigns the acting user. Missing/invalid/revoked token ‚Üí `401`. Updates `last_used_at`.
	4. [x] **Role checks reused unchanged.** Authorization runs through the existing `owner` / `editor` / `viewer` checks on the acting user ‚Äî the token adds identification only, no new authorization layer. A permission failure ‚Üí `403`.
	5. [x] **Per-token rate limiting (Q4).** A fixed-window / token-bucket cap (N requests per window, per token) enforced in the pipeline; over-limit ‚Üí `429` with a retry hint. Limits live in config (tunable). Tiered / per-plan quotas are out.
	6. [x] **JSON request/response + error conventions.** A documented envelope: success shapes, a single-error shape (`status`, `code`, `message`), and the **per-op error shape** the atomic endpoint (worklist 3) reports against ‚Äî pinned here so the write surface has a contract to fill. Consistent field casing and timestamp format.

2. [x] **Read surface**

	The GET side; reads of everything the write surface and the MCP need as context.

	1. [x] **Whole-Initiative tree read.** `GET /api/v1/initiatives/:id` returns the nested Initiative in one response ‚Äî the same shape already assembled server-side for the LiveView (tree of tasks with order, nesting, roll-up progress, the m02.07 index label). `GET /api/v1/initiatives` lists the acting user's Initiatives.
	2. [x] **Activity rollup read.** `GET` for an Initiative- and subtree-level activity rollup over `activity_events` (which already carries both `initiative_id` and `task_id`) ‚Äî read queries + a surface, **no schema change** (carrier table). Subtree scoping by task id; chronological, paginated.
	3. [x] **Member + comment reads.** `GET` for an Initiative's members (with roles) and for a task's comments (including tombstones for soft-deleted ones, per Q6).
	4. [x] **MCP smoke-client pulled forward.** A tiny throwaway client that drives the read endpoints (and, once worklist 3 lands, a write) to pressure-test API ergonomics while the contract is still cheap to change ‚Äî not the Arc 2 server, just an ergonomics probe.

3. [x] **Atomic mutation surface**

	The write side: one general endpoint over the reversible op set.

	1. [x] **`POST /api/v1/operations` ‚Äî the atomic-operations endpoint (Q7).** Takes an **ordered list of operations**, applied **all-or-nothing** in one `Ecto.Multi` transaction. Adopts the JSON:API `atomic:operations` pattern.
	2. [x] **`lid` local-id resolution.** A create may assign a local id (`lid`); later ops in the same batch reference it (create parent ‚Üí create child under it). Resolve `lid` ‚Üí real id within the transaction.
	3. [x] **Per-op error reporting.** Each op reports against the per-op error shape from worklist 1; a failure rolls the whole batch back (all-or-nothing) and identifies the offending op.
	4. [x] **The reversible operation set (Q6).** Wire the ops over the exposed surface: task CRUD / progress / reorder / reparent, create Initiative, Initiative update, comment add / edit / soft-delete-with-tombstone, membership add / remove / role-change, notifications read / mark-read, archive / hide, and move-to-Trash (soft delete). A single-item write is a batch of one.
	5. [x] **Irreversible ops excluded.** Permanent delete / empty-Trash, transfer ownership, and account self-management are **not** reachable here ‚Äî they stay LiveView-only (Q6). The endpoint rejects them.

4. [x] **Cross-references**

	The worklist-parity capability: ID-anchored task‚Üítask links.

	1. [x] **Link data model.** A new task‚Üítask link table anchored on **stable task IDs** (a link survives reorder); a reference resolves to the target task's **live index label** (m02.07 ¬ß1.7) at render time, so the displayed reference never rots.
	2. [x] **Create / remove via the atomic surface.** Adding and removing a cross-reference are reversible ops, so they're operations on worklist 3's endpoint ‚Äî not their own verbs.
	3. [x] **Read via the tree / task read.** A task's outgoing (and, where useful, incoming) cross-references surface in worklist 2's reads, each carrying the target's id and current index label.

## Cross-cutting requirements

- **Schema.** New `api_tokens` table (worklist 1) and the task‚Üítask cross-reference link table (worklist 4). The read surface adds **no** schema (activity rollup is read queries over existing `activity_events`).
- **Reuse over reinvention.** The tree read reuses the LiveView's assembled shape; authorization reuses the existing role checks; the write surface mirrors LiveView paths (Preconditions) so behavior stays consistent across UI and API.
- **Contract stability (Q3).** All additions within `/api/v1` stay additive; a breaking change earns `/api/v2`.
