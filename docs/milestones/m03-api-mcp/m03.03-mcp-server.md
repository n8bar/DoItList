# m03.03 — MCP server

_Parent: [`m03-api-mcp.md`](m03-api-mcp.md)_

The thin **stdio** MCP adapter that lets an AI agent — Codex, Claude Code, desktop agents, web agents, personal assistants like Hermes and OpenClaw, to name a few examples — drive an Initiative as its working worklist (the north-star use case), once it has a local/CLI launcher: stdio needs a process to spawn it, so a purely hosted/remote client waits for the M05 remote transport, not this arc. It consumes the **public HTTP API** over the wire (Arc 1) and is **never** a shortcut into the Elixir contexts — routing MCP through the real API is what keeps "design it like an MCP consumes it" honest. This arc lands after Arc 1 and Arc 2 (roll-up progress & completion) are solid — the north-star efficiency claim below depends on Arc 2's fix, not just this arc's own build.

1. [x] **MCP server**

	The stdio adapter over the API.

	1. [x] **stdio adapter skeleton (MCP transport).** A local stdio process on `anubis_mcp` (Hex). Its own sibling mix project — no path dependency on the `doit` app, so it can't reach the Elixir contexts even by accident. Runs inside the existing `web` container via `docker compose exec`; no second container, no host install beyond Docker.
	2. [x] **Token auth via env var.** Reads `DOITLIST_API_TOKEN` and sends it as `Authorization: Bearer <token>` on every call — the MCP acts as that user under their existing role. No separate auth path.
	3. [x] **API-over-HTTP only.** Every operation is an HTTP call to `/api/v1`. No domain logic, no context access.
	4. [x] **Tools-vs-resources mapping.** One tool per {verb, type, concern} in Arc 1's op table (`DoItWeb.Api.Operations`), each a one-op batch to `POST /operations`, plus `apply_operations` mirroring the batch endpoint directly (lid forward-refs, bootstrap creates). Resources mirror the read endpoints 1:1: Initiative list, tree, activity, members, comments, `/me`.
	5. [x] **Local setup docs, per client.** `claude mcp add --transport stdio -e DOITLIST_API_TOKEN=<token> doit-list -- ./bin/mcp_server.sh` (Claude Code); any other stdio client adapts the same command + env vars (`DOITLIST_API_TOKEN` required, `DOITLIST_API_URL` optional). The script forwards stdio into the running `web` container.

2. [ ] **Batch reliability.** 

	1. [ ] **Hard cap on batch size.** `POST /api/v1/operations` (and its MCP mirror `apply_operations`) should reject an oversized batch up front, before touching the DB. Benchmark **after** Arc 2 (Roll-up progress & completion) lands — the real per-op cost, and therefore the safe number, depends on Arc 2's fix. Target the 100-250 range (matching/exceeding the JSON:API `atomic:operations` reference implementation's configurable ceiling) as a starting point.
		1. [ ] Benchmark a synthetic large batch (100, 150, 200, 250 ops, several `done: true` cascades included) against Arc 2's fixed path; record real wall-clock time.
		2. [ ] Set `@max_batch_size` in `DoItWeb.Api.Operations`, checked before the `Ecto.Multi` starts (zero DB work on rejection) — a clean 422 naming the actual count and the limit.
		3. [ ] Document the cap in `apply_operations.ex`'s MCP-facing moduledoc so an agent knows the ceiling before hitting it.
	2. [ ] **Idempotency key.** A client-side timeout for any reason (not just an oversized batch) leaves no way to tell whether the batch committed; a retry risks duplicating it.
		1. [ ] Design call: client-supplied key (e.g. an `Idempotency-Key` header, mirroring Stripe's convention) vs. server-derived — client-supplied is standard, since only the client knows "this is the same logical request as my last, failed one."
		2. [ ] New table (or column) recording `(user_id, idempotency_key)` → the original response, with a retention window.
		3. [ ] In the operations pipeline, before executing: look up `(user, key)`; if found, return the stored result unchanged instead of re-executing.
		4. [ ] After the batch resolves (success or the per-op-error rollback shape), store the result keyed by `(user, key)`.
		5. [ ] Wire an optional `idempotency_key` param into `apply_operations`'s MCP tool schema so an agent can supply one and retry safely.

3. [ ] **Testing and Verification**

[x] **[Agent] Automated tests.** 23 ExUnit tests over `Req.Test` stubs (no live server): token reaches the API as a Bearer header; every granular tool + `apply_operations` build the right batch shape; each resource read returns the right shape. Caught two real bugs: `apply_operations`'s partial-failure reply never set the MCP `isError` flag (only `ok: false` inside the JSON body) — a client checking just that flag would read a rolled-back batch as a success; and neither `reply/2` nor `reply_batch/2` handled a crash-shaped error body (a raw 500 past Phoenix's default error view, `%{"errors" => %{"detail" => ...}}`, doesn't match the app's own `{"error": ...}` envelope) — would have raised inside the adapter itself instead of surfacing the failure. ## Cross-cutting requirements

- **No new schema.** The MCP rides the Arc 1 API entirely; it adds no tables and no domain logic. (Item 2.2's idempotency-key table is an exception, scoped to that item and noted there.)
- **Thin-layer guardrail.** The adapter stays a translation layer — request shaping, tool/resource wiring, transport. Any capability gap is fixed in the API (Arc 1), not worked around in the adapter.
- **Concurrency: no new mechanism.** A conflicting write fails with Arc 1's existing specific per-op error (m03.01); no locking. (The shared-root lock contention found during this arc's testing is Arc 2's fix, not this arc's — see [`m03.02-rollup-progress.md`](m03.02-rollup-progress.md).)
