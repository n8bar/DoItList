# m03.03 — MCP server

_Parent: [`m03-api-mcp.md`](m03-api-mcp.md)_

The thin **stdio** MCP adapter that lets an AI agent — Codex, Claude Code, desktop agents, web agents, personal assistants like Hermes and OpenClaw, to name a few examples — drive an Initiative as its working worklist (the north-star use case), once it has a local/CLI launcher: stdio needs a process to spawn it, so a purely hosted/remote client waits for the M05 remote transport, not this arc. It consumes the **public HTTP API** over the wire (Arc 1) and is **never** a shortcut into the Elixir contexts — routing MCP through the real API is what keeps "design it like an MCP consumes it" honest. This arc lands after Arc 1 and Arc 2 (roll-up progress & completion) are solid — the north-star efficiency claim (verified through m03.04's gate) depends on Arc 2's fix, not just this arc's own build.

1. [x] **MCP server**

	The stdio adapter over the API.

	1. [x] **stdio adapter skeleton (MCP transport).** A local stdio process on `anubis_mcp` (Hex). Its own sibling mix project — no path dependency on the `doit` app, so it can't reach the Elixir contexts even by accident. Runs inside the existing `web` container via `docker compose exec`; no second container, no host install beyond Docker.
	2. [x] **Token auth via env var.** Reads `DOITLIST_API_TOKEN` and sends it as `Authorization: Bearer <token>` on every call — the MCP acts as that user under their existing role. No separate auth path.
	3. [x] **API-over-HTTP only.** Every operation is an HTTP call to `/api/v1`. No domain logic, no context access.
	4. [x] **Tools-vs-resources mapping.** One tool per {verb, type, concern} in Arc 1's op table (`DoItWeb.Api.Operations`), each a one-op batch to `POST /operations`, plus `apply_operations` mirroring the batch endpoint directly (lid forward-refs, bootstrap creates). Resources mirror the read endpoints 1:1: Initiative list, tree, activity, members, comments, `/me`.
	5. [x] **Local setup docs, per client.** `claude mcp add --transport stdio -e DOITLIST_API_TOKEN=<token> doitlist -- ./bin/mcp_server.sh` (Claude Code); any other stdio client adapts the same command + env vars (`DOITLIST_API_TOKEN` required, `DOITLIST_API_URL` optional). The script forwards stdio into the running `web` container.

2. [x] **Batch reliability.** 

	1. [x] **Hard cap on batch size.** `POST /api/v1/operations` (and its MCP mirror `apply_operations`) should reject an oversized batch up front, before touching the DB. Benchmark **after** Arc 2 (Roll-up progress & completion) lands — the real per-op cost, and therefore the safe number, depends on Arc 2's fix. Target the 100-250 range (matching/exceeding the JSON:API `atomic:operations` reference implementation's configurable ceiling) as a starting point.
		1. [x] Benchmark a synthetic large batch (100, 150, 200, 250 ops, several `done: true` cascades included) against Arc 2's fixed path; record real wall-clock time.
		2. [x] Set `@max_batch_size` in `DoItWeb.Api.Operations`, checked before the `Ecto.Multi` starts (zero DB work on rejection) — a clean 422 naming the actual count and the limit.
		3. [x] Document the cap in `apply_operations.ex`'s MCP-facing moduledoc so an agent knows the ceiling before hitting it.
	2. [x] **Idempotency key.** A client-side timeout for any reason (not just an oversized batch) leaves no way to tell whether the batch committed; a retry risks duplicating it.
		1. [x] Design call: client-supplied key (e.g. an `Idempotency-Key` header, mirroring Stripe's convention) vs. server-derived — client-supplied is standard, since only the client knows "this is the same logical request as my last, failed one."
		2. [x] New table (or column) recording `(user_id, idempotency_key)` → the original response, with a retention window.
		3. [x] In the operations pipeline, before executing: look up `(user, key)`; if found, return the stored result unchanged instead of re-executing.
		4. [x] After the batch resolves (success or the per-op-error rollback shape), store the result keyed by `(user, key)`.
		5. [x] Wire an optional `idempotency_key` param into `apply_operations`'s MCP tool schema so an agent can supply one and retry safely.

3. [x] **Task cross-reference UI — `%`-notation links + a Link-task picker.**

	How task→task links get created and shown in the UI. **Not a rebuild** — the substrate is built (Arc 1 / m02.07: id-anchored `task_links`, `add`/`remove link` ops + MCP tools, tree read exposing links with live labels), so this reuses `task_links` for the structured title/description edges, plus an inline id-token in each field for live rendering (comments and ephemeral chat are inline-only); net-new is the UI. Same-Initiative only; cross-Initiative deliberately out of scope. (Folded into this arc per the nitpick session, though it's product/LiveView work, not MCP.)

	1. [x] **Inline `%`-notation.** Typing `%` + a task's dotted index path — any depth (`%1`, `%1.5`, `%1.5.1.3`) — in a task's title, description, comment, or chat creates a link to that task, resolved at entry (label → stable id) and **anchored on the stable id everywhere**, then rendered with the target's *live* current label — so a `%1.1.1.1` link shows as `1.1.1.2` the moment a sibling is inserted before the target, in chat too, for as long as the message is visible. The user never *sees* or *types* the id — they reference by the `%`-label — but the id is **not** kept out of the DOM: the rendered link carries it as a data attribute, so the client cheaply re-resolves and updates the displayed label on any change (switching the numbering style, a reorder, an inserted sibling, etc.). "No user-facing ids" is about what the user sees and types, not a DOM exclusion. Persisted fields store the id in the DB; chat is ephemeral (broadcast, not persisted), so its id rides in the live message payload — same anchor-and-render behavior, just not stored past the message's life (so its label re-resolves on tree changes while shown).
		1. [x] **Storage model.** Title/description persist an inline id-token **and** a `task_links` edge (source = the owning task) feeding the API's `cross_references` / `referenced_by`; comments and chat are inline-token only (a comment isn't a task; chat is ephemeral). The inline token drives live-label rendering everywhere.
		2. [x] **Token format.** One id-anchored token shared across title / description / comment / chat that survives HTML-escaping and won't collide with prose.
		3. [x] **Client `%`-path resolver (label → id).** In `app.js`, scan a field for every `%` + dotted-index path (any depth; **multiple per field**) and resolve each independently against the initiative's current Task numbering (`index_style`) using the live tree in the DOM (`#task-<id>`, `data-task-id`, `[data-task-index]`) — no server round-trip. A `%` that doesn't resolve to a valid task stays literal; with numbering off (`none`), every `%` is literal.
		4. [x] **Deliberate literal `%` (escaping).** Standard backslash escaping: `\` makes the next char literal — `\%` → literal `%` (no link), `\\` → literal `\`; so `\\%` renders as literal `\%` when no valid reference follows, or literal `\` + live link when one does. Backslash is significant **only** before a `%` or another `\` — before any other char it stays a literal backslash (prose like `C:\path` is untouched). Escapes are applied only in the rendered output; stored/edit text keeps them.
		5. [x] **Resolve-at-entry + rehydrate.** On save/blur, swap each resolved `%`-label for its id-token before persisting; on entering edit mode, rehydrate stored id-tokens back to the `%` + current-label form, so the box shows labels, never ids.
		6. [x] **Edge sync (title/description only).** On a title/description save, diff the field's resolved `%`-set against its stored one and add/remove `task_links` rows via the existing `add`/`remove link` ops, keeping the API's `cross_references` in step; comments and chat write no edges.
		7. [x] **Chat carries the id in the ephemeral payload.** Extend `send_chat` (`initiative_workspace_live.ex:1662`) and the `.Chat` hook (`:3695`) so a chat `%`-ref's resolved id rides the broadcast message (unpersisted); the renderer re-resolves its label while shown.
	2. [x] **"Link task" picker button.** A helper button only for the fields that warrant one — the add-task title and the details-pane title/description: opens a same-Initiative task picker; on choose, inserts the reference at the cursor of whichever field last had focus (title only in add-task). Comments and chat take the `%`-notation by typing — no button there.
		1. [x] **Track last-focused eligible field.** Remember which of the add-task title / details title / details description last held focus, so the picker inserts there (add-task: title only).
		2. [x] **Same-Initiative picker.** A button opening a searchable same-Initiative task list (live label + title per row); cross-Initiative out of scope.
		3. [x] **Insert at cursor.** On choose, insert the `%`-label token at the cursor of the last-focused field; comments and chat take `%` by typing — no button.
	3. [x] **Render as a clickable link** outside the edit box, showing the target's *current* numbering label, at every `%`-bearing site (title / description / comment / chat).
		1. [x] **Inline renderer.** A client renderer that splits each rendered title / description / comment / chat body into text + `%`-link spans, replacing the plain `whitespace-pre-wrap` output (`initiative_workspace_live.ex:5667` / `:5684` / `:6709` / `:3660`); model on `.Chat`'s DOM building (`:3742`) and the `[data-copy-index]` handler (`app.js:1221`).
		2. [x] **Live label from id.** Each link carries `data-task-id` and shows the target's current label, read live from its row's `[data-task-index]` — an insert/reorder updates it in place, no numbering re-derived in JS.

		1. [x] **Delegated click.** A `%`-link click runs the existing select-and-scroll primitive — `DoitSelection.set(id, {scroll: true})` + `DoitPush("select_task", {id})` (`initiative_workspace_live.ex:2933`).

	5. [x] **Initiative-level fields (Details pane).** Extend `%`-notation to the Initiative **Name**, **Subtitle**, and **Description** — same render, entry (resolve-at-entry + rehydrate), picker, and the 3.5 save-confirm. ## Cross-cutting requirements

- **No new schema.** The MCP rides the Arc 1 API entirely; it adds no tables and no domain logic. (Item 2.2's idempotency-key table is an exception, scoped to that item and noted there.)
- **Thin-layer guardrail.** The adapter stays a translation layer — request shaping, tool/resource wiring, transport. Any capability gap is fixed in the API (Arc 1), not worked around in the adapter.
- **Concurrency: no new mechanism.** A conflicting write fails with Arc 1's existing specific per-op error (m03.01); no locking. (The shared-root lock contention found during this arc's testing is Arc 2's fix, not this arc's — see [`m03.02-rollup-progress.md`](m03.02-rollup-progress.md).)
