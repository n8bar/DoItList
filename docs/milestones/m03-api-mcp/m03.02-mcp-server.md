# m03.02 — MCP server

_Parent: [`m03-api-mcp.md`](m03-api-mcp.md)_

_Draft — pending operator approval._

The thin **stdio** MCP adapter that lets Claude Code drive an Initiative as its working worklist (the north-star use case). It consumes the **public HTTP API** over the wire (Arc 1) and is **never** a shortcut into the Elixir contexts — routing MCP through the real API is what keeps "design it like an MCP consumes it" honest. This arc lands after Arc 1 is solid, and it carries two deferred decisions (Open Questions below) that don't block Arc 1.

1. [ ] **MCP server**

	The stdio adapter over the API.

	1. [ ] **stdio adapter skeleton (MCP transport).** A small local process Claude Code launches directly; speaks MCP over **stdio** (the decided transport). No hosting, no OAuth. Remote streamable-HTTP is out of scope here — deferred to hosting (M06), and additive when it comes.
	2. [ ] **Token auth via env var.** The adapter reads the user's API token from an env var and sends it as `Authorization: Bearer <token>` on every API call — so the MCP acts as that user under their existing role (Q2). No separate auth path.
	3. [ ] **API-over-HTTP only.** Every operation is an HTTP call to `/api/v1` (Arc 1). The adapter holds no domain logic and never touches the contexts — if the API can't do it, the MCP can't either.
	4. [ ] **Tools-vs-resources mapping.** Map the API's read/mutate split onto MCP: **mutations → tools** (the atomic-operations surface and its ops), **read context → resources** (whole-Initiative tree read, activity rollup; member/comment reads as needed). Largely mechanical given Arc 1's split (see Open Question 2).
	5. [ ] **`claude mcp add` local setup.** A documented one-command local install (`claude mcp add …`) wiring the adapter with the per-user token env var, so the local Claude Code experience just works.

2. [ ] **Testing**

	The end-of-arc gate.

	1. [ ] **Tests.** Adapter calls hit `/api/v1` (never the contexts) — assert via the API boundary; token from env reaches the API as a Bearer header and acts as the right user; a tool invocation maps to the right atomic operation and surfaces per-op errors back to the client; a resource read returns the tree / activity rollup shape.
	2. [ ] **Manual MCP pass.**
		1. [ ] [Human] `claude mcp add` installs the server with a personal token; Claude Code lists its tools and resources.
		2. [ ] [Guided] Drive a real reshape (create a sub-tree, reorder, update progress) through the MCP and confirm it lands in the web UI live; revoking the token cuts MCP access.
		3. [ ] [Human] Confirm the bar from the north-star use case: driving the tree through MCP is at least as efficient as editing the equivalent `.md` hierarchy.

## Open Questions

These are the two deferred Arc 2 decisions — settle them when building this arc; they don't block Arc 1.

1. **Server runtime / library.** An endpoint in the same Phoenix app vs. a separate process; an Elixir MCP library (e.g. Hermes) vs. another runtime. Gated by the stdio/local-smoothness requirement — whatever we pick **must** support the stdio/local path without forcing a remote/OAuth setup. Leaning: a separate thin stdio process so the MCP can only reach the API over HTTP, never the contexts.
2. **Tools-vs-resources mapping detail.** Which exact operations are MCP *tools* (actions/mutations) and which are *resources* (read-only context — e.g. the activity rollup). Informed by Arc 1's already-decided read/mutate split, so largely mechanical, but the final per-operation assignment is pinned here.

## Cross-cutting requirements

- **No new schema.** The MCP rides the Arc 1 API entirely; it adds no tables and no domain logic.
- **Thin-layer guardrail.** The adapter stays a translation layer — request shaping, tool/resource wiring, transport. Any capability gap is fixed in the API (Arc 1), not worked around in the adapter.
