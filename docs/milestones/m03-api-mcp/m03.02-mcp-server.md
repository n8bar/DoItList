# m03.02 — MCP server

_Parent: [`m03-api-mcp.md`](m03-api-mcp.md)_

The thin **stdio** MCP adapter that lets an AI agent — Codex, Claude Code, desktop agents, web agents, personal assistants like Hermes and OpenClaw, to name a few examples — drive an Initiative as its working worklist (the north-star use case), once it has a local/CLI launcher: stdio needs a process to spawn it, so a purely hosted/remote client waits for the M06 remote transport, not this arc. It consumes the **public HTTP API** over the wire (Arc 1) and is **never** a shortcut into the Elixir contexts — routing MCP through the real API is what keeps "design it like an MCP consumes it" honest. This arc lands after Arc 1 is solid.

1. [ ] **MCP server**

	The stdio adapter over the API.

	1. [ ] **stdio adapter skeleton (MCP transport).** A small local process the client launches directly, on `hermes_mcp` (Hex; stdio transport + server-side tool definitions). Runs as its own OTP release, never an endpoint in the Phoenix app, so it can't import the Elixir contexts even by accident. No hosting, no OAuth. Remote streamable-HTTP is out of scope here — deferred to hosting (M06); `hermes_mcp` also supports Phoenix/HTTP-SSE, so it carries forward to it.
	2. [ ] **Token auth via env var.** The adapter reads the user's API token from an env var and sends it as `Authorization: Bearer <token>` on every API call — so the MCP acts as that user under their existing role (Q2). No separate auth path.
	3. [ ] **API-over-HTTP only.** Every operation is an HTTP call to `/api/v1` (Arc 1). The adapter holds no domain logic and never touches the contexts — if the API can't do it, the MCP can't either.
	4. [ ] **Tools-vs-resources mapping.** Granular, intent-level tools (`create_task`, `complete_task`, `update_progress`, `move_task`, `add_comment`, …), each building a one-op batch for `POST /operations`. Plus one `apply_operations` tool mirroring the batch endpoint directly, for atomic/multi-step requests (forward-refs via `lid`, bootstrap-style creates). Resources mirror the read endpoints 1:1: the Initiative list, an Initiative's tree, its activity, its members, its comments, and `/me`.
	5. [ ] **Local setup docs, per client.** A documented one-command local install for Claude Code (`claude mcp add …`, wiring the adapter with the per-user token env var) plus the generic stdio config block any other MCP client (Codex, a desktop agent, etc.) can adapt directly.

2. [ ] **Testing**

	The end-of-arc gate.

	1. [ ] **Tests.** Adapter calls hit `/api/v1` (never the contexts) — assert via the API boundary; token from env reaches the API as a Bearer header and acts as the right user; each granular tool and `apply_operations` map to the right batch shape and surface per-op errors back to the client; a resource read returns the tree / activity rollup shape.
	2. [ ] **Manual MCP pass.**
		1. [ ] [Human] Install via `claude mcp add` (or the generic stdio config, for another client) with a personal token; the client lists the adapter's tools and resources.
		2. [ ] [Guided] Drive a real reshape (create a sub-tree, reorder, update progress) through the MCP and confirm it lands in the web UI live; revoking the token cuts MCP access.
		3. [ ] [Guided] Edit the same Initiative from the web UI while the agent is mid-task; confirm the agent's next write either lands cleanly or fails with a specific, actionable error — never silent corruption.
		4. [ ] [Human] Confirm the bar from the north-star use case: driving the tree through MCP is at least as efficient as editing the equivalent `.md` hierarchy.

## Cross-cutting requirements

- **No new schema.** The MCP rides the Arc 1 API entirely; it adds no tables and no domain logic.
- **Thin-layer guardrail.** The adapter stays a translation layer — request shaping, tool/resource wiring, transport. Any capability gap is fixed in the API (Arc 1), not worked around in the adapter.
- **Concurrency: no new mechanism.** A conflicting write fails with Arc 1's existing specific per-op error (m03.01); no locking. Tools that reorder/move re-fetch current state first (position-sensitive).
