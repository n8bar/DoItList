# m02.03 — Task Tree: Reorganization & Layout
_Parent: [`m02-ux-buildout.md`](m02-ux-buildout.md)_

Reshaping and laying out the task tree within an Initiative: drag / keyboard reorganization (per [`ProductSpec.md`](../../ProductSpec.md) § Reorganization), the single-root-task model, and responsive tree layout (per [`ProductSpec.md`](../../ProductSpec.md) § Task Tree Display). Cross-Initiative moves are out of scope.

## 1 — Drag & drop

1. [x] **Drag handle on each task row.** A 3-vertical-dots glyph at the far left of the row (left of the chevron / checkbox) is the only place a drag starts. Prevents accidental drags from row-body clicks. Handle has its own touch target meeting the 44×44 px rule from UX_GUARDRAILS.
2. [x] **Persistence + roll-up recompute.** Every drop computes new `parent_id` + `sort_order`, persists in a single transaction, and recomputes ancestor progress for both the old and new parent chains. Failures roll back the entire drop and surface a flash; the UI reverts to pre-drag state. Lands as a `Tasks.move_task/3` context function before any gesture wiring so items .03.01.03–.07 can call into a real backend rather than stubs.
3. [x] **Vertical drag → reparent.** Dropping task A onto task B's row makes A a child of B (appended at the end of B's children). Drop target highlights during hover. After commit, ancestor progress recomputes for both old and new parent chains (via .03.01.02).
4. [x] ~~**Horizontal drag → indent / dedent.**~~ While dragging, horizontal cursor position controls depth: dragging right past a threshold indents A under its previous sibling; dragging left dedents A toward an ancestor's level. A subtle indent guide tracks the cursor so the target depth is visible before drop. _Superseded by .03.03.06._
5. [x] ~~**Snap-back zone.**~~ A horizontal cursor band inside the row that means "no change" — letting go inside that band aborts the move with no commit. Lets the user back out mid-drag without committing a wrong drop. _Superseded by .03.03.06._
6. [x] **Keyboard alternative — reuse task-selected state.** A task is "selected" when its Details panel is open (the existing selection model from Arc 2 item 6 / 21). While a task is selected, keyboard shortcuts reorder and reparent that task: `Alt+↑/↓` reorders within siblings; `Alt+←/→` dedents / indents. No mouse needed; no separate "drag focus" concept — selection IS the keyboard focus for reorganization.
7. [x] **Cross-list moves within the same Initiative.** Dragging across List trees changes A's root List. Cross-Initiative moves remain out of scope (ProductSpec § Reorganization). Verified by the cross-list test in `test/doit/tasks_test.exs` (covers source / dest root recompute and the Initiative-level aggregate change).
8. [x] **Mobile touch — long-press to start drag.** Long-press (~400 ms) on the drag handle starts the drag; the rest of the gesture mirrors mouse behavior. Short taps on the handle do nothing — the handle is drag-only, not a button.
9. [x] **Completion-state confirmation modal.** Create + move actions that would auto-flip an ancestor's `status: done` (in either direction) prompt a confirmation listing the affected tasks before committing. Without this, a drag can silently uncomplete a parent (or auto-complete one) — too surprising. `Tasks.preview_create/2` and `Tasks.preview_move/3` dry-run via transaction rollback, snapshot ancestor chains before / after, and classify as scenario 1 (would uncomplete), 2 (would complete), 3 (both), or `nil` (no flip). The modal renders in 1/2/3; `nil` commits straight through. Driven by a `:pending_action` socket assign + `confirm_pending` / `cancel_pending` events. Side effect: branch progress reconciliation moves out of `Progress.compute` (which used to short-circuit `status: done → 100`) and into `Tasks` proper — branches always derive from children, so a parent that gains an incomplete child reflects truth instead of masking it. Branches still reach 100 once all leaves do.
10. [x] ~~**Drop-zone two-mode redesign.**~~ A two-mode model selected by vertical distance from the source row's center: horizontal mode (within ±~1.5× row height of source center) treats horizontal cursor position as indent / dedent — .03.01.04 + .03.01.05 behaviors apply; vertical mode (cursor moved off the source's vertical slot) treats the row under the cursor as anchor, drops as child-of-anchor — .03.01.03 behavior applies. Wider snap-back band (64 px each side) in horizontal mode. Tunable constants live in `assets/js/app.js`. _Superseded by .03.03.06 — only the vertical-mode half survives; horizontal mode and its tuning constants are removed._
11. [x] **Suppressible, fully-styled confirmations.** Route every confirmation through one styled modal and let the user silence the non-destructive ones.
    - **One modal, all confirms.** Fold every confirmation into the `completion_confirm` modal's `pending_action` mechanism: existing `:move` / `:create` (completion-flip) and `:cascade_sort`, plus new `:cascade_complete` / `:cascade_incomplete` (branch checkbox) and `:delete_task` / `:delete_initiative`. The two native `data-confirm`s and the `DeleteTask` JS `confirm()` are removed — no native dialogs remain.
    - **Confirm classes**, each its own suppression key: `completion-flip` ("completion changes"), `cascade-sort` ("large branch reorgs"), `cascade-complete` ("branch completion changes"). The cascade-sort body reads e.g. _"This is a large branch reorg affecting X tasks."_ Deletes (`:delete_task`, `:delete_initiative`) are their own modal kinds but **not** a suppressible class.
    - **"Don't show this again for {class}"** checkbox on the three suppressible classes only, the wording naming the class (e.g. "Don't show this again for large branch reorgs"). Deletes never carry it.
    - **Per-class suppression in localStorage** — `doit:confirm-skip:<class>`. A mount hook reads the flags and pushes them to the LiveView (`@confirm_skips`); a suppressible action checks the assign and commits straight through instead of opening the modal. Checking the box on Proceed writes the flag (localStorage + the assign). Per the localStorage-now convention (.06.2 / BACKLOG), a future profile/settings page clears these.
    - **Optimistic delete survives.** Routing delete through the modal keeps the .03.03.08 optimistic row-removal — it fires on Proceed, not the (now-gone) JS confirm.
12. [ ] **Confirm modals acknowledge the click** *(Draft — pending operator approval)*. Restores UX_GUARDRAILS 6.1/6.2 — today Proceed gives no feedback and the modal sits frozen until the write finishes.
    - **Pressed state.** `active:` styles on the modal's buttons so the click visibly lands.
    - **Instant disable.** `phx-disable-with` on Proceed — the button changes state the moment it's clicked, and double-submits die for free.
    - **Optimistic dismiss.** The modal hides on Proceed before the round trip (`JS.hide` alongside the submit); the maybe-write hue already shows the write in flight, and the server render reconciles. Cancel already feels instant (client `phx-click-away` aside, it's a tiny no-op event).

## 2 — Sorting

1. [x] **Sort-mode storage + inheritance.** Add nullable `sort_mode` column and non-null `sort_reverse` boolean (default false) to `tasks` and `initiatives`. `sort_mode` values: `"manual"`, `"alphabetical"`, `"completion"`, `"computed_progress"`, `"priority"`, `"created"`, `"updated"`; `nil` = inherit. `"weight"` is supported by the engine but not surfaced in the UI yet — deferred. `sort_reverse` flips each mode's natural direction. Helper `Tasks.resolve_sort/1` walks parent → root → initiative → fallback `{"manual", false}` and returns `{mode, reverse}` so direction always travels with the mode that owns it. Pure, unit-tested.
2. [x] **Sibling sort engine.** Pure function `Tasks.Sort.apply/2`: takes children + resolved mode, returns the list with new `sort_order` values. No DB. `"manual"` is a no-op. Stable sort with `id` as tiebreaker. Empty / single-element lists return unchanged. Tested standalone.
3. [x] **Sort menu in Details panel.** "Sort children by" dropdown in the Details panel for branch tasks only. Options: `Inherit` (default), criteria from .03.02.01 (no `Weight` — deferred), `Manual` at the bottom. A "Reverse" checkbox sits next to the dropdown — disabled when the resolved mode is `Inherit` or `Manual`. Selecting a criterion or toggling reverse → set `sort_mode` + `sort_reverse`, run engine (.03.02.02), persist new `sort_order`, broadcast. UI sits below the "Computed from children: X%" line. Auto-sorted parents also re-sort their children automatically whenever a child's sort-key field changes (status, computed_progress, priority, title) — `Tasks.maybe_resort_children/1` is called at every mutation site (create / update / move / recompute walk-up) so the order stays in sync with live data.
4. [x] **Resort all posterity.** "Apply to all descendants" link in the sort menu (.03.02.03). `Tasks.cascade_sort/4` walks the subtree, sets `sort_mode` + `sort_reverse` on the root and every descendant branch in one transaction, and runs the engine at each branch via `set_sort_body`. One `sort_cascaded` activity event on the root; one broadcast. Confirm modal fires when the subtree contains more than 10 branches (counts root + descendant branches); reversible only via Arc 6 undo.

## 3 — Drag placement & refinements

1. [x] **Drag drop-band semantics.** Row-internal drop bands in `assets/js/app.js`. Center 50% → Reparent (.03.01.03 behavior). Top 25% → Sibling reorder above. Bottom 25% → Sibling reorder below. Edge bands render a placeholder in the destination slot; center keeps the row-highlight.
2. [x] **Sibling reorder mode flip.** Any reorder action (drag bands .03.03.01, root overlay .03.03.03, or keyboard `Alt+↑/↓`) sets parent `sort_mode = "manual"` and overwrites `sort_order` with the new sequence. Same transaction as the reorder; broadcast carries both.
3. [x] **Top + bottom root-level overlay drop zones.** Two slim overlay drop targets, rendered only during drag — one above the first root, one below the last. Non-root source → Promotion (lands at top or bottom of root list). Root source → root-list reorder. Hook adds them on `beginDrag`, removes on `cleanup`.
4. [x] **New-task placement defaults.** Update `Tasks.create_task/2` and `Tasks.move_task/3`:

    - Auto-sorted parent → run engine (.03.02.02); insert at sort position.
    - Manual parent, via entry form → form's slot (existing `parent_id` + sibling-position params already handle).
    - Move-in to a different parent → top of new parent (was: appended end). Auto-sort overrides.

    Tests expanded in `tasks_test.exs` and `preview_test.exs`.
5. [x] **Manual progress on branches — preserve, disable, explain.** On branches, render the `manual_progress` slider as `disabled` at the dormant value (so the preserved value is visible). Below it, muted hint: _"Ignored — this task has subtasks."_ Info icon opens a popover: _"Progress on a task with subtasks is calculated from its subtasks instead of being set manually. Your manual value is kept and will start being used again if you remove all subtasks."_ Keep the existing "Computed from children: X%" line.

    Builds a reusable `<.info_hint>` component in `CoreComponents` (popover-on-click, reuse-ready for future "explain UI semantics" uses — weight, leaf-done snap, roll-up formula).

    Two files: `core_components.ex`, `initiative_show_live.ex` (`task_editor/1`). No `Tasks` / schema changes. Light + dark visual review.
6. [x] **Vertical-only drag — horizontal mode rolled back.** Single-mode drag: cursor picks the row under it (anchor); drop = child-of-anchor, appended. No horizontal sensitivity, no indent guide, no snap-back. Row-highlight stays. Supersedes .03.01.04, .03.01.05, and the horizontal half of .03.01.10. Sibling reorder via drag is covered by .03.03.01–.04.
7. [x] **Smooth drag highlight + drop-zone geometry.** Stable, unambiguous drop feedback with depth kept off the horizontal axis (.03.01.04–.05 stay rolled back). Refines completed .03.03.01. Touches `assets/js/app.js` (band geometry), the `task_node/1` template (tail padding), and `assets/css/app.css`.
    - **No special-casing for the progress bar.** Bands are thin strips at the row's top / bottom edges, so the whole middle — progress underbar included — falls in center; the bar is treated like any other part of the row.
    - **Center = reparent as first child** (per .03.03.04): dropping on a row's body makes the source that row's first child.
    - **Edge bands hug the row.** Thin top / bottom strips at the row's own edges; the indicator renders at the hovered row's edge, never below its subtree.
    - **Top band = sibling immediately above** this task — always available; the "before me" slot is adjacent to every row's top edge.
    - **Bottom band = sibling immediately below**, but only on rows with nothing visible beneath them (leaves and collapsed branches). An expanded branch has no bottom band — its lower area is all center.
    - **Tail padding = last child.** Every expanded branch reserves a band-height drop strip after its last child; dropping there appends the source as that branch's last child. Always present (no reflow when a drag starts), subtle when idle, highlighted during a drag. At nested tails the strips stack — one per ancestor level, innermost top → outermost bottom, each indented to its branch's child level (a leftward staircase) — and the target is chosen by vertical position.
8. [x] **Optimistic task writes, with a "saving" pink hue.** Every task write gives instant feedback, not gated on the round-trip; affected elements get a faint, mostly-transparent pink hue that clears when the DB write lands. Covers **all** write paths — reorder/sort (drag, `Alt+arrow`, sort menu / cascade), **completion toggles** (leaf checkbox + branch cascade), **attribute edits** (priority / weight / assignee via the Details form and the `P` / `W` / `A` keyboard steppers), and **deletion** (red button / `Del`).
    - **Pink exactly what the DB changes.** The hue marks only the rows / containers the write actually persists:
        - Reorder/sort don't touch progress (order-independent) → the reordered children + their `<ul>` container + the parent (whose `sort_mode` flips to manual). No ancestor pinking.
        - Completion + weight changes roll up → the toggled / edited task **and** every ancestor whose progress or status the write recomputes.
        - Priority / assignee don't roll up → just the edited row (plus its container if an auto-sort reorders siblings on the changed key).
    - **Truly optimistic (move the DOM now):** drag, `Alt+arrow` reorg, and the leaf checkbox — apply the visible change immediately, pink the affected set, let the server's re-render clear the hue (client-added class, so morphdom strips it). Revert the DOM + flash on a failed write.
    - **Pink-during-round-trip (server computes, client only hues):** sort menu / cascade, branch-cascade completion, and attribute edits — the client can't cheaply predict the result (sort criteria, cascade fan-out, rollup), so it marks the affected set "saving," then the server re-renders authoritative and the hue clears.
    - **Deletion is its own shape — remove, don't pink.** The deleted row (and its subtree) vanishes from the DOM immediately rather than turning pink; its surviving ancestor chain pinks during the round-trip while progress recomputes. On a failed write, re-insert the row and flash. This is what makes a large-subtree delete feel instant instead of waiting on the cascade + re-render.

## 4 — Task-tree layout

1. [x] **Responsive task-tree width.** Deliver the ProductSpec "Task Tree Display" standard: readable rows, scroll over squeeze, uniform roots, wrapping titles. Touches `initiative_show_live.ex` (`task_node/1` + the tree wrapper), `assets/js/app.js` (a `TreeWidth` hook), `assets/css/app.css`.
    - **Content floor + horizontal scroll.** Floor each task's content at a usable minimum width; the task-tree region (not the page chrome) scrolls horizontally when content exceeds the available width.
    - **Measured uniform width.** A `TreeWidth` hook computes one width `W = max(column width, deepest *visible* indent + floor)` for the whole initiative and applies it to the scroll-content wrapper, so every row — roots included — renders at `W` (no ragged right edge). Recompute on mount, resize, collapse/expand, and tree mutations, debounced to an animation frame.
    - **Titles wrap, never truncate.** Titles wrap within their row's `W − indent`; the attribute-chip group clips only as a last resort.
    - **Collapse shrinks the tree.** Collapsing a deep branch lowers the deepest *visible* indent, so `W` shrinks and the horizontal scroll recedes.
2. [x] **Drag edge auto-scroll.** Dragging near the top / bottom edge of the scroll area auto-scrolls it (speed ramps with depth into the edge zone) so off-screen drop targets are reachable without releasing; the drop highlight re-resolves on each scroll frame. `assets/js/app.js` only.
3. [ ] **Incremental tree updates** *(Draft — pending operator approval)*. Restores ProductSpec § Collaboration Model (propagation work scales with the change, not the tree). Today every write and every PubSub broadcast runs `load_tree` (full reload + full re-render + full diff for every connected member).
    - **Scope the reload.** A change to task X reloads and re-renders X's affected lineage, not the whole initiative; broadcasts carry enough (task id + kind) for receivers to do the same.
    - **Keep it boring.** Prefer narrowing assigns/diffs (or LiveView streams for the root list) over inventing a patch protocol; correctness stays anchored by the existing reconcile tests.
    - **Measure before/after** with the §8-style timing pass (per-event ms on a deep tree) so the win is a number, not a feeling.

Sort menu extracted to a shared `<.sort_menu>` component reused by the Task pane. Members stay in their own panel (not folded in). Reuses `set_sort` / `cascade_sort` on the root task — no separate Initiative-sort backend.

## 7 — Details Pane & Keyboard shortcuts

## Cross-arc note

The drag infrastructure picked here (library, hook abstraction, motion conventions, drop-feedback visuals) needs to be **reusable for Arc 5 item 11** — drag-a-person from the cross-Initiative people pane onto an Initiative entry in the left rail. Same primitives, different source / target. Pick something that can host both gestures without forking; favor a small custom hook over a heavy library if the heavy library only fits one of the two cases.

## Implementation Notes

- **Library vs. roll-your-own.** Custom `phx-hook` over pointer events. Sortable.js wrappers don't extend cleanly to the cross-pane Arc 5 item 11 case; one small custom hook serves both gestures without forking.
- **Sort order.** Use float-style ordering (`sort_order` as a numeric column) so single-row updates are common and full re-numbering is rare. Re-number lazily when ordering precision degrades.
- **Drop target visuals.** Center-band drops highlight the anchor row (.03.01.03); edge-band drops render a placeholder in the destination slot (.03.03.01). Both respect motion-reduce.
- **Optimistic vs. server-confirmed.** Render the move client-side immediately on drop; if the server transaction fails, revert and flash. Don't wait on the round-trip to update UI.
- **Cross-list within-Initiative bookkeeping.** When A moves between Lists, both the source List and the destination List need ancestor-progress recompute (their root-task progress changes), and the Initiative-level aggregate underbar (Arc 2 item 7) updates accordingly.
