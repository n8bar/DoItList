# m02.05 — Wide-Width Layout
_Parent: [`m02-ux-buildout.md`](m02-ux-buildout.md)_

Make the app earn its width on larger displays. Two phases inside one arc:

1. **Responsive groundwork (items 1–3).** The existing two-pane layout (main + right-rail) opens up at `xl:` and `2xl:` so we stop wasting horizontal real estate on wide monitors.
2. **Unified ultrawide layout (items 4–11).** Above the widest breakpoint, a left rail joins the layout — it IS the Initiatives index, plus a cross-Initiative people pane. The whole account is in chrome; the main column is the current Initiative's task tree; the right rail is the existing Details slot. People in the people pane can be dragged onto an Initiative entry to add them as a member.

Operator-approved 2026-05-09. Items below are scoped; checkboxes flip to `[x]` as each ships.

## Items

1. [x] **Container max-width refresh.** Today the Initiative-show and Initiatives-index pages live inside `max-w-6xl` (1152 px), leaving large empty gutters at `xl:`/`2xl:`. Widen (or remove) the cap at those breakpoints so the layout breathes. Header `max-w-6xl` follows the body so the chrome stays visually aligned with content. Pick a widening that keeps line-length readable for body text — not unbounded.
2. [x] **Right-rail sizing at `xl:`/`2xl:`.** Right rail is a fixed 360 px from `lg:` up. Grow it stepwise at `xl:` and `2xl:` so Task details / Initiative editor / Members aren't cramped on wide displays — and so the rail stays visually proportional once the main column widens.
3. [x] **Task-row breathing room on wide screens.** Larger horizontal padding and gutter spacing at `xl:`/`2xl:`; allow the task description to wrap to a **two-line clamp with ellipsis** (instead of `truncate`) once room exists — two lines keeps row heights predictable; consider a slightly larger title size at depth 0 for List rows on wide screens. Keep the row composable single-line on narrow.
4. [x] **Triple-pane layout at an ultrawide breakpoint.** Pick a breakpoint above `2xl:` at which the layout becomes left-rail · main · right-rail — sized to where three comfortable columns fit (~1800–2000 px), not to when a monitor counts as "ultrawide" (~2560 px). Tailwind has no built-in `3xl:`, so add a custom one or use a min-width media query. Below the breakpoint, the layout is the wide-tuned two-pane from items 1–3. Crossing the threshold causes no visible jank.
5. [x] **Left rail — Initiatives list (the index, in chrome).** The rail isn't a bespoke widget — it's the Initiatives index, rendered in compressed form for the rail width. Same data, same components where practical. Each entry: grove icon, Initiative name, role badge (consistent with Arc 2 item 26), small progress indicator. The current Initiative is visually highlighted.
6. [x] **Center "pick an Initiative" prompt + right-pane "Assigned to Me" placeholder when no Initiative is selected.** At `/initiatives` on ultrawide, the rail shows the list; the **center** shows a "pick an Initiative" prompt (not the card list — the rail covers that); and the **right (third) pane** — where Details sits on an open Initiative — holds a titled **Assigned to Me** placeholder, the reserved home for the cross-Initiative task list Arc 7 ([`m02.07`](m02.07-personal-workspace.md)) builds out (instead of sitting empty). Below the ultrawide breakpoint, `/initiatives` keeps today's behavior (the card list in main, no rail).
7. [x] **Click the current Initiative on the rail to close it.** Clicking the highlighted (current) Initiative entry navigates back to `/initiatives` — i.e. closes it, returning to the empty-state main column from item 6. Clicking any other Initiative entry navigates to that Initiative's show page as usual.
8. [x] **Left rail — Collaborators pane.** Labeled **My Collaborators** in the UI. Below the Initiatives list. Aggregates everyone the current user shares any Initiative with, deduplicated. Default sort: most-shared-Initiatives-first. Each row: avatar (with a **global online dot** — lit whenever that person is connected to the app *anywhere*, not just in a shared Initiative), name, count of shared Initiatives. When the current user shares no Initiative with anyone, the pane renders nothing — no empty-state hint.
9. [x] **Click a person to add or remove from the current Initiative.** Click surfaces a small menu / action with the contextual choice: "Add to <current Initiative>" if they aren't a member, "Remove from <current Initiative>" if they are. This is the keyboard / a11y / non-drag path; honored at every breakpoint where the people pane appears. If no Initiative is selected (main column is the empty state), the click is a no-op (or surfaces a hint that an Initiative needs to be open first). Add uses the **viewer** role by default — the conservative choice for a one-gesture add with no role step; the Members panel is where the role is bumped after the fact.
10. [x] **Drag a person onto an Initiative entry to add them.** Direct-manipulation shortcut on top of item 9. Drag a person row from the people pane onto any Initiative entry in the rail to add them as a member of that Initiative (same default **viewer** role as item 9). If they're already a member, the drop is a no-op with a brief visual indicator. Drag-to-remove is intentionally NOT a gesture — removal is click-only via item 9. Reuses the drag infrastructure from Arc 3 (see the cross-arc note in `m02.03`).
11. [x] **Mobile / narrow flyout for the left rail.** Below the ultrawide breakpoint, the left rail collapses out of the layout. On `/initiatives/:id`, a flyout affordance appears (mirroring the right-rail flyout from Arc 2 item 16) that brings up the left rail as an overlay. The affordance does NOT appear on `/initiatives` itself — there, the index IS the main column, so the rail would be redundant. Drag-to-add (item 10) is desktop-only; touch users use click (item 9).

## Cross-cutting requirements

- **No new schema for the layout / people-pane.** The people-pane data comes from a single Ecto query joining `initiatives` → `initiative_members` → `users`; cache or precompute later only if measured to be slow. (Scoped to the layout work — the co-assignees omission, item 12.1, does add its own `task_co_assignees` table.)
- **Doesn't conflict with per-Initiative Members panel.** The right-rail Members panel (today's Initiative-scoped membership UI) stays as the canonical "who's on THIS Initiative" surface and the place to change roles after the fact. The left-rail people pane is a different question — "who do I work with across everything?" — so it's a different answer.
- **No layout shift across any breakpoint.** Resizing across `xl:`, `2xl:`, and the ultrawide threshold is smooth — no layout jank, no scroll-position loss, no flashes. Honors `UX_GUARDRAILS` B.1.
- **Polished light + dark at every breakpoint.** Both themes pass visual review at every breakpoint that introduces or alters chrome.
- **Drag a11y.** Item 10 is the rich-input shortcut; item 9 is the universal path. Every drag-to-add must be achievable via click-to-add. Same rule as Arc 3.

## Implementation Notes

- **Re-render frequency.** The people-pane data changes only on membership writes — subscribe via PubSub on the existing membership topic rather than polling.
- **Selection state in the rail.** Current Initiative comes from the URL / LiveView assigns; no separate state.
- **Header alignment.** The header's container width must mirror whatever max-width policy the body adopts (item 1) so the chrome doesn't visually float relative to content.
- **Index-as-rail reuse.** Item 5 should reuse the existing index card components in a compressed variant (probably a slot prop or a `compact` flag) rather than forking a parallel widget. This keeps the "rail IS the index" promise true at the code level too, and means future index changes propagate to the rail for free.
- **Flanking rails are chrome.** Both the left rail and the optional right pane (`Layouts.app`'s `rail_right` slot — e.g. the index's Assigned-to-Me pane, item 6) live in the layout as siblings flanking the page content, so they top-align with each other and never sit below page-body elements like the New Initiative button. Exception per item 4: the Initiative-show page keeps its **existing** in-content Details pane as its right rail rather than routing through the slot.
