# m02.08 — Personal Workspace

_Parent: [`m02-ux-buildout.md`](m02-ux-buildout.md)_

The user's personal, cross-Initiative surfaces, plus several cross-cutting
features that close out M02:

- **Assigned to Me** (items 1–5): a new nav item beside Initiatives opening a
  flat, cross-Initiative list of every task on the user's plate — the
  assignment-centric counterpart to the ownership-centric Initiatives list.
- **Notifications** (item 6): an unread feed of things that happened *to* the
  user (added / removed, assigned / unassigned), with a flyout in the user
  menu and a red dot on the nav avatar.
- **Live collaboration** (items 7–8): an ephemeral per-Initiative viewer chat,
  and a comment lifecycle — authors can edit (with history) or delete (leaving
  a tombstone).
- **Archive** (item 9): a per-user shelf — set any Initiative aside into an
  Archived list (offered on completion), with per-member **Hidden** as a lighter
  sub-variant restored through that same list.
- **Account security** (items 10–11): one-time recovery codes (the
  public-opening recovery path while the mailer stays deferred) and TOTP
  two-factor.
- **Membership** (item 12): an Admin role — delegated roster management between
  owner and editor.

**Depends on co-assignees ([`m02.05`](m02.05-wide-width-layout.md) item 12)**
for the direct-vs-co distinction — so this arc runs after Arc 5.

**Draft — pending operator approval.**

## Items

1. [ ] **Page + route + nav item.** Authenticated `live "/assigned"` (or
   similar), reachable from a new **Assigned to Me** nav entry beside
   Initiatives (desktop nav + mobile hamburger). View title "Assigned to Me",
   matching the "My Initiatives" h1 style. This view also fills the **ultrawide
   index right-pane placeholder** reserved by [`m02.05`](m02.05-wide-width-layout.md)
   item 6 — at `3xl:` the index's third pane (where Details sits on an open
   Initiative) is A2M's home; build the list as a component usable in both spots.
2. [ ] **Cross-Initiative query.** Every task where the user is **primary or
   co-assignee**, scoped to Initiatives they're **currently a member of** — so
   tasks in Initiatives they've left / been removed from / that were trashed
   fall out automatically (no special-casing; it's a current-member join).
   Completed tasks **hidden by default**; an on-page toggle reveals them.
3. [ ] **Row display.** Each row clearly differentiates **direct (primary)**
   from **co-assignee**, and shows the task's leaf/child count (reuse the
   mode-aware badge from `m02.04`-era work). The list is **flat — not
   expandable here**; the count conveys subtree size without a second tree UI.
4. [ ] **Group by Initiative.** An on-page, user-level **persistent** "Group
   by Initiative" checkbox, **unchecked by default** (flat list). Same
   control-on-page / storage-on-prefs pattern as the index sort
   ([`m02.04`](m02.04-account-details.md) item 2.6) — the toggle lives on this
   page, its state on the `user_preferences` record, so it follows the account.
5. [ ] **Deep-link to task (the one new mechanism).** A row can open its
   Initiative landing with that task **selected, its collapsed ancestors
   expanded, and scrolled into view**. The tree has no deep-link-to-a-task
   today (selection is client/DOM-owned), so this adds: a task param the show
   LiveView reads on mount → select it, expand the ancestor chain so it's
   visible, and scroll it into view. Honors the no-round-trip-before-input and
   no-layout-jank guardrails.
6. [ ] **Notifications.** A per-user, cross-Initiative **unread feed** — distinct from the per-task Activity log (that's the full per-task record; this is the unread subset of things that happened *to me*).
	1. [ ] New `notifications` table: `user_id`, `kind`, `data` (JSON), `read_at`, `inserted_at`.
	2. [ ] **Auto-generated** on: added / removed as an Initiative member; a task's **primary** assignee set or cleared on you; **co-assignee** added / removed (being co-assigned counts as being assigned). **Never notify the actor about their own action** — actor == recipient → skip. Hooks live where we already broadcast `members_changed` / record activity events (local additions, no new plumbing).
	3. [ ] **Flyout** in the User menu (desktop dropdown) and the hamburger (mobile): recent notifications, newest first, each linking to its subject (the Initiative or — via this arc's deep-link (item 5) — the task); opening the flyout marks them read; a "mark all read" affordance.
	4. [ ] **Unread indicator:** a **red dot** on the nav avatar while any unread exist; clears when read.
	5. [ ] **Real-time:** new notifications push live on a per-user PubSub topic, so the dot appears without a reload.
	6. [ ] Tests: generation per event type, self-exclusion, unread count, mark-read, live push.
7. [ ] **Live chat for concurrent viewers.** A lightweight chat in the lower-left
   for everyone currently viewing an Initiative, so live back-and-forth doesn't
   co-opt the per-task comment threads. Presence-scoped to current viewers and
   fully ephemeral: no history is saved, and the chat clears when the last
   viewer leaves the Initiative. (Comments themselves went live in
   [`m02.06`](m02.06-undo-redo.md) item 15.3; the chat's home in the layout is
   set by Arc 7.)
8. [ ] **Comment lifecycle — edit & delete.** Let a comment's author edit their
   own comment (a minimal popup surfaces prior versions) or delete it, leaving a
   "comment deleted" tombstone so the thread's shape and references survive.
   Needs version storage (a `comment_versions` table or a history column) plus a
   deleted marker (who / when); edits and deletes broadcast live like any
   comment change. Non-authors can neither edit nor delete.
9. [ ] **Archive (per-user).** A per-user flag on any Initiative that moves it
   out of your active index / ultrawide rail into an **Archived list** — your
   personal "set aside" shelf, restorable from that list. Per-user, so it never
   touches anyone else's view, and allowed on any Initiative regardless of
   completion or ownership. `archived_at` on your `initiative_members` row + the
   Archived list/filter. Distinct from Trash (owner-level, global, going away).
	1. [ ] **Archive offer on completion.** Hitting 100% on an Initiative
	   *prompts* to archive it — it never auto-archives.
	2. [ ] **Confirm only on unfinished work.** Archiving confirms first only if
	   *you* still have incomplete assignments or co-assignments in that
	   Initiative; otherwise it archives immediately.
	3. [ ] **Hidden (per-member) — subfeature.** A lighter "just get it off my
	   dashboard" hide, also per-user (`hidden_at` on `initiative_members`);
	   hidden Initiatives drop from the active index and are unhidden through the
	   same Archived list UI.
10. [ ] **Recovery codes (account security).** One-time recovery codes
   generated on the account page — shown once, hashed at rest — plus a "use a
   recovery code" login path. With the mailer deferred to M4 (email-based reset
   rides with it), these are the recovery mechanism that lets the app open to the
   public; admin resets cover the private phase until then.
	1. [ ] Generate / regenerate a set on the account page; display once; store
	   only hashes (a `recovery_codes` table — hashed code + `used_at`, per user).
	2. [ ] A "use a recovery code" path on the login screen that consumes one
	   code (marks it used) and signs you in, where you can reset your password.
11. [ ] **TOTP two-factor (account security).** Authenticator-app 2FA, with
   recovery codes (item 10) as the lockout exit.
	1. [ ] Enroll (QR + secret), verify, and disable on the account page; secret
	   encrypted at rest (`totp_secret` / `totp_enabled` on the user).
	2. [ ] A second-factor prompt at login for enrolled users; a recovery code
	   bypasses a lost authenticator.
12. [ ] **Admin role (membership).** A delegated tier between owner and editor:
   manage the roster — add / remove members, change roles below admin — without
   ownership (no Initiative delete, no ownership transfer). Lets owners hand off
   member management. When it lands, ownership transfer will demote the old owner
   to **admin** instead of today's **editor**. Extends the existing role field
   (an `admin` value); no new table.
13. [ ] **Tests.** Query correctness (primary + co; current-member scoping;
   completed hidden/shown; left-Initiative tasks excluded); group-by toggle
   persistence; deep-link selects + reveals the right task; a comment edit
   records a version and restores prior text, and a delete leaves a tombstone;
   archiving / hiding an Initiative drops it from that member's index only and
   both restore through the Archived list, and archiving confirms first when you
   have your own incomplete assignments/cos there; a recovery code logs in once
   then can't be reused; an enrolled user needs the TOTP factor at login (a
   recovery code bypasses it); an admin can manage the roster but can't delete
   the Initiative or transfer ownership (live chat is presence/PubSub — its
   checks are [Human] below).
14. [ ] **Usability testing** — the [Human] eye-pass once the above are built;
   mix tests ride each implementation item.
	1. [ ] [Human] Assigned tasks across several Initiatives all appear; direct vs co-assignee reads at a glance; light + dark.
	2. [ ] [Human] Completed hidden by default; the reveal toggle shows/hides them.
	3. [ ] [Human] "Group by Initiative" groups, persists across a reload, and follows the account on another device/browser.
	4. [ ] [Human] Leaf/child counts read correctly; rows don't expand here.
	5. [ ] [Human] Clicking a row opens its Initiative with that task selected, ancestors expanded, and scrolled into view — no jank.
	6. [ ] [Human] A task in an Initiative you've left no longer appears.
	7. [ ] [Human] Notifications: being added/assigned (by someone else) raises the red dot + a flyout entry; your own self-actions never notify; opening the flyout clears the dot; a notification arriving live raises the dot without a reload.
	8. [ ] [Human] Live chat: two viewers of one Initiative see each other's messages in real time; a fresh viewer sees no history; the chat clears once the last viewer leaves.
	9. [ ] [Human] Editing or deleting your own comment updates live for other viewers — an edit shows the history popup, a delete leaves a tombstone; you can't edit or delete someone else's.
	10. [ ] [Human] Archive: archiving sets an Initiative aside into the Archived list (and completing one offers to archive); restore brings it back; the confirm fires only when you have your own unfinished assignments/cos there.
	11. [ ] [Human] Hidden: hiding drops an Initiative from your dashboard only (others still see it); unhide runs through the Archived list.
	12. [ ] [Human] Recovery codes generate + show once on the account page; logging in with one consumes it (no reuse); admin reset still works.
	13. [ ] [Human] TOTP: enroll via authenticator, log out, and the second-factor prompt appears at login; a recovery code gets you in without the authenticator; disabling 2FA removes the prompt.
	14. [ ] [Human] Admin: an admin can add/remove members and change roles below admin, but has no delete-Initiative or transfer-ownership controls; transferring ownership demotes the old owner to admin.

## Cross-cutting requirements

- **Schema.** A `notifications` table (item 6), comment-version storage (item
  8), `archived_at` + `hidden_at` on `initiative_members` (item 9), a
  `recovery_codes` table (item 10), TOTP fields on the user (item 11), an `admin`
  value on the role field (item 12), and one boolean column on `user_preferences`
  for the group-by toggle (item 4). The Assigned-to-Me list itself is a read over
  `tasks` + the co-assignee table + the membership join.
- **Light + dark visual review** of the new page and nav entry.
- **A11y.** The list is keyboard-navigable; the deep-link open is reachable by
  keyboard with visible focus, and the landed-on task takes focus / is
  announced.
