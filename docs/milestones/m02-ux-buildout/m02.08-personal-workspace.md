# m02.08 — Personal Workspace

_Parent: [`m02-ux-buildout.md`](m02-ux-buildout.md)_

Cross-Initiative surfaces — Assigned-to-Me worklist + Notifications feed — plus live collaboration and per-user archiving.

**Depends on co-assignees ([`m02.05`](m02.05-wide-width-layout.md) item 12)** for the direct-vs-co distinction; runs after Arc 5.

1. [ ] **Assigned to Me**

	1. [x] **Page + route.** Build authenticated `live "/assigned"`, titled "Assigned to Me" in the "My Initiatives" h1 style, with the list as a reusable component (shared with item 1.5 below).
	2. [x] **Nav entry.** Add an **Assigned to Me** nav entry beside Initiatives, in both the desktop nav and the mobile hamburger.
	3. [x] **Fill the ultrawide placeholder.** Render the list in the **ultrawide index right-pane placeholder** ([`m02.05`](m02.05-wide-width-layout.md) item 6) — at `3xl:`, the index's third pane (where Details sits on an open Initiative).
	4. [x] **Cross-Initiative query.** Every task where the user is **primary or co-assignee**, scoped to Initiatives they're a **current member** of — tasks they've left / been removed from / trashed drop out via the current-member join. Completed **hidden by default**; on-page toggle reveals. Tasks from your **archived / hidden** Initiatives are **also hidden by default** — a separate **unchecked** checkbox on the A2M list reveals them when checked.
	5. [x] **Row display.** Distinguish **direct (primary)** from **co-assignee**; show the leaf/child count (mode-aware badge from `m02.04`-era work). Rows are **flat — not expandable here**; the count conveys subtree size without a tree.
	6. [x] **Group by Initiative.** Add a persistent **Group by Initiative** checkbox (off by default, flat); control on-page, state on `user_preferences` like the index sort ([`m02.04`](m02.04-account-details.md) item 2.6) → follows the account.
	7. [x] **Deep-link to task** (the one new mechanism). A row opens its Initiative landing with the task **selected, collapsed ancestors expanded, scrolled into view**. None exists today (selection is client/DOM-owned), so add a task param the show LiveView reads on mount: select, expand ancestors, scroll. Honors no-round-trip-before-input, no-layout-jank.

2. [ ] **Notifications**

	Per-user, cross-Initiative **unread feed** — distinct from the per-task Activity log (the full per-task record; this is the unread subset of what happened *to me*).

	1. [x] New `notifications` table: `user_id`, `kind`, `data` (JSON), `read_at`, `inserted_at`.
	2. [x] **Auto-generated** on: added / removed as Initiative member; **primary** assignee set/cleared on you; **co-assignee** added / removed (co counts as assigned). **Never notify the actor about their own action** — actor == recipient → skip. Hooks where we already broadcast `members_changed` / record activity (local additions, no new plumbing).
	3. [x] **Flyout** in User menu (desktop dropdown) and hamburger (mobile): recent notifications newest-first, each linking its subject (the Initiative or — via this arc's deep-link, worklist 1 item 7 — the task); opening marks read; "mark all read" affordance.
	4. [x] **Unread indicator:** **red dot** on the nav avatar while any unread exist; clears when read.
	5. [x] **Real-time:** new notifications push live on a per-user PubSub topic; dot appears without reload.
	6. [x] Tests: generation per event type, self-exclusion, unread count, mark-read, live push.

3. [ ] **Live collaboration**

	1. [x] **Live chat for concurrent viewers.** Lightweight lower-left chat for everyone currently viewing an Initiative — separate from per-task comment threads. Presence-scoped, fully ephemeral: no history, clears when the last viewer leaves. (Comments went live in [`m02.06`](m02.06-undo-redo.md) item 15.3.) This worklist owns the chat's placement — a fixed lower-left overlay; Arc 7 reserved no chat home.
	2. [x] **Comment lifecycle.**

		1. [x] Add version storage (a `comment_versions` table) and a deleted marker (who / when).
		2. [x] Let an author edit their own comment, with a minimal popup surfacing prior versions; broadcast the edit live.
		3. [x] Let an author delete their own comment, leaving a "comment deleted" tombstone so thread shape + references survive; broadcast the delete live.
		4. [x] Block non-authors from editing or deleting.

4. [x] **Archive (per-user).** A per-user flag moving any Initiative out of your active Initiatives into a restorable **Archived list**. Never touches anyone else's view; allowed regardless of completion or ownership. `archived_at` on your `initiative_members` row + the Archived list/filter. Distinct from Trash (owner-level, global, going away).
	1. [x] **Archive offer on completion.** Hitting 100% *prompts* to archive — never auto-archives.
	2. [x] **Confirm only on unfinished work.** Archiving confirms first only if there's unfinished work you'd care about — for a member, *your own* incomplete assignments or co-assignments; for the **Initiative owner**, *any* incomplete task in the Initiative — otherwise archives immediately.
	3. [x] **Hidden (per-member) — subfeature.** A lighter "off my dashboard" hide, also per-user (`hidden_at` on `initiative_members`); hidden Initiatives drop from your active Initiatives; unhide through the same Archived list UI, where hidden items are **off by default** behind a **Show hidden** checkbox that **does not persist** — it resets every visit, so you re-check it each time (hidden stays out of sight by default, unlike the persistent Group-by toggle).

5. [ ] **Testing**

	1. [ ] **Tests.** Query correctness (primary + co; current-member scoping; completed hidden/shown; archived/hidden Initiatives' tasks hidden by default and revealed by the checkbox; left-Initiative tasks excluded); group-by persistence; deep-link selects + reveals the right task; comment edit records a version and restores prior text, delete leaves a tombstone; archive / hide drops an Initiative from that member's index only, both restore via Archived list, archive confirms first with your own incomplete assignments/cos (or, as owner, any incomplete task) (live chat is presence/PubSub — [Human] below).
	2. [ ] **Usability testing** — [Human] eye-pass once built; mix tests ride each impl item.
		1. [ ] [Human] Tasks from several Initiatives appear; direct vs co reads at a glance; light+dark.
		2. [ ] [Human] Completed hidden by default; reveal toggle shows/hides.
		3. [ ] [Human] "Group by Initiative" groups, persists across reload, follows the account on another device/browser.
		4. [ ] [Human] Leaf/child counts read correctly; rows don't expand here.
		5. [ ] [Human] Row click opens its Initiative, task selected, ancestors expanded, scrolled into view — no jank.
		6. [ ] [Human] Task in an Initiative you've left drops out.
		7. [ ] [Human] Notifications: added/assigned by someone else raises red dot + flyout entry; self-actions never notify; opening flyout clears the dot; a live one raises it without reload.
		8. [ ] [Human] Live chat: two viewers of one Initiative see each other's messages live; fresh viewer sees no history; clears once last viewer leaves.
		9. [ ] [Human] Editing/deleting your own comment updates live for others — edit shows history popup, delete leaves tombstone; can't touch someone else's.
		10. [ ] [Human] Archive: archiving sets an Initiative into the Archived list (completing offers to archive); restore brings it back; confirm fires only with your own unfinished assignments/cos (owner: any incomplete task in the Initiative).
		11. [ ] [Human] Hidden: hiding drops an Initiative from your dashboard only (others still see it); in the Archived list hidden items are off by default and the Show-hidden checkbox resets each visit; unhide from there.

## Cross-cutting requirements

- **Schema.** `notifications` table (worklist 2), comment-version storage (worklist 3 item 2), `archived_at` + `hidden_at` on `initiative_members` (worklist 4), one boolean on `user_preferences` for the group-by toggle (worklist 1 item 6). Assigned-to-Me is a read over `tasks` + the co-assignee table + the membership join.
- **Light + dark visual review** of the new page and nav entry.
- **A11y.** List keyboard-navigable; deep-link open keyboard-reachable with visible focus; landed-on task takes focus / is announced.
