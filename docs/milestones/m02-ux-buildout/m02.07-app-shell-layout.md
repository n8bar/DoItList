# m02.07 — Initiative Layout & Task Tree, Revisited

_Parent: [`m02-ux-buildout.md`](m02-ux-buildout.md)_

A return to the Initiative view that Arc 3 ([`m02.03`](m02.03-task-tree.md)) first built — now to give it a real **app-shell layout** plus a pass of **task-tree refinements**. The view becomes a viewport-height workspace (the task tree and the right-side panes each scroll on their own, the header stays put above the tree, and the Members panel stays on screen so it's reachable to drag onto a task), and the task rows + add-task flow get the presentation and keyboard polish they've been missing.

The app-shell half began as an Arc 6 observation: once Members afford dragging onto task rows ([`m02.05`](m02.05-wide-width-layout.md) item 12.8, [`m02.06`](m02.06-undo-redo.md) item 15.2), they have to stay visible while the Details pane is open — which the single-scroll right rail doesn't do. Bigger than an O&C tweak, so it's its own arc.

## Goal

On desktop/tablet (`lg:`+), turn the Initiative page from whole-page scroll into a viewport-height shell with independent scroll regions:

- **Center column** — the initiative header (title + roll-up bar + New List) sits *outside* the tree's scroll box; the task tree scrolls beneath it, with a theme-matched fade at the top/bottom edges signifying more content.
- **Right pane** — the Members panel stays pinned and on screen; the Details (or initiative editor) content scrolls independently beneath it.

Below `lg:`, keep today's behavior unchanged: page scroll, the Details flyout overlay + backdrop, and the header's Show/Hide Members toggle (drag-to-assign is desktop-only).

Beyond the shell, the arc polishes the task rows (color-coded priority pills, positional task indexes) and the add-task flow (Esc to close, arrow keys to reposition) — items 6–9.

## Items

1. [ ] **App-shell container (`lg:`+).** Convert the Initiative-show main area from page scroll to a viewport-height shell — two columns (center `1fr`, right pane keeping the existing `360 / 400 / 440px` responsive widths), each owning its own vertical scroll. The page itself stops scrolling at `lg:`; below it, nothing changes.

2. [ ] **Header outside the scroll box.** Move the initiative header (title, roll-up progress bar, New List) out of the full-width row above the grid into the top of the center column, as a fixed sibling *above* the tree's scroll container — not `position: sticky` inside it. The tree scrolls under it; the header never moves.

3. [ ] **Tree scroll-fade signifier.** A theme-matched gradient at the scroll-box edges signifying more content — and softening the header/content boundary so the first row never reads as "a new list starting."
	1. [ ] **Top fade** appears only when the tree's `scrollTop > 0`; **bottom fade** appears while more content sits below, and hides at the end.
	2. [ ] **Theme-matched:** the gradient stop is the center column's exact background token, with a dark-mode variant — a fade, not a gray smudge.
	3. [ ] **Click-through:** `pointer-events: none` on the overlay — it must never intercept a click or drag on the rows beneath (hard requirement; the top rows are drag-reorder + select targets).
	4. [ ] **Toggle:** a small scroll hook flips `data-scrolled` / `data-at-end` on the container; the fades show/hide off those in CSS (preferred over `animation-timeline: scroll()` for cross-browser safety).

4. [ ] **Right pane — Members pinned, Details scrolls.** Restructure the right `aside` so the Members panel stays on screen (pinned at the top of the pane, capped height with its own overflow when the list is long) while the Details / initiative-editor content scrolls independently beneath it — so Members is reachable to drag from no matter how far Details has scrolled. The pane scrolls independently of the center tree. With no task selected, the pane shows Members (Details empty) and keeps its column position.

5. [ ] **Responsive fallback (`< lg:`).** Unchanged: page scroll, Details as a fixed flyout overlay with backdrop, Members via the header's Show/Hide toggle. The shell, pinned Members, and scroll-fades are `lg:`+ only.

6. [ ] **Color-coded priority pills.** Map each priority (`high` / `low` / …) to its own chip color instead of monochrome zinc, so priority reads at a glance; `normal` stays the empty dashed placeholder. One-spot change in `task_node/1`'s priority chip. Pick colors that survive dark mode *and* stay colorblind-distinguishable — don't rely on hue alone (keep the label).

7. [ ] **Task indexes — positional numbering.** A per-task index label between the botanical icon and the pills, computed from sibling position at every level. Display-only, derived from sort order — recomputed on reorder / move.
	1. [ ] Fixed styles only: **Outline** (`I.A.1.a.i`), **Numerical** (`1.1.2`), **Roman**, **Alphabetical**; **None** is the default.
	2. [ ] Style is a **per-Initiative** preference (an index scheme is a property of the tree, not the account) — one column on the Initiative + a selector in the Initiative editor.
	3. [ ] Deferred to follow-ups: Custom Outline, a depth-flat variant, and a "continue across collapse" toggle.

8. [ ] **Esc closes the New Task UI.** Esc dismisses the add-task form and discards the typed title, no confirm — an exception to keyboard-suppression so it fires even while the title box is focused (Esc edits no text). Reuses the Esc-closes pattern from the delete / transfer confirms; order it so it doesn't collide with those modals' global Esc handlers.

9. [ ] **Arrow keys reposition the add-task form.** While the add-task title box is focused, Up / Down walk the form's insertion point through the tree (Left / Right stay with the text cursor). The walk stops at *both* the "child of this task" and "sibling" slots — indentation shows which — so you nest or stay level by stepping vertically; the form visibly relocates and the typed title rides along. `preventDefault` so the cursor doesn't jump. Signify it in the placeholder (e.g. `New task...  (↑↓ to move)`).

10. [ ] **Testing** — a [Human] visual/interaction pass once the above are built; `mix test` rides each item (mostly markup/CSS/JS, so the suite mainly guards against LiveView render regressions).
	1. [ ] [Human] Header stays put while a long tree scrolls; the tree and the right pane scroll independently of each other.
	2. [ ] [Human] Drag a member onto a task after scrolling Details far down — Members is still visible and draggable, and the drop assigns.
	3. [ ] [Human] Top fade shows only when scrolled down, bottom fade only when more is below; both are theme-matched in light + dark and let clicks/drags pass through to the rows beneath.
	4. [ ] [Human] `< lg:` unchanged — page scrolls, Details flies out over a backdrop, the Members toggle works; no regression.
	5. [ ] [Human] Priority pills read at a glance by color in light + dark, distinguishable without relying on hue alone (item 6).
	6. [ ] [Human] Task indexes render in each style, recompute correctly on reorder / move, and the per-Initiative style preference persists (item 7).
	7. [ ] [Human] Esc closes the adder from inside the title box; the add-task arrows reposition the form (child + sibling stops), the title rides along, and the placeholder signifies it (items 8–9).
	8. [ ] [Agent] Full `mix test` green.

## Cross-cutting requirements

- **Minimal schema.** Only the per-Initiative index-style column (item 7); the rest is layout / CSS / client-hook + presentational work.
- **Light + dark visual review** of the shell, header, fades, and right pane.
- **A11y.** The independent scroll regions stay keyboard-reachable; focus order is header → tree → right pane; the scroll-fade is decorative (`aria-hidden`, `pointer-events: none`).
- **No regression** to selection (client/DOM-owned), the Details flyout, or the mobile Members toggle.

## Implementation Notes

- The right rail already co-locates Members + initiative editor + Details (`initiative_show_live.ex`, `aside#details-rail`); this splits its single scroll into a pinned Members region + an independently scrolling content region, and makes it a persistent column rather than only-when-`data-open`.
- Reuse the existing `data-open` flyout mechanism for `< lg:`; the shell changes are additive at `lg:`+.
- Pairs with [`m02.06`](m02.06-undo-redo.md) item 15.8 (Details pane renders from the loaded tree node without a round trip) — that fills the pane's content; this owns where it lives and how it scrolls.
