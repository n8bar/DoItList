# m02.09 — Client-Interaction Foundation

_Parent: [`m02-ux-buildout.md`](m02-ux-buildout.md)_

> **Make life not suck on LongPoll.**

Arc 9 brings the whole app into compliance with the **client-first interaction** guardrails — [`UX_GUARDRAILS`](../../UX_GUARDRAILS.md) §6 (action feedback, acknowledge-every-action, interactive-from-first-paint, any-transport) and §7.3–7.4 (client owns ephemeral UI state + optimistic; presence continuity) — and makes them hold on LongPoll, not just WebSocket. It's the deliberate build of the **client half** we'd been retrofitting one §6 violation at a time. **Not** a rewrite and **not** abandoning LiveView — it hardens the client layer on top of it. Arc 8 made point fixes (the client-side confirm sweep, the client-side editor open) and the transport groundwork is in place (`longPollFallbackMs` raised; a partial connect/reconnect re-assert added); this arc makes the re-assert robust and codifies the pattern.

**The problem, precisely (measured, not theorized).** On a WebSocket session the page **paints in ~0.1s** (tree, members, panes — looks fully ready) but the LiveView isn't **live until ~2.9s**: a **~2.8-second dead window** where the page is a static photograph and nothing the user does takes. The connect *taking over* at the end of that window is what's felt as the "major reload" — and that takeover is what stomps client state opened during the window. There is **no second render that re-stomps the whole page**; the villain is the **painted→interactive gap**, not a re-render. **Transport was never the villain** — WebSocket only *shrank* the gap (warm LongPoll is ~25% slower, not the earlier multi-second disaster, which was a cold/slow connect stacked on LongPoll). The fix is to **close the paint→interactive gap and make the in-between fully usable, on any transport** — don't re-chase the transport.

1. [ ] **Optimistic & client-owned by default**

	A systematic pass over every direct-manipulation action, not a re-do of Arc 8's point fixes. The goal: each such action gives immediate client feedback and is client-owned wherever the state is ephemeral, and the remaining round-trip-before-input gaps close.

	1. [x] **Inventory direct-manipulation actions** into a committed table — the **spine of this arc**: item 3 draws its fix subitems from it, and worklist 3 drives the dead-window handling from the same columns.
		1. [x] **Enumerate the actions** — every direct-manipulation interaction (selection, expand/collapse, focus, editor/details open-close, optimistic flips, add-form open, reorder/move), one row each.
		2. [x] **Record the three columns per action:** feedback immediate? · state location (client vs. server) · pre-connect behavior (optimistic / queue / signify).
		3. [x] **Commit the table** as the shared artifact the rest of the arc reads.
	2. [x] **Classify against §6.** For each inventoried action, mark whether it satisfies §6.2 (optimistic for fast ops), §6.5 (view-state never waits on network), and §6.6 (confirm preserves optimism); flag any that still gate input or feedback on a round trip.
	3. [ ] **Close the remaining gaps — populated from the inventory.** One fix per flagged action, grouped below by shared mechanism, drawn from the inventory's 28 discrete-action violations; the cross-cutting §6.8 dead-window row is worklist 3, not duplicated here. Each makes feedback instant and the action client-owned where the state is ephemeral. Acknowledge — do not redo — the Arc 8 client-side confirm sweep and the client-side initiative-editor open as already-passing. Pre-connect *survival* of these acknowledgements is worklist 3 (§6.8): 1.3 builds the acknowledgement, worklist 3 makes it hold before connect. _Subitems below: draft — pending operator approval._
		1. [ ] **Archive-confirm slow-open (reframed seed — not a round-trip).** The owner-path confirm already opens client-side (`app.js:1066-1069`), so the lived ~4-beat latency is not a §6.5 network wait. Diagnose the client cause — the document-level click handler's `#task-tree [data-task-row]:not([data-done])` scan on a large tree, and/or the full-screen overlay's paint — and make the open instant. Diagnose against a real interaction, not a synthetic `el.click()`.
		2. [ ] **Client-side view-state reveals (§6.5).** Flip at click on the client; the server assign only reconciles, never gates the reveal. Covers the Assigned-view show-completed / show-archived+hidden / group-by toggles (group headers computed client-side for the reflow), the Archived-panel show-hidden toggle, and the archive-on-completion banner dismiss.
		3. [ ] **Client-owned open/close for editors & popovers (§6.5).** Open/close at click; the server assign reconciles. Covers the comment inline editor open + cancel; the comment edit-history popup (open immediately with a loading state while versions fetch — don't gate the open on the fetch — close client-side); and the hand-off-cancel modal close.
		4. [ ] **Convert the remaining server-raised confirms to client-opened (§6.5/§6.7).** The eye-pass + diagnosis found this is NOT a lone modal — three confirms still round-trip before they appear: (a) **cascade-sort** (>10 branches) — server `@pending_action` (initiative_show_live.ex:1030-1049); (b) **member-handoff** (remove member w/ assignments) — server `@pending_handoff` (:1456-1461), opens only after the round-trip; (c) **move/create completion-confirm backstop** — the server `@pending_action` can fire as the FIRST modal instead of the client `#move-flip-confirm` / `#cascade-confirm`. Convert each to the Arc-8 static client-opened pattern (predict counts from the DOM, open synchronously, server authoritative on commit). For handoff, whose body needs server data (assignment count), at minimum show an in-flight signifier at click and fetch only what isn't client-known.
		5. [ ] **Drop-time optimism for drags (§6.2).** Apply the result at drop, snapshot/revert on failure. Covers the InitiativeDrag card reorder and drag-collaborator-onto-initiative (optimistic viewer chip).
		6. [ ] **Optimistic echo for server-gated writes that show their own result (§6.7).** Apply at submit, reconcile/dedupe on the reply. 2. [ ] **Ephemeral UI state survives the LiveView lifecycle**

	Client-owned UI state (selection, open panes, expand/collapse, optimistic flips, presence, focus/caret/scroll position) must survive every server re-render, reconnect, and initial connect. Reapply it *inside* the morphdom patch via LiveView's `dom` callbacks — preventively, so it's never clobbered and there's no race — driven by a declarative attribute convention and a small client-side store of the truth. One transport-agnostic path, replacing the per-feature observers and `KeepOpen`.

	1. [x] **Define the client-owned convention + unified state store.** A declarative, **server-rendered** marker (`data-keep="<kind>"`) for what's client-owned — selection, open panes, expand/collapse, optimistic flips, presence, and focus/caret/scroll position — plus a single JS store of the truth (`window.DoitState`) that survives re-renders and seeds re-added elements, replacing the eight fragmented `window.Doit*` objects.
	2. [ ] **Apply it through the patch.** `onBeforeElUpdated` carries the store's truth onto elements updated in place (reconcile the incoming node *before* morphdom commits, so the wrong value never paints); `onNodeAdded` re-applies from the store onto elements morphdom removes and re-adds (reorder, reset re-stream, the initial-join replace). Together they cover both cases; handle attribute removal too, and preserve focus/caret/scroll — the highest-stakes state — through the patch.
	3. [ ] **Migrate existing state onto the convention** — selection, editor/details panes, expand/collapse, optimistic flips, presence badges, focus/caret/scroll. (The optimistic-flips migration also closes the eye-pass gap where a leaf checkoff doesn't optimistically sync the selected task's pane manual-slider and the pending-toggle handle is cleared on reply before the patch reconciles.)
	4. [ ] **Retire the racing observers, keep the store.** Remove the `DoitSelection` / `DoitInitiativeEditor` MutationObserver re-assert code and `KeepOpen`; the underlying state store stays — it's now the single source the `dom` callbacks read.

3. [ ] **Input always registers — no "looks ready but isn't" dead window**

	The user must see their input did something up front. Eliminate the gap where the page is painted but not yet interactive and actions are silently lost.

	1. [ ] **Connecting / in-flight signifier.** The page never looks fully interactive before it is.
		1. [ ] **Detect + render.** Track connect / in-flight state via the socket lifecycle (`onOpen` / `onClose`, `phx:page-loading-start` / `-stop`) and show a signifier — extend the existing topbar (already wired to `phx:page-loading-*`); clear when live/settled; a11y: not color-only, announced.
	2. [ ] **No dropped pre-connect actions.**
		1. [ ] **Design the capture-and-replay path** — how pre-connect actions are captured, held in order, deduped, and flushed on connect (deliberate design, not guessed).
		2. [ ] **Apply or enqueue per the 1.1 inventory** — client-ownable actions applied optimistically at click; server-gated ones enqueued.
		3. [ ] **Flush on connect** — replay the queue in order and reconcile, surfacing anything the server rejects.
	3. [ ] **Up-front acknowledgement.**
		1. [ ] **Wire instant acknowledgement for each flagged action** — for every action the 1.1 inventory marks not-immediate, acknowledge at the moment of the action (optimistic apply or in-flight signifier), independent of connect state.

4. [ ] **Seamless navigation — master-detail, no remount**

	Opening an initiative from the workspace should patch the detail region, not remount and swap the whole page. The workspace list and initiative view become one LiveView (or one `live_session`) navigated by `push_patch` — the established collaborative-app pattern that keeps the live session, and your presence, intact instead of flickering you out and back in for everyone else viewing.

	1. [ ] **Map the current two LiveViews.** How `InitiativeIndexLive` and `InitiativeShowLive` are structured today — mounts, assigns, subscriptions, presence, routes, shared components — so the merge is grounded in the real code, not guessed.
	2. [ ] **Design the merge.** From that map, decide the structure: one LiveView vs. one `live_session`; the list↔detail state split; where the detail renders (likely a LiveComponent); the `push_patch` / `handle_params` routing — and what stays mounted vs. re-renders per detail.
	3. [ ] **Implement it.** Route workspace↔initiative navigation through `push_patch` / `handle_params` into the kept-mounted shell; the socket, PubSub subscriptions, and presence stay intact across navigation — no re-track, no flicker for other viewers.
	4. [ ] **Instant, non-stomping open.** Opening patches only the detail region — instant view-state plus an optimistic / in-flight fill — and rides worklist 2's preserve path so it never stomps client state.

5. [ ] **Sorting a branch must not steal selection**

	Acting on a branch to sort its children hijacks the selection — the branch becomes the selected task, so you can't reorder a branch while a descendant stays selected (only moving the selected task itself works).

	1. [ ] **Investigate** how a branch's sort control triggers selection — whether the control's click bubbles into the row-select handler, or selection is a side effect of the sort path.
	2. [ ] **Fix so the sort affordance acts without changing the selection** — reorder a branch while keeping your place.

## Cross-cutting requirements

- **Spec compliance (WHAT vs HOW).** The end-state is the spec — [`UX_GUARDRAILS`](../../UX_GUARDRAILS.md) §6 + §7.3–7.4; this arc is the HOW that brings the app to it. The standing release gate in [`CLAUDE.md`](../../../CLAUDE.md) (Engineering Style) keeps new features inside it. (Subsumes the old LongPoll-parity, no-round-trip-before-input, and not-a-rewrite notes — now in the spec / intro.)
- **One preserve path.** Worklist 2's `onBeforeElUpdated` hook is the single owner of keeping client-owned state through patches/reconnects/initial connect — preventive, not corrective; per-feature guards collapse into it (worklist 2 item 4).
- **A11y.** The connecting / in-flight signifier is perceivable (not color-only) and announced; restored focus after a patch lands where the user left it.
