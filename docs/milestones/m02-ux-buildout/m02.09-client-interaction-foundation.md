# m02.09 — Client-Interaction Foundation

_Parent: [`m02-ux-buildout.md`](m02-ux-buildout.md)_

**Draft — pending operator approval.**

> **Make life not suck on LongPoll.**

We built the interaction layer reactively. LiveView's default runs interactions on the **server**, so anything we didn't deliberately make client-first quietly became a round-trip, and we've been fixing [`UX_GUARDRAILS`](../../UX_GUARDRAILS.md) §6 violations one at a time. The framework is not the mistake — LiveView earns its place for the real-time-collaborative core (presence, live multi-user updates), and the right model already lives in our engineering style: **the client owns ephemeral UI state + optimistic writes; the server owns durable data + sync.** The mistake was never building the **client half** as a first-class foundation, so each new feature could quietly re-introduce a round-trip. This arc builds that half deliberately and systematically, so we stop the whack-a-mole.

This is **not** a rewrite to an SPA and **not** abandoning LiveView. It hardens the client interaction layer on top of LiveView. Arc 8 already made point fixes here — the client-side confirm sweep and the client-side initiative-editor open — and the transport groundwork is in place (`longPollFallbackMs` raised so WebSocket wins the connect on clean loads; a partial connect/reconnect re-assert added). This arc owns making that re-assert robust and codifying the pattern so it holds.

1. [ ] **Optimistic & client-owned by default**

	A systematic pass over every direct-manipulation action, not a re-do of Arc 8's point fixes. The goal: each such action gives immediate client feedback and is client-owned wherever the state is ephemeral, and the remaining round-trip-before-input gaps close.

	1. [ ] **Inventory direct-manipulation actions.** Enumerate every direct-manipulation interaction (selection, expand/collapse, focus, editor/details open-close, optimistic progress flips, add-form open, reorder/move) and record, per action, whether feedback is immediate and where its state lives (client vs. server).
	2. [ ] **Classify against §6.** For each inventoried action, mark whether it satisfies §6.2 (optimistic for fast ops), §6.5 (view-state never waits on network), and §6.6 (confirm preserves optimism); flag any that still gate input or feedback on a round trip.
	3. [ ] **Close the remaining gaps.** Fix the flagged actions so each gives instant client feedback and is client-owned where ephemeral. Acknowledge — do not redo — the Arc 8 client-side confirm sweep and the client-side initiative-editor open as already-passing cases.
		1. [ ] **Known violation — archive confirm latency.** The operator counts ~4 beats between clicking Archive and the confirm dialog appearing, on a fully-connected WebSocket session (i.e. not the connect/major-reload, and WebSocket did NOT fix it). Feedback is waiting on the network when it shouldn't be. A synthetic `el.click()` bridge test falsely reported this instant — the lived report is ground truth; diagnose against a real interaction, not a synthetic dispatch.

2. [ ] **Ephemeral UI state survives the LiveView lifecycle**

	**One** general mechanism that re-asserts all client-owned ephemeral state after any server patch, after reconnect, and on the initial connect "major reload" — replacing today's scattered per-feature `MutationObserver` guards.

	1. [ ] **Single re-assert mechanism.** Build one client mechanism that re-applies all ephemeral UI state — task selection, open editor / details panes, expand/collapse, optimistic flips, presence badges — after **any** server patch, **reconnect**, and the **initial connect re-render**. Replace the per-hook guards (`DoitSelection`, `DoitInitiativeEditor`, and siblings re-asserting their own state) with this one path.
	2. [ ] **Fix the known-failing connect case.** On a slow LongPoll connect, the join re-render hides a pane the user opened during the pre-connect window — bridge-verified still failing as of the latest commit. The re-assert must restore that pane on connect.
	3. [ ] **Hold on LongPoll, not just WebSocket.** Verify the re-assert holds under the LongPoll fallback (a production-grade transport), not only WebSocket — the failing case above is a LongPoll case.
	4. [ ] **Retire the scattered guards.** Once the single mechanism is in place, remove the now-redundant per-feature re-assert code so there's one source of truth for re-applying ephemeral state.

3. [ ] **Input always registers — no "looks ready but isn't" dead window**

	The user must see their input did something up front. Eliminate the gap where the page is painted but not yet interactive and actions are silently lost.

	1. [ ] **Connecting / in-flight signifier.** Show a client signifier while the socket is connecting or an action is in flight, so the page never looks fully interactive before it is.
	2. [ ] **No dropped pre-connect actions.** Ensure actions taken in the pre-connect window are handled client-first (queued or applied optimistically), so an action during the dead window is never silently lost.
	3. [ ] **Up-front acknowledgement.** Confirm every input produces immediate visible acknowledgement at the moment of the action, independent of connect state.

4. [ ] **Audit & anti-regression**

	1. [ ] **Whole-app §6 sweep.** Walk the full app against [`UX_GUARDRAILS`](../../UX_GUARDRAILS.md) §6 (6.1 progress < ~100ms, 6.2 optimistic, 6.5 view-state off the network, 6.6 confirm preserves optimism) and record findings.
	2. [ ] **Codify the client-first pattern.** Document the client-first interaction pattern (where ephemeral state lives, how optimistic writes reconcile, how new UI joins the re-assert path) so new features inherit it instead of re-introducing round-trips. Note the guardrail in [`UX_GUARDRAILS`](../../UX_GUARDRAILS.md) §6 if a line is missing.

5. [ ] **Testing**

	1. [ ] **Tests.** `mix test` coverage for any new client/server contract surfaced by the re-assert mechanism and the in-flight signifier (the automated gate per house rules).
	2. [ ] **Usability testing** — [Human] / bridge eye-pass once built; mix tests ride each impl item.
		1. [ ] [Human] Open a pane (details or initiative editor) immediately on page load and confirm it survives the connect on **LongPoll** — not just WebSocket.
		2. [ ] [Human] Reconnect (drop and restore the socket) with selection + an open pane + an expand/collapse state set; all ephemeral UI state is restored after reconnect.
		3. [ ] [Human] Input feedback is immediate on every direct-manipulation action — selection, expand/collapse, editor open, optimistic flip — with no visible wait on the network.
		4. [ ] [Human] During the pre-connect window the page shows a connecting/in-flight signifier; an action taken then is acknowledged and not silently dropped once connected.
		5. [ ] [Human] Light + dark: the connecting / in-flight signifier reads correctly in both modes.

## Cross-cutting requirements

- **One re-assert path.** The mechanism in worklist 2 is the single owner of re-applying ephemeral UI state across patches, reconnects, and the initial connect; per-feature guards collapse into it (worklist 2 item 4).
- **LongPoll parity.** Every behavior here must hold under the LongPoll fallback, not only WebSocket — LongPoll is the production-grade fallback and the known-failing case lives there.
- **No new round-trip-before-input.** Nothing server-gated stands between the user and acting; the audit (worklist 4) is the standing check, and the codified pattern (worklist 4 item 2) keeps new features inside it.
- **A11y.** The connecting / in-flight signifier is perceivable (not color-only) and announced; restored focus after a re-assert lands where the user left it.
- **Not a rewrite.** No move to an SPA and no removal of LiveView; this hardens the client layer on top of it.
