# m02.09 — Client-Interaction Foundation

_Parent: [`m02-ux-buildout.md`](m02-ux-buildout.md)_

**Draft — pending operator approval.**

> **Make life not suck on LongPoll.**

Arc 9 brings the whole app into compliance with the **client-first interaction** guardrails — [`UX_GUARDRAILS`](../../UX_GUARDRAILS.md) §6 (action feedback, acknowledge-every-action, interactive-from-first-paint, any-transport) and §7.3–7.4 (client owns ephemeral UI state + optimistic; presence continuity) — and makes them hold on LongPoll, not just WebSocket. It's the deliberate build of the **client half** we'd been retrofitting one §6 violation at a time. **Not** a rewrite and **not** abandoning LiveView — it hardens the client layer on top of it. Arc 8 made point fixes (the client-side confirm sweep, the client-side editor open) and the transport groundwork is in place (`longPollFallbackMs` raised; a partial connect/reconnect re-assert added); this arc makes the re-assert robust and codifies the pattern.

**The problem, precisely (measured, not theorized).** On a WebSocket session the page **paints in ~0.1s** (tree, members, panes — looks fully ready) but the LiveView isn't **live until ~2.9s**: a **~2.8-second dead window** where the page is a static photograph and nothing the user does takes. The connect *taking over* at the end of that window is what's felt as the "major reload" — and that takeover is what stomps client state opened during the window. There is **no second render that re-stomps the whole page**; the villain is the **painted→interactive gap**, not a re-render. **Transport was never the villain** — WebSocket only *shrank* the gap (warm LongPoll is ~25% slower, not the earlier multi-second disaster, which was a cold/slow connect stacked on LongPoll). The fix is to **close the paint→interactive gap and make the in-between fully usable, on any transport** — don't re-chase the transport.

1. [ ] **Optimistic & client-owned by default**

	A systematic pass over every direct-manipulation action, not a re-do of Arc 8's point fixes. The goal: each such action gives immediate client feedback and is client-owned wherever the state is ephemeral, and the remaining round-trip-before-input gaps close.

	1. [ ] **Inventory direct-manipulation actions.** Enumerate every direct-manipulation interaction (selection, expand/collapse, focus, editor/details open-close, optimistic progress flips, add-form open, reorder/move) and record, per action: whether feedback is immediate, where its state lives (client vs. server), and its **pre-connect behavior** (optimistic / queue / signify). The result is a **committed inventory table — the spine of this arc**: item 3 draws its fix subitems from it, and worklist 3 drives the dead-window handling from the same columns.
	2. [ ] **Classify against §6.** For each inventoried action, mark whether it satisfies §6.2 (optimistic for fast ops), §6.5 (view-state never waits on network), and §6.6 (confirm preserves optimism); flag any that still gate input or feedback on a round trip.
	3. [ ] **Close the remaining gaps — one subitem per flagged action.** The subitems here are **populated from the committed inventory** (items 1–2): one per action that fails §6, added once that table exists — not begun from memory, and not before the inventory does. Each fixes its action so feedback is instant and the action is client-owned where ephemeral. Acknowledge — do not redo — the Arc 8 client-side confirm sweep and the client-side initiative-editor open as already-passing cases.
		1. [ ] **Archive confirm latency (known seed — not the scope).** One entry the inventory must capture, not the whole of item 3. The operator counts ~4 beats between clicking Archive and the confirm dialog appearing, on a fully-connected WebSocket session (i.e. not the connect/major-reload, and WebSocket did NOT fix it). Feedback is waiting on the network when it shouldn't be. A synthetic `el.click()` bridge test falsely reported this instant — the lived report is ground truth; diagnose against a real interaction, not a synthetic dispatch.

2. [ ] **Ephemeral UI state survives the LiveView lifecycle**

	Client-owned UI state (selection, open panes, expand/collapse, optimistic flips, presence) must survive every server re-render, reconnect, and initial connect. Reapply it *inside* the morphdom patch via LiveView's `dom` callbacks — preventively, so it's never clobbered and there's no race — driven by a declarative attribute convention and a small client-side store of the truth. One transport-agnostic path, replacing the per-feature observers and `KeepOpen`.

	1. [ ] **Define the client-owned convention + state store.** A declarative marker (e.g. a `data-keep-*` prefix) for what's client-owned — selection, open panes, expand/collapse, optimistic flips, presence — plus the small JS store of the truth (selected id, expanded ids) that survives re-renders and seeds re-added elements.
	2. [ ] **Apply it through the patch.** `onBeforeElUpdated` carries the marked attributes/classes onto elements updated in place; `onNodeAdded` re-applies from the store onto elements morphdom removes and re-adds (reorder, reset re-stream). Together they cover both cases; handle attribute removal too.
	3. [ ] **Migrate existing state onto the convention** — selection, editor/details panes, expand/collapse, optimistic flips, presence badges.
	4. [ ] **Retire the racing observers, keep the store.** Remove the `DoitSelection` / `DoitInitiativeEditor` MutationObserver re-assert code and `KeepOpen`; the underlying state store stays — it's now the single source the `dom` callbacks read.
	5. [ ] **Verify the known-failing case.** A pane opened during the pre-connect window survives the join patch (bridge-verified failing today) — confirm on **LongPoll**, where it surfaced, not just WebSocket.

3. [ ] **Input always registers — no "looks ready but isn't" dead window**

	The user must see their input did something up front. Eliminate the gap where the page is painted but not yet interactive and actions are silently lost.

	1. [ ] **Connecting / in-flight signifier.** Show a client signifier while the socket is connecting or an action is in flight, so the page never looks fully interactive before it is.
	2. [ ] **No dropped pre-connect actions.** Handle each pre-connect action per the 1.1 inventory's **pre-connect-behavior** column — client-ownable applied optimistically, server-gated queued — so an action during the dead window is never silently lost.
	3. [ ] **Up-front acknowledgement.** Close the "feedback immediate?" gaps the 1.1 inventory flags: every input produces visible acknowledgement at the moment of the action, independent of connect state.

4. [ ] **Seamless navigation — master-detail, no remount**

	Opening an initiative from the workspace should patch the detail region, not remount and swap the whole page. The workspace list and initiative view become one LiveView (or one `live_session`) navigated by `push_patch` — the established collaborative-app pattern that keeps the live session, and your presence, intact instead of flickering you out and back in for everyone else viewing.

	1. [ ] **List + detail in one LiveView.** Navigation between the workspace and an initiative becomes `push_patch` / `handle_params`, not `push_navigate` / remount. Keep the module tidy (LiveComponents / a clear list-vs-detail state split) so it doesn't become a monolith.
	2. [ ] **Patch the detail, keep the session.** Navigating re-renders only the detail region; the socket, PubSub subscriptions, and presence stay intact — no re-track, no flicker for other viewers, no reconnect.
	3. [ ] **Opens instantly.** With no remount, opening an initiative is instant view-state plus an optimistic / in-flight fill of the detail — never a frozen swap. The patch rides the same re-assert path as worklist 2 (it must not stomp client state).

5. [ ] **Audit & anti-regression**

	1. [ ] **Whole-app §6 sweep.** Walk the full app against [`UX_GUARDRAILS`](../../UX_GUARDRAILS.md) §6 (6.1 progress < ~100ms, 6.2 optimistic, 6.5 view-state off the network, 6.6 confirm preserves optimism) and record findings.
	2. [ ] **Codify the client-first pattern.** Document the client-first interaction pattern (where ephemeral state lives, how optimistic writes reconcile, how new UI joins the re-assert path) so new features inherit it instead of re-introducing round-trips. Note the guardrail in [`UX_GUARDRAILS`](../../UX_GUARDRAILS.md) §6 if a line is missing.

6. [ ] **Testing**

	1. [ ] **Tests.** `mix test` coverage for any new client/server contract surfaced by the re-assert mechanism and the in-flight signifier (the automated gate per house rules).
	2. [ ] **Usability testing** — [Human] / bridge eye-pass once built; mix tests ride each impl item.
		1. [ ] [Human] Open a pane (details or initiative editor) immediately on page load and confirm it survives the connect on **LongPoll** — not just WebSocket.
		2. [ ] [Human] Reconnect (drop and restore the socket) with selection + an open pane + an expand/collapse state set; all ephemeral UI state is restored after reconnect.
		3. [ ] [Human] Input feedback is immediate on every direct-manipulation action — selection, expand/collapse, editor open, optimistic flip — with no visible wait on the network.
		4. [ ] [Human] During the pre-connect window the page shows a connecting/in-flight signifier; an action taken then is acknowledged and not silently dropped once connected.
		5. [ ] [Human] Light + dark: the connecting / in-flight signifier reads correctly in both modes.

## Cross-cutting requirements

- **Spec compliance (WHAT vs HOW).** The end-state is the spec — [`UX_GUARDRAILS`](../../UX_GUARDRAILS.md) §6 + §7.3–7.4; this arc is the HOW that brings the app to it. The standing release gate in [`CLAUDE.md`](../../../CLAUDE.md) (Engineering Style) keeps new features inside it. (Subsumes the old LongPoll-parity, no-round-trip-before-input, and not-a-rewrite notes — now in the spec / intro.)
- **One preserve path.** Worklist 2's `onBeforeElUpdated` hook is the single owner of keeping client-owned state through patches/reconnects/initial connect — preventive, not corrective; per-feature guards collapse into it (worklist 2 item 4).
- **A11y.** The connecting / in-flight signifier is perceivable (not color-only) and announced; restored focus after a patch lands where the user left it.
