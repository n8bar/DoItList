# m02.07 — Personal Workspace

_Parent: [`m02-ux-buildout.md`](m02-ux-buildout.md)_

The user's personal, cross-Initiative surfaces — what concerns *them* across
all their Initiatives, not any single one. Two pieces:

- **Assigned to Me** (items 1–5): a new nav item beside Initiatives opening a
  flat, cross-Initiative list of every task on the user's plate — the
  assignment-centric counterpart to the ownership-centric Initiatives list.
- **Notifications** (item 6): an unread feed of things that happened *to* the
  user (added / removed, assigned / unassigned), with a flyout in the user
  menu and a red dot on the nav avatar.

Promoted from BACKLOG (operator, 2026-06-14); Notifications added the same day.
**Depends on co-assignees ([`m02.05`](m02.05-wide-width-layout.md) item 13)**
for the direct-vs-co distinction — so this arc runs after Arc 5.

**Draft — pending operator approval.**

## Items

1. [ ] **Page + route + nav item.** Authenticated `live "/assigned"` (or
   similar), reachable from a new **Assigned to Me** nav entry beside
   Initiatives (desktop nav + mobile hamburger). View title "Assigned to Me",
   matching the "My Initiatives" h1 style.
2. [ ] **Cross-Initiative query.** Every task where the user is **primary or
   co-assignee**, scoped to Initiatives they're **currently a member of** — so
   tasks in Initiatives they've left / been removed from / that were trashed
   fall out automatically (no special-casing; it's a current-member join).
   Completed tasks **hidden by default**; an on-page toggle reveals them.
3. [ ] **Row display.** Each row clearly differentiates **direct (primary)**
   from **co-assignee**, and shows the task's leaf/child count (reuse the
   mode-aware badge from `m02.04`-era work). The list is **flat — not
   expandable here**; the count conveys subtree size without a second tree UI.
4. [ ] **Group by Initiative.** An on-page, user-level **persistent** "Group
   by Initiative" checkbox, **unchecked by default** (flat list). Same
   control-on-page / storage-on-prefs pattern as the index sort
   ([`m02.04`](m02.04-account-details.md) item 2.6) — the toggle lives on this
   page, its state on the `user_preferences` record, so it follows the account.
5. [ ] **Deep-link to task (the one new mechanism).** A row can open its
   Initiative landing with that task **selected, its collapsed ancestors
   expanded, and scrolled into view**. The tree has no deep-link-to-a-task
   today (selection is client/DOM-owned), so this adds: a task param the show
   LiveView reads on mount → select it, expand the ancestor chain so it's
   visible, and scroll it into view. Honors the no-round-trip-before-input and
   no-layout-jank guardrails.
6. [ ] **Notifications.** A per-user, cross-Initiative **unread feed** — distinct from the per-task Activity log (that's the full per-task record; this is the unread subset of things that happened *to me*).
	1. [ ] New `notifications` table: `user_id`, `kind`, `data` (JSON), `read_at`, `inserted_at`.
	2. [ ] **Auto-generated** on: added / removed as an Initiative member; a task's **primary** assignee set or cleared on you; **co-assignee** added / removed (being co-assigned counts as being assigned). **Never notify the actor about their own action** — actor == recipient → skip. Hooks live where we already broadcast `members_changed` / record activity events (local additions, no new plumbing).
	3. [ ] **Flyout** in the User menu (desktop dropdown) and the hamburger (mobile): recent notifications, newest first, each linking to its subject (the Initiative or — via Arc 7's deep-link — the task); opening the flyout marks them read; a "mark all read" affordance.
	4. [ ] **Unread indicator:** a **red dot** on the nav avatar while any unread exist; clears when read.
	5. [ ] **Real-time:** new notifications push live on a per-user PubSub topic, so the dot appears without a reload.
	6. [ ] Tests: generation per event type, self-exclusion, unread count, mark-read, live push.
7. [ ] **Tests.** Query correctness (primary + co; current-member scoping;
   completed hidden/shown; left-Initiative tasks excluded); group-by toggle
   persistence; deep-link selects + reveals the right task.
8. [ ] **Usability testing** — the [Human] eye-pass once the above are built;
   mix tests ride each implementation item.
	1. [ ] [Human] Assigned tasks across several Initiatives all appear; direct vs co-assignee reads at a glance; light + dark.
	2. [ ] [Human] Completed hidden by default; the reveal toggle shows/hides them.
	3. [ ] [Human] "Group by Initiative" groups, persists across a reload, and follows the account on another device/browser.
	4. [ ] [Human] Leaf/child counts read correctly; rows don't expand here.
	5. [ ] [Human] Clicking a row opens its Initiative with that task selected, ancestors expanded, and scrolled into view — no jank.
	6. [ ] [Human] A task in an Initiative you've left no longer appears.
	7. [ ] [Human] Notifications: being added/assigned (by someone else) raises the red dot + a flyout entry; your own self-actions never notify; opening the flyout clears the dot; a notification arriving live raises the dot without a reload.

## Cross-cutting requirements

- **No new schema.** Pure read over `tasks` + the co-assignee table + the
  membership join. The persistent group-by toggle reuses the existing
  `user_preferences` record (one new boolean column).
- **Light + dark visual review** of the new page and nav entry.
- **A11y.** The list is keyboard-navigable; the deep-link open is reachable by
  keyboard with visible focus, and the landed-on task takes focus / is
  announced.
