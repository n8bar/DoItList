## Goal

## Cross-cutting requirements

- **No silent data loss.** An undo that can't fully restore prior state (e.g. cascading deletes lost some child rows) MUST tell the user what was preserved and what wasn't.
- **Real-time correctness.** Undo / redo broadcasts the same `task_*` PubSub events as any other mutation â€” other users' open tabs reflect the change immediately.
- **Light + dark visual review** of the toolbar affordance and any undo-related flash messages.
- **A11y.** Keyboard path is the universal one; toolbar buttons are tab-reachable with visible focus rings.

## Open Questions

- **Scope of the stack: per-Initiative or global?** Per-Initiative is simpler and matches how users mentally model the work; global is more familiar from text editors. Initial cut: per-Initiative. Revisit if the global model feels missing.
- **Depth.** 50 entries per (user, Initiative)? Higher? Should this be configurable per-user later?
- **Delete reversibility.** Restoring a deleted task at its original id requires either preserving the id (allow Repo.insert with explicit id) or accepting that the restored task gets a new id (breaks anything that referenced the old id â€” e.g. a link someone copied). Decide before implementing item 2's `deleted` inverse.
- **Macro-undo for bulk operations.** If we add bulk operations later (multi-select reorganize, bulk weight change), should they undo as one atom or per-event? Out of scope for v1, but worth flagging.
- **Group-by-time-window?** Some editors collapse adjacent same-kind edits ("typed 'hello'" rather than 5 keystrokes). Title-rename keystrokes are the obvious candidate. Initial cut: no grouping â€” every event is its own undoable unit. Revisit if it feels noisy.
- **Cross-session persistence.** Stack pointer persists for the user in DB? Or resets on logout? Initial cut: persists.

## Implementation Notes

- **Hook into existing event recording.** Every mutation in `Tasks` already calls `record_event/4`. Extending each call site to also stash `inverse_payload` is local and reviewable. Resist creating a parallel "undo log" â€” one event table, one source of truth.
- **Don't make every mutation undoable on day one.** Start with the destructive / surprising ones: move, create, delete, status flip, weight change. Add the rest (title, description, manual_progress changes) once the engine is proven.
- **Optimistic UI.** Like Arc 3 drops, render undo/redo client-side immediately on keystroke; revert on server failure.
- **Keyboard plumbing.** The global key listener already exists for Alt+arrows (Arc 3 item 6). Extending it to Ctrl+Z / Ctrl+Shift+Z is a small addition, not a new mechanism.
