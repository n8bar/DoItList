# m02.06 — Undo & Trash
_Parent: [`m02-ux-buildout.md`](m02-ux-buildout.md)_

A per-user undo/redo stack for the destructive / surprising operations that Arc 3 and the rest of M02 now expose, plus a Trash for Initiative-level deletion (promoted from BACKLOG — Account deletion in [`m02.04`](m02.04-account-details.md) needs it). Drag-and-drop especially can flip ancestor checkmarks (Arc 3 introduced a confirmation modal for that — see [`m02.03`](m02.03-task-tree.md)), but a confirm-after-the-fact safety net is still the better long-term answer than per-operation modals.

## Goal

Make every mutation reversible in one keystroke. The user should never feel they have to be careful — they can experiment freely, and any unwanted change is one `Ctrl+Z` away. Same shape Redo for `Ctrl+Shift+Z` / `Ctrl+Y`.

## Items

1. [x] **Activity-event-backed action log.** Reuse the existing `activity_events` schema (where `parent_changed`, `reordered`, `created`, `deleted`, etc. already land) as the source of truth. Add an `inverse_payload` JSON column so each event carries enough state to reverse itself. No separate undo table. (Plus the soft-delete foundation: `tasks.deleted_at`, reads filtered, `delete_task`→soft-delete-subtree, `restore_tasks`.)
2. [x] **Inverse-action engine.** _(Done as `Tasks.undo/2` + `redo/2`. status_changed deferred — not currently recorded as an event + cascade reversal; co-assignee set events deferred too.)_ A `Tasks.undo_event/2` (and `redo_event/2`) function that takes an event row + actor, applies the inverse mutation in a transaction, and records a *new* event (`undid`/`redid`) so the stack stays linear and PubSub broadcasts fire normally. Inverse implementations per event type: `created`→soft-delete, `deleted`→restore (clear `deleted_at`), `parent_changed`→move back, `reordered`→reorder back, `progress_changed`→reset, `title_changed`→reset, `status_changed`→reset. The non-delete inverses read the `from`/`to` the events already store; delete/create lean on the soft-delete flag (id, comments, co-assignees, and the subtree all preserved).
3. [x] **Per-user-per-Initiative stack semantics.** Undo only pops *your own* events on the current Initiative. Another user's edits in between yours don't get undone — they're stepped over. Stack pointer per (user, Initiative) tracked server-side so a refresh / reconnect doesn't lose position.
4. [x] **Keyboard shortcuts.** `Ctrl+Z` / `Cmd+Z` = undo; `Ctrl+Shift+Z` / `Cmd+Shift+Z` / `Ctrl+Y` = redo. Wire through the same global listener as the Alt+arrow keys from Arc 3 item 6. Suppress when focus is in an input/textarea/contenteditable so native field-level undo still works.
5. [x] **Toolbar affordance.** Undo / Redo buttons in the Initiative header (or wherever feels right at implementation — judgment call). Disabled state when no stack to pop in that direction. Tooltip shows the action that would be undone/redone ("Undo: Move 'Buy milk' under 'Errands'").
6. [x] **Bounded depth.** Per (user, Initiative) keep the most recent N events as candidates for undo; older ones drop off. N is a config constant — start at 50, revisit after dogfooding.
7. [x] **Conflict handling.** If an inverse would fail (target task was deleted by another user, parent no longer exists, etc.), surface a friendly flash ("Couldn't undo — 'Buy milk' has been deleted") and step the stack pointer past the dead entry. Never leave the user stuck.
8. [x] **Activity feed integration.** Undo / redo entries appear in the activity feed labeled as such ("undid 'Move Buy milk under Errands'"). The feed is the canonical record; it doesn't try to hide the round-trip.
9. [x] **Tests.** Per-event-type round-trip tests (do → undo → state matches pre-do; do → undo → redo → state matches post-do). Cross-user test: A's undo doesn't touch B's edits. Conflict tests: undoing a deleted-target case surfaces the flash and advances the pointer.

10. [x] **Trash (Initiative-level)**

	1. [x] **Trash — owner-only soft-delete for Initiatives.** Deleting an Initiative routes it to Trash (`trashed_at`) instead of hard delete; global — a trashed Initiative leaves every member's dashboard. A Trash surface (probably on/near the Initiatives index) lists the owner's trashed Initiatives with Restore; permanent deletion from Trash stays owner-only (today's hard delete becomes that final step).
	2. [x] **Retention window + sweep.** Trashed Initiatives purge automatically after a retention period (config constant — start at 30 days; this is the "duration" `m02.04` § Account details item 10 leans on). Owner can purge sooner manually. _(Sweep runs lazily on the index mount; a scheduled job can call `purge_expired_trash/0` later.)_
	3. [x] **Account-deletion handoff** (pairs with `m02.04` § Account details item 10). Deleting an account that owns a **multi-member** Initiative **transfers ownership** to the highest-ranked surviving member — rank **editor > viewer+ > viewer** (the viewer+ tier applies only when that Initiative's `viewer_plus` is on; viewer+ = a viewer who is the *primary* assignee of ≥1 task there). Ties within the top rank go to whoever **joined the Initiative earliest** (oldest `initiative_members` row, any role). **Solo-owned** Initiatives are deleted with the account. `owner_id` stays non-null — the transfer reassigns it to a live member before the account is removed, satisfying the `on_delete: :restrict` FK, so no schema change.
	4. [x] **Tests.** Trash → restore → purge round-trips ✅; retention sweep ✅; trashed Initiatives invisible to all members ✅; account-deletion handoff — transfer + selection rule (editor > viewer+ > viewer, oldest tiebreak, viewer_plus-off) ✅.

11. [x] **Undo re-scope — one shared timeline per Initiative.** Reworks the per-user stack from items 2–3 (and the depth bound in item 6): undo/redo act on the Initiative's own event timeline, not each user's, so anyone with rights reverses the most recent action regardless of who made it.
	1. [x] Undo (and redo) target the newest un-undone event **on the Initiative**, any actor — gated on whether the acting user has rights for that op (owner/editor: any task op; viewer+: only ops within their privileges).
	2. [x] **viewer+ cutoff.** The first top-of-stack op a viewer+ lacks rights for is a hard wall — they undo the contiguous run above it and stop; no skipping or reordering past it. Redo is symmetric.
	3. [x] **Shared redo.** Any actor's fresh action clears the Initiative's redo stack — one timeline, so a new branch discards the old redo for everyone.
	4. [x] **Depth 500 per Initiative** (supersedes the 50-per-(user, Initiative) bound in item 6).
	5. [x] Tests: a user undoes another's most-recent action when they have rights; viewer+ undo stops at the first out-of-rights op; a fresh action clears redo; depth bounded at 500.

12. [x] **One undo event per text edit (focus → blur).** A title / description edit is a single undoable unit spanning focus to blur — not per keystroke or per change; focusing and selecting are not events. While a text field has focus the browser's native field-level undo stays in charge (item 4 already suppresses our shortcut there); our event records on blur.

13. [x] **Select + scroll the affected task on undo/redo.** After an inverse applies, the affected task becomes the selection and scrolls into view, so the user sees exactly what changed.

14. [x] **Make completion (done ⇄ open) undoable.** The complete / incomplete toggle isn't recorded as an undoable event (deferred in item 2), so it can't be reversed — a gap against the arc's "every mutation reversible in one keystroke" goal, and the undo users most expect. Worse, completion isn't even silent: the done⇒`manual_progress`=100 coupling (`maybe_set_done_progress`) plus the up/down status cascade each fire their own `progress_changed`, so one completion **fragments into N events up the ancestor chain to the root** — and on the shared timeline an out-of-domain ancestor event tops the stack and walls a viewer+ off from undoing their own in-subtree completion ("Nothing to undo"; reproduced on Cheese).
	1. [x] Record completion as **one** `status_changed` event on the task the user toggled — not per keystroke, not one per cascaded task. Inverse data captures every task the flip touched (the task itself, descendants flipped by an Arc-3 down-cascade, **and** ancestors flipped by the up-cascade in `check_completed_ancestors` / `uncheck_done_ancestors`), each with its prior `status` and `manual_progress`.
	2. [x] Stop the fragmentation: the done⇒`manual_progress` coupling and the cascade must not each emit their own undoable `progress_changed` / `status_changed` — the single completion event subsumes them. (Done via a `record_events?: false` pass on `update_task`; without this, the out-of-domain root event keeps walling the viewer+ off.)
	3. [x] Inverse: undo restores prior `status` + `manual_progress` for every captured task and recomputes roll-ups; redo re-applies. Rights-gated on the acted task (a viewer+ may undo a completion within their led subtree).
	4. [x] Broadcast the reversal like the forward path: one `{:task_updated}` per task the flip touched. `patch_task` refreshes the acted task's whole lineage (so the leaf flip and any up-cascade land in one patch), and a branch down-cascade's descendants each get their own — incremental for everyone, no structural reload.
	5. [x] Tests: complete a leaf that cascades ancestors done → one event → undo reopens leaf + ancestors atomically → redo re-completes; complete a branch that cascades descendants → same; cross-user; a viewer+ can undo a completion in their led subtree with no out-of-domain ancestor event blocking them; the reversal patches each affected task rather than a full reload.

## Cross-cutting requirements

- **No silent data loss.** An undo that can't fully restore prior state (e.g. cascading deletes lost some child rows) MUST tell the user what was preserved and what wasn't.
- **Real-time correctness.** Undo / redo broadcasts the same `task_*` PubSub events as any other mutation — other users' open tabs reflect the change immediately.
- **Light + dark visual review** of the toolbar affordance and any undo-related flash messages.
- **A11y.** Keyboard path is the universal one; toolbar buttons are tab-reachable with visible focus rings.

## Open Questions

- **Macro-undo for bulk operations.** If we add bulk operations later (multi-select reorganize, bulk weight change), should they undo as one atom or per-event? Decide per bulk-action-type when each lands — out of scope for v1.

## Implementation Notes

- **Hook into existing event recording.** Every mutation in `Tasks` already calls `record_event/4`. Extending each call site to also stash `inverse_payload` is local and reviewable. Resist creating a parallel "undo log" — one event table, one source of truth.
- **Don't make every mutation undoable on day one.** Start with the destructive / surprising ones: move, create, delete, status flip, weight change. Add the rest (title, description, manual_progress changes) once the engine is proven.
- **Optimistic UI.** Like Arc 3 drops, render undo/redo client-side immediately on keystroke; revert on server failure.
- **Keyboard plumbing.** The global key listener already exists for Alt+arrows (Arc 3 item 6). Extending it to Ctrl+Z / Ctrl+Shift+Z is a small addition, not a new mechanism.
