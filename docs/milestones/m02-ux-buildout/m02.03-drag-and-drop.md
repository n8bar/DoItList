# m02.03 — Drag-and-Drop Reorganization
_Parent: [`m02-ux-buildout.md`](m02-ux-buildout.md)_

Drag-and-drop reorder + reparent within an Initiative, per [`ProductSpec.md`](../../ProductSpec.md) § Reorganization. Cross-Initiative moves are out of scope.

Owner-approved 2026-05-09. Items below are scoped; checkboxes flip to `[x]` as each ships.

## Items

_Items 9 and 10 were added 2026-05-15 — completion-state confirmation modal and the drop-zone redesign — as natural follow-ons to the original 8 items shipping in `f1db66d`. Arc 3's theme (drag-and-drop reorganization) covers both._

1. [x] **Drag handle on each task row.** A 3-vertical-dots glyph at the far left of the row (left of the chevron / checkbox) is the only place a drag starts. Prevents accidental drags from row-body clicks. Handle has its own touch target meeting the 44×44 px rule from UX_GUARDRAILS.
2. [x] **Persistence + roll-up recompute.** Every drop computes new `parent_id` + `sort_order`, persists in a single transaction, and recomputes ancestor progress for both the old and new parent chains. Failures roll back the entire drop and surface a flash; the UI reverts to pre-drag state. Lands as a `Tasks.move_task/3` context function before any gesture wiring so items 3–7 can call into a real backend rather than stubs.
3. [x] **Vertical drag → reparent.** Dropping task A onto task B's row makes A a child of B (appended at the end of B's children). Drop target highlights during hover. After commit, ancestor progress recomputes for both old and new parent chains (via item 2).
4. [x] ~~**Horizontal drag → indent / dedent.**~~ While dragging, horizontal cursor position controls depth: dragging right past a threshold indents A under its previous sibling; dragging left dedents A toward an ancestor's level. A subtle indent guide tracks the cursor so the target depth is visible before drop. _Superseded 2026-05-19 by item 13 — horizontal mode rolled back._
5. [x] ~~**Snap-back zone.**~~ A horizontal cursor band inside the row that means "no change" — letting go inside that band aborts the move with no commit. Lets the user back out mid-drag without committing a wrong drop. _Superseded 2026-05-19 by item 13 — gone with horizontal mode._
6. [x] **Keyboard alternative — reuse task-selected state.** A task is "selected" when its Details panel is open (the existing selection model from Arc 2 item 6 / 21). While a task is selected, keyboard shortcuts reorder and reparent that task: `Alt+↑/↓` reorders within siblings; `Alt+←/→` dedents / indents. No mouse needed; no separate "drag focus" concept — selection IS the keyboard focus for reorganization.
7. [x] **Cross-list moves within the same Initiative.** Dragging across List trees changes A's root List. Cross-Initiative moves remain out of scope (ProductSpec § Reorganization). _Note: this item required no new code — the backend (item 2) and the drag gesture (items 3–5) have no list-specific code paths, so cross-list moves work transparently. Verified by an explicit "cross-list" test in `test/doit/tasks_test.exs` covering both source/dest root recompute and the Initiative-level aggregate change._
8. [x] **Mobile touch — long-press to start drag.** Long-press (~400 ms) on the drag handle starts the drag; the rest of the gesture mirrors mouse behavior. Short taps on the handle do nothing — the handle is drag-only, not a button.
9. [x] **Completion-state confirmation modal.** Create + move actions that would auto-flip an ancestor's `status: done` (in either direction) prompt a confirmation listing the affected tasks before committing. Without this, a drag can silently uncomplete a parent (or auto-complete one) — too surprising. `Tasks.preview_create/2` and `Tasks.preview_move/3` dry-run via transaction rollback, snapshot ancestor chains before / after, and classify as scenario 1 (would uncomplete), 2 (would complete), 3 (both), or `nil` (no flip). The modal renders in 1/2/3; `nil` commits straight through. Driven by a `:pending_action` socket assign + `confirm_pending` / `cancel_pending` events. Side effect: branch progress reconciliation moves out of `Progress.compute` (which used to short-circuit `status: done → 100`) and into `Tasks` proper — branches always derive from children, so a parent that gains an incomplete child reflects truth instead of masking it. Branches still reach 100 once all leaves do.
10. [x] ~~**Drop-zone two-mode redesign.**~~ The original items 4–5 specified a single horizontal band whose meaning depended on the cursor's horizontal position alone — which made dropping on a *different* row ambiguous with indent / dedent on the *source* row. The 2026-05-13 redesign splits drop-zone semantics into two modes selected by vertical distance from the source row's center: **horizontal mode** (within ±~1.5× row height of source center) treats horizontal cursor position as indent / dedent — items 4 + 5 behaviors apply; **vertical mode** (cursor moved off the source's vertical slot) treats the row under the cursor as anchor, drops as child-of-anchor — item 3 behavior applies. Wider snap-back band (64 px each side) in horizontal mode reflects the increased clarity. Indent guide renders across both modes. Tunable constants live in `assets/js/app.js`: `SNAP_BAND_PX`, `DEDENT_STEP_PX`, `VISUAL_INDENT_PX`, `SOURCE_SLOT_TOLERANCE`. _Superseded 2026-05-19 by item 13 — only the vertical-mode half survives; horizontal mode and its tuning constants are removed._

12. [ ] **Manual progress on branches — preserve, disable, explain.** _Added 2026-05-19, surfaced during 11.4.1 in-browser testing._

    When a task gains children and becomes a branch, its `manual_progress` is dormant — `Progress.compute` reads `manual_progress` only in `leaf_progress/1`, which never runs for a task with children. The value is preserved in the DB and returns to use if all children are later removed. But today the Details panel hides this from the user entirely: for branches, the editable slider is replaced by the italic line _"Computed from children: X%"_ and the manual value is invisible. This creates a quiet round-trip surprise — leaf → branch → leaf restores a value the user thought was gone (or "should be gone," if they manually edited it once and assumed it was overwritten).

    Replace the branch-mode rendering in `task_editor/1` with:

    - The same `<input type="range">` slider as leaves, but `disabled`, set to the dormant `manual_progress` value. The slider's filled track + thumb position visibly carries the value, so the user can see their work isn't lost.
    - A short inline hint below the slider in muted text: _"Ignored — this task has subtasks."_
    - A small info icon next to the slider's label that opens a popover with the longer explanation: _"Progress on a task with subtasks is calculated from its subtasks instead of being set manually. Your manual value is kept and will start being used again if you remove all subtasks."_
    - Keep the existing "Computed from children: X%" line nearby so the active value is also visible.

    New reusable `<.info_hint>` component in `CoreComponents` — small popover-on-click pattern, reuse-ready for future "explain UI semantics" cases (weight on tasks with weighted children, the leaf-done snap to 100, the roll-up formula on the underbar, etc.). Build the component minimal but composable: pass the popover content as a slot or attribute; keep position / styling consistent so future uses don't drift.

    No `Tasks` context changes, no schema, no migration. Two files: `lib/doit_web/components/core_components.ex` (new component) and `lib/doit_web/live/initiative_show_live.ex` (`task_editor/1` template). Light + dark visual check on the disabled slider and popover.
13. [ ] **Vertical-only drag — horizontal mode rolled back.** _Added 2026-05-19, supersedes items 4, 5, and the horizontal half of item 10._

    In-browser testing of items 4–5 surfaced that horizontal cursor-position-as-depth is non-obvious; the snap-back band exists only to manage the resulting ambiguity. Meanwhile keyboard `Alt+←/→` (item 6) already covers indent / dedent cleanly and is discoverable through the Details-panel selection model. The horizontal mode is cleverness we don't need.

    Drag gesture becomes single-mode: cursor position picks the row under the cursor (anchor); drop = child-of-anchor, appended to the anchor's children. No horizontal sensitivity, no indent guide, no snap-back band. The current row-highlight visual feedback stays.

    **Sibling reorder via drag is not in scope here.** Today vertical drag only changes parent; with horizontal gone, sibling reorder remains keyboard-only (`Alt+↑/↓`). Owner has flagged sibling reorder via drag as forthcoming Arc 3 work, paired with a broader sibling sort-order toolset — design pending; a separate item (and likely a ProductSpec update) will land when those decisions are made.

    Code changes — `assets/js/app.js` only:

    - Remove the HORIZONTAL MODE branch from `handleMove` / `computeHorizontalPlan`; keep only vertical-mode anchor resolution.
    - Remove the `SNAP_BAND_PX`, `DEDENT_STEP_PX`, `VISUAL_INDENT_PX`, `SOURCE_SLOT_TOLERANCE` constants.
    - Remove the indent-guide rendering.
    - `SOURCE_SLOT_TOLERANCE`-based "snap-back when over source" → small explicit guard: hovering the source row is a no-op drop (matches item 11.4.5).

## Cross-arc note

The drag infrastructure picked here (library, hook abstraction, motion conventions, drop-feedback visuals) needs to be **reusable for Arc 4 item 11** — drag-a-person from the cross-Initiative people pane onto an Initiative entry in the left rail. Same primitives, different source / target. Pick something that can host both gestures without forking; favor a small custom hook over a heavy library if the heavy library only fits one of the two cases.

## Implementation Notes

- **Library vs. roll-your-own.** _Decided 2026-05-09: custom `phx-hook` over pointer events._ Sortable.js wrappers handle vertical reorder cleanly but don't extend naturally to the horizontal indent guide, snap-back zone, or the cross-pane Arc 4 item 11 gesture (drag-a-person onto an Initiative entry). One small custom hook serves both arcs without forking.
- **Sort order.** Use float-style ordering (`sort_order` as a numeric column) so single-row updates are common and full re-numbering is rare. Re-number lazily when ordering precision degrades.
- **Drop target visuals.** Highlight on hover; show indent guide for horizontal position; respect motion-reduce.
- **Optimistic vs. server-confirmed.** Render the move client-side immediately on drop; if the server transaction fails, revert and flash. Don't wait on the round-trip to update UI.
- **Cross-list within-Initiative bookkeeping.** When A moves between Lists, both the source List and the destination List need ancestor-progress recompute (their root-task progress changes), and the Initiative-level aggregate underbar (Arc 2 item 7) updates accordingly.
