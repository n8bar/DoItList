# m02.03 ŌĆö Drag-and-Drop Reorganization
_Parent: [`m02-ux-buildout.md`](m02-ux-buildout.md)_

Drag-and-drop reorder + reparent within an Initiative, per [`ProductSpec.md`](../../ProductSpec.md) ┬¦ Reorganization. Cross-Initiative moves are out of scope.

## Items

1. [x] **Drag handle on each task row.** A 3-vertical-dots glyph at the far left of the row (left of the chevron / checkbox) is the only place a drag starts. Prevents accidental drags from row-body clicks. Handle has its own touch target meeting the 44├Ś44 px rule from UX_GUARDRAILS.
2. [x] **Persistence + roll-up recompute.** Every drop computes new `parent_id` + `sort_order`, persists in a single transaction, and recomputes ancestor progress for both the old and new parent chains. Failures roll back the entire drop and surface a flash; the UI reverts to pre-drag state. Lands as a `Tasks.move_task/3` context function before any gesture wiring so items 3ŌĆō7 can call into a real backend rather than stubs.
3. [x] **Vertical drag ŌåÆ reparent.** Dropping task A onto task B's row makes A a child of B (appended at the end of B's children). Drop target highlights during hover. After commit, ancestor progress recomputes for both old and new parent chains (via item 2).
4. [x] ~~**Horizontal drag ŌåÆ indent / dedent.**~~ While dragging, horizontal cursor position controls depth: dragging right past a threshold indents A under its previous sibling; dragging left dedents A toward an ancestor's level. A subtle indent guide tracks the cursor so the target depth is visible before drop. _Superseded by item 20._
5. [x] ~~**Snap-back zone.**~~ A horizontal cursor band inside the row that means "no change" ŌĆö letting go inside that band aborts the move with no commit. Lets the user back out mid-drag without committing a wrong drop. _Superseded by item 20._
6. [x] **Keyboard alternative ŌĆö reuse task-selected state.** A task is "selected" when its Details panel is open (the existing selection model from Arc 2 item 6 / 21). While a task is selected, keyboard shortcuts reorder and reparent that task: `Alt+Ōåæ/Ōåō` reorders within siblings; `Alt+ŌåÉ/ŌåÆ` dedents / indents. No mouse needed; no separate "drag focus" concept ŌĆö selection IS the keyboard focus for reorganization.
7. [x] **Cross-list moves within the same Initiative.** Dragging across List trees changes A's root List. Cross-Initiative moves remain out of scope (ProductSpec ┬¦ Reorganization). Verified by the cross-list test in `test/doit/tasks_test.exs` (covers source / dest root recompute and the Initiative-level aggregate change).
8. [x] **Mobile touch ŌĆö long-press to start drag.** Long-press (~400 ms) on the drag handle starts the drag; the rest of the gesture mirrors mouse behavior. Short taps on the handle do nothing ŌĆö the handle is drag-only, not a button.
9. [x] **Completion-state confirmation modal.** Create + move actions that would auto-flip an ancestor's `status: done` (in either direction) prompt a confirmation listing the affected tasks before committing. Without this, a drag can silently uncomplete a parent (or auto-complete one) ŌĆö too surprising. `Tasks.preview_create/2` and `Tasks.preview_move/3` dry-run via transaction rollback, snapshot ancestor chains before / after, and classify as scenario 1 (would uncomplete), 2 (would complete), 3 (both), or `nil` (no flip). The modal renders in 1/2/3; `nil` commits straight through. Driven by a `:pending_action` socket assign + `confirm_pending` / `cancel_pending` events. Side effect: branch progress reconciliation moves out of `Progress.compute` (which used to short-circuit `status: done ŌåÆ 100`) and into `Tasks` proper ŌĆö branches always derive from children, so a parent that gains an incomplete child reflects truth instead of masking it. Branches still reach 100 once all leaves do.
10. [x] ~~**Drop-zone two-mode redesign.**~~ A two-mode model selected by vertical distance from the source row's center: horizontal mode (within ┬▒~1.5├Ś row height of source center) treats horizontal cursor position as indent / dedent ŌĆö items 4 + 5 behaviors apply; vertical mode (cursor moved off the source's vertical slot) treats the row under the cursor as anchor, drops as child-of-anchor ŌĆö item 3 behavior applies. Wider snap-back band (64 px each side) in horizontal mode. Tunable constants live in `assets/js/app.js`. _Superseded by item 20 ŌĆö only the vertical-mode half survives; horizontal mode and its tuning constants are removed._
11. [x] **Sort-mode storage + inheritance.** Add nullable `sort_mode` column and non-null `sort_reverse` boolean (default false) to `tasks` and `initiatives`. `sort_mode` values: `"manual"`, `"alphabetical"`, `"completion"`, `"computed_progress"`, `"priority"`, `"created"`, `"updated"`; `nil` = inherit. `"weight"` is supported by the engine but not surfaced in the UI yet ŌĆö deferred. `sort_reverse` flips each mode's natural direction. Helper `Tasks.resolve_sort/1` walks parent ŌåÆ root ŌåÆ initiative ŌåÆ fallback `{"manual", false}` and returns `{mode, reverse}` so direction always travels with the mode that owns it. Pure, unit-tested.
12. [x] **Sibling sort engine.** Pure function `Tasks.Sort.apply/2`: takes children + resolved mode, returns the list with new `sort_order` values. No DB. `"manual"` is a no-op. Stable sort with `id` as tiebreaker. Empty / single-element lists return unchanged. Tested standalone.
13. [x] **Sort menu in Details panel.** "Sort children by" dropdown in the Details panel for branch tasks only. Options: `Inherit` (default), criteria from item 11 (no `Weight` ŌĆö deferred), `Manual` at the bottom. A "Reverse" checkbox sits next to the dropdown ŌĆö disabled when the resolved mode is `Inherit` or `Manual`. Selecting a criterion or toggling reverse ŌåÆ set `sort_mode` + `sort_reverse`, run engine (item 12), persist new `sort_order`, broadcast. UI sits below the "Computed from children: X%" line. Auto-sorted parents also re-sort their children automatically whenever a child's sort-key field changes (status, computed_progress, priority, title) ŌĆö `Tasks.maybe_resort_children/1` is called at every mutation site (create / update / move / recompute walk-up) so the order stays in sync with live data.
14. [ ] **Resort all posterity.** "Apply to all descendants" action in the sort menu (item 13). Walks the subtree, sets each descendant parent's `sort_mode`, re-sorts via engine (item 12) ŌĆö one transaction. Confirm modal if the subtree contains more than ~10 parent tasks (overwrites prior sort decisions; reversible only via Arc 5 undo).
15. [ ] **Drag drop-band semantics.** Row-internal drop bands in `assets/js/app.js`. Center 50% ŌåÆ Reparent (item 3 behavior). Top 25% ŌåÆ Sibling reorder above. Bottom 25% ŌåÆ Sibling reorder below. Edge bands render a placeholder in the destination slot; center keeps the row-highlight.
16. [ ] **Top + bottom root-level overlay drop zones.** Two slim overlay drop targets, rendered only during drag ŌĆö one above the first root, one below the last. Non-root source ŌåÆ Promotion (lands at top or bottom of root list). Root source ŌåÆ root-list reorder. Hook adds them on `beginDrag`, removes on `cleanup`.
17. [ ] **New-task placement defaults.** Update `Tasks.create_task/2` and `Tasks.move_task/3`:

    - Auto-sorted parent ŌåÆ run engine (item 12); insert at sort position.
    - Manual parent, via entry form ŌåÆ form's slot (existing `parent_id` + sibling-position params already handle).

18. [ ] **Sibling reorder mode flip.** Any reorder action (drag bands item 15, root overlay item 16, or keyboard `Alt+Ōåæ/Ōåō`) sets parent `sort_mode = "manual"` and overwrites `sort_order` with the new sequence. Same transaction as the reorder; broadcast carries both.

## Cross-arc note

The drag infrastructure picked here (library, hook abstraction, motion conventions, drop-feedback visuals) needs to be **reusable for Arc 4 item 11** ŌĆö drag-a-person from the cross-Initiative people pane onto an Initiative entry in the left rail. Same primitives, different source / target. Pick something that can host both gestures without forking; favor a small custom hook over a heavy library if the heavy library only fits one of the two cases.

## Implementation Notes

- **Library vs. roll-your-own.** Custom `phx-hook` over pointer events. Sortable.js wrappers don't extend cleanly to the cross-pane Arc 4 item 11 case; one small custom hook serves both gestures without forking.
- **Sort order.** Use float-style ordering (`sort_order` as a numeric column) so single-row updates are common and full re-numbering is rare. Re-number lazily when ordering precision degrades.
- **Drop target visuals.** Center-band drops highlight the anchor row (item 3); edge-band drops render a placeholder in the destination slot (item 15). Both respect motion-reduce.
- **Optimistic vs. server-confirmed.** Render the move client-side immediately on drop; if the server transaction fails, revert and flash. Don't wait on the round-trip to update UI.
- **Cross-list within-Initiative bookkeeping.** When A moves between Lists, both the source List and the destination List need ancestor-progress recompute (their root-task progress changes), and the Initiative-level aggregate underbar (Arc 2 item 7) updates accordingly.
