# m02.04 — Account Details
_Parent: [`m02-ux-buildout.md`](m02-ux-buildout.md)_

A logged-in user needs one place to view and edit their own account — today the
only account surfaces are the register / log-in / log-out routes. This arc adds
an Account Details page, and introduces **usernames** as a stable, human-typable
identity used for login and as the at-mention handle across the app. User
Preferences including such as items as default attributes for initiatives and 
tasks or anything else we think of can live here as a section unless/until those 
get a dedicated spot.

## Items

### Account details
1. [ ] **Account page + route.** Authenticated `live "/account"` (or similar), reachable from the logged-in nav.
2. [ ] **`username` field on `users`.** Add a unique, normalized username (lowercase, restricted charset, length bounds); case-insensitive uniqueness. Backfill existing users (migration is free — all data is test data for now). - **Username rules:** allowed characters (`a-z 0-9 _ -`?), min/max length, and case-insensitive matching on login + uniqueness.
3. [ ] **Set / edit username** on the account page, with live-validated uniqueness + format.
4. [ ] **Registration gains a username field.**
5. [ ] **Login accepts username _or_ email.** Field becomes `type="text"` labeled "Username or email"; auth lookup tries both (a `get_user_by_email_or_username`-style function). Replaces the current email-only `get_user_by_email`.
6. [ ] **Edit display name** (`name` stays the free-form display label, distinct from username).
7. [ ] **Change password.**
8. [ ] **Change email** — with a confirm/verify flow if we want one.
9. [ ] **Assignee chip shows `@username`.** Swap the task-row assignee chip from display name to `@username` — short and normalized, which ends the Row 1 wrap that long display names cause. Touches `initiative_show_live.ex`'s `task_node/1`.
10. [ ] **Delete account** — relationship to owned Initiatives: (ties into the Trash / ownership work in BACKLOG) if no members exist, the initiatives are deleted, otherwise they sit in trash for the duration period allowed for trashed items before being deleted permanently.
11. [ ] **Avatars.** Per-user avatar set on the account page (upload, with a generated fallback — initials or identicon); rendered wherever a user appears (member list, assignee chip, activity log).
12. [ ] **Selection-presence avatars on task rows.** When a user has a task selected, their avatar shows on that task's row for other members — probably after the pills. Selection is client-owned but the server already hears about it (`select_task` / `close_task`, debounced), so this rides Phoenix.Presence per initiative: track `{user, task_id}` on select / deselect / disconnect, broadcast, render other members' avatars on matching rows. Depends on item 12; kin to Arc 3's parked "broadcast the maybe-write hue to collaborators" idea — same fan-out shape, different signal.
