# m02.04 — Account Details
_Parent: [`m02-ux-buildout.md`](m02-ux-buildout.md)_

> **Draft — pending operator sign-off.** The username items (1–6) are scoped per
> operator direction; the rest of the account page (7–11) is still candidate.
> Confirm the full list before any implementation.

A logged-in user needs one place to view and edit their own account — today the
only account surfaces are the register / log-in / log-out routes. This arc adds
an Account Details page, and introduces **usernames** as a stable, human-typable
identity used for login and as the at-mention handle across the app.

## Preconditions
- Arc 3 (drag-and-drop) shipped; this arc is independent of it.

## Items

### Usernames + account page
1. [ ] **Account page + route.** Authenticated `live "/account"` (or similar), reachable from the logged-in nav.
2. [ ] **`username` field on `users`.** Add a unique, normalized username (lowercase, restricted charset, length bounds); case-insensitive uniqueness. Backfill existing users (migration is free — all data is test data for now).
3. [ ] **Set / edit username** on the account page, with live-validated uniqueness + format.
4. [ ] **Registration gains a username field.**
5. [ ] **Login accepts username _or_ email.** Field becomes `type="text"` labeled "Username or email"; auth lookup tries both (a `get_user_by_email_or_username`-style function). Replaces the current email-only `get_user_by_email`.
6. [ ] **Assignee chip shows `@username`.** Swap the task-row assignee chip from display name to `@username` — short and normalized, which ends the Row 1 wrap that long display names cause. Touches `initiative_show_live.ex`'s `task_node/1`.

### Rest of the account page (candidate)
7. [ ] **Edit display name** (`name` stays the free-form display label, distinct from username).
8. [ ] **Change email** — with a confirm/verify flow if we want one.
9. [ ] **Change password.**
10. [ ] **Theme preference** — persist the light/dark/system choice to `users.theme` instead of (or alongside) localStorage.
11. [ ] **Delete account** — relationship to owned Initiatives TBD (ties into the Trash / ownership work in BACKLOG).

## Open Questions
- **Username rules:** allowed characters (`a-z 0-9 _ -`?), min/max length, and confirm case-insensitive matching on login + uniqueness.
- Reuse the standard phx.gen.auth settings patterns, or hand-roll to match the app's visual style?
- Does email change require re-verification, or is this still test-data-loose for now?
- Account deletion vs. Initiative ownership: what happens to Initiatives the user owns?
