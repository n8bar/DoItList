# m02.04 — Account Details
_Parent: [`m02-ux-buildout.md`](m02-ux-buildout.md)_

A logged-in user needs one place to view and edit their own account — today the
only account surfaces are the register / log-in / log-out routes. This arc adds
an Account Details page, and introduces **usernames** as a stable, human-typable
identity used for login and as the at-mention handle across the app. User
Preferences — default attributes for initiatives and tasks, and anything else we
think of — live here as a section unless/until those get a dedicated spot.

## Items

1. [x] **Account details**
	1. [x] **Account page + route.** Authenticated `live "/account"` (or similar), reachable from the logged-in nav. Hand-rolled UI matching the app's visual style, reusing the `phx.gen.auth`-style context functions — the login/register pages set the precedent.
	2. [x] **`username` field on `users`.** Add a unique, normalized username (lowercase, restricted charset, length bounds); case-insensitive uniqueness. Backfill existing users (migration is free — all data is test data for now).
		1. [x] Username rules: allowed characters (`a-z 0-9 _ -`?), min/max length, and case-insensitive matching on login + uniqueness.
	3. [x] **Set / edit username** on the account page, with live-validated uniqueness + format.
	4. [x] **Registration gains a username field.**
	5. [x] **Login accepts username _or_ email.** Field becomes `type="text"` labeled "Username or email"; auth lookup tries both (a `get_user_by_email_or_username`-style function). Replaces the current email-only `get_user_by_email`.
	6. [x] **Edit display name** (`name` stays the free-form display label, distinct from username).
	7. [x] **Change password.**
	8. [x] **Change email** — plain field edit, no verification. Email stays a required field (so a future mailer never inherits email-less accounts), but no mailer in M02: forgotten passwords are admin resets at private scale; recovery codes or the mailer gate any public opening (BACKLOG § Account).
	9. [x] **Assignee chip shows `@username`.** Swap the task-row assignee chip from display name to `@username` — short and normalized, which ends the Row 1 wrap that long display names cause. Touches `initiative_show_live.ex`'s `task_node/1`.
	10. [x] **Delete account.** Owned Initiatives with no other members are deleted with the account; deletion is blocked while the user owns Initiatives that other members belong to ("transfer or delete these first"). The Trash flow in [`m02.06`](m02.06-undo-redo.md) supersedes the block when it lands — those Initiatives go to Trash for the retention window instead.
	11. [x] **Avatars.** Per-user generated avatar (initials or identicon — no upload in M02; upload is BACKLOG, it drags in file storage/serving/processing); rendered wherever a user appears (member list, assignee chip, activity log).
	12. [x] **Selection-presence avatars on task rows.** When a user has a task selected, their avatar shows on that task's row for other members — probably after the pills. Selection is client-owned but the server already hears about it (`select_task` / `close_task`, debounced), so this rides Phoenix.Presence per initiative: track `{user, task_id}` on select / deselect / disconnect, broadcast, render other members' avatars on matching rows. Depends on item 11; kin to Arc 3's parked "broadcast the maybe-write hue to collaborators" idea — same fan-out shape, different signal.

2. [x] **User Preferences**
	1. [x] **Theme preference** — persist the light/dark/system choice to `users.theme` instead of localStorage. Needs first-paint handling so a page load never flashes the wrong theme.
	2. [x] **My Initiative Defaults** — attributes applied to new initiatives the user creates.
		1. [x] Sort by (initiative-level `sort_mode` + reverse).
		2. [x] Progress calculation (`leaf_average` / `single_level`).
	3. [x] **My Task Defaults** — attributes applied to new tasks; each setting also offers a "Match parent" choice. These apply to tasks in initiatives the user owns; other tasks use the initiative owner's defaults, unless future work adds initiative-level task defaults that override them.
		1. [x] Sort by ("Match parent" = inherit, today's `nil`).
		2. [x] Priority ("Match parent" copies the parent task's priority; root-level tasks fall back to `normal`).
		3. [x] Assign to owner (checkbox — new tasks auto-assign to the initiative owner).
	4. [x] **Display elements**
		1. [x] Show Task Activity log (checkbox).
	5. [x] **Reset confirmation prompts.** Clears the `doit:confirm-skip:*` suppression flags — `.03.01.11` parked them in localStorage "until a profile/settings page"; this is that page.
	6. [x] **Initiative-list sort persists server-side** (promoted from BACKLOG). The Initiatives index's existing sort control (mode + reverse + manual drag order) writes through to per-user server storage instead of localStorage, so it follows the account across devices — mode/reverse on the user-preferences record, manual order on `initiative_members.sort_order`. No new UI on the account page; the control stays on the index.

3. [ ] **Usability testing** — the eye-pass once the worklists above are built; mix tests ride each implementation item.
	1. [x] [Human] Register a new account with a username; log in with the username, then again with the email.
	2. [ ] [Human] Account page pass — each field edits, live-validates, and saves with the optimistic hue; light + dark.
	3. [ ] [Human] Theme preference: set dark, hard-reload — no wrong-theme flash at first paint.
	4. [ ] [Human] Avatars read correctly everywhere they render (member list, assignee chip, activity log); light + dark.
	5. [ ] [Human] Two-window presence: select a task in one window — its row shows your avatar in the other; deselect and disconnect both clear it.
	6. [ ] [Human] Preferences apply — a new initiative/task picks up the defaults; "Match parent" inherits; "Reset confirmation prompts" makes a suppressed confirm reappear.
	7. [ ] [Human] Delete-account paths — a sole-member initiative goes with the account; multi-member ownership blocks with the "transfer or delete first" message.
